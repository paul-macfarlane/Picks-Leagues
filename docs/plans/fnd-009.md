# Plan: FND-009 — Vite + React + TanStack Router skeleton

**Status:** IMPLEMENTED
**Mode:** collaborative
**Author:** planner agent

## Goal

Turn the placeholder `apps/web` package into a real running SPA: Vite + React 19 +
TypeScript with TanStack Router file-based routing and a single `/` route showing
"Hello Picks Leagues." This is the frontend bootstrap every later web ticket builds
on (FND-010 Tailwind/shadcn, FND-011 TanStack Query, FND-014/015 auth). It also
unblocks the `import.meta.env` typing that `apps/web/src/lib/api.ts` is explicitly
waiting on (see its WHY comment). No product surface yet — this is pure scaffolding
that has to land conforming to the established monorepo conventions (strict TS via
`tsconfig.base.json`, the existing flat ESLint config, Prettier import ordering).

## Acceptance criteria (from ticket)

- `pnpm dev` opens the app at `localhost:5173`.
- TanStack Router devtools available in dev.
- File-based route generation works.

  partial scaffold from FND-008: `src/lib/api.ts`, `src/lib/api-client/` (generated,

## Decisions (human-confirmed, collaborative mode)

These points were raised as open questions and answered by the human; they
are now settled and reflected in the Approach below:

1. **Commit the generated `routeTree.gen.ts`.** Confirmed yes — matches the FND-008
   precedent of committing generated client output. Keep it ESLint-ignored (already
   configured); never hand-edit it. See Approach step 8.
2. **`server.strictPort: true`.** Confirmed yes — `pnpm dev` should fail loudly if
   5173 is taken so the "opens at localhost:5173" acceptance criterion is
   deterministic. See Approach step 2.
3. **Wire `vite build` in this ticket.** Confirmed yes — add the `build` script now
   so root `pnpm -r build` is not left half-broken, even though the stated ACs are
   dev-only. See Approach step 9.
4. **Scaffold on React 19, not React 18 as the ticket literally states.** Rationale:
   React 19 stable since Dec 2024; TanStack Router requires "react v18 or later" and
   supports 19; downstream FND-010 (shadcn), FND-011 (TanStack Query), FND-014
   (Better Auth client) all support React 19; starting on the current major avoids a
   future migration ticket. The ticket text in `docs/backlog/01-foundation.md` has
   been updated to React 19 with a parenthetical noting the deviation. No
   React-19-specific codemod or API change is needed for this skeleton: there are no
   `forwardRef` components to unwrap, no `propTypes`/`defaultProps` on function
   components, no legacy `ReactDOM.render` — the entry already uses `createRoot`.
   `createRoot` (from `react-dom/client`) and `<StrictMode>` are unchanged in React
   19, so Approach steps 4–6 are unaffected. See Approach steps 1 and 4–6.

## Approach

1. Add web app dependencies to `apps/web/package.json`:
   - `dependencies`: `react@^19`, `react-dom@^19`,
     `@tanstack/react-router` (latest 1.x), `@tanstack/react-router-devtools`
     (the devtools package; in current TanStack Router the router devtools moved
     out of `@tanstack/react-devtools` — implementer to confirm exact package name
     and version against Context7/registry at install time).
   - `devDependencies`: `vite@^7` (or latest stable), `@vitejs/plugin-react`,
     `@tanstack/router-plugin` (provides the Vite plugin for file-based route
     generation), `@types/react@^19`, `@types/react-dom@^19`.
   - Keep existing `openapi-fetch` / `openapi-typescript` entries untouched.
   - **Confirm exact versions at install time.** Resolve the latest stable
     versions against the registry/Context7 rather than pinning blind, and
     additionally verify the React 19 peer ranges line up:
     `@vitejs/plugin-react`'s current major declares React 19 support (it does),
     `@tanstack/react-router` / `@tanstack/react-router-devtools` /
     `@tanstack/router-plugin` accept React 19 as a peer, and `@types/react` /
     `@types/react-dom` are on the `^19` line so JSX typings match the runtime.

2. Create `apps/web/vite.config.ts`:
   - Plugins: `tanstackRouter({ target: "react", autoCodeSplitting: true })`
     **before** `react()` (TanStack Router plugin must run before the React plugin).
   - `resolve.alias`: map `@` → `./src` (mirrors the tsconfig path alias) using
     `fileURLToPath`/`import.meta.url` (no Node `path`/`__dirname` — keep ESM-clean).
   - `server.port: 5173`, `server.strictPort: true` (Decision 2) so the acceptance
     criterion ("opens at localhost:5173") is deterministic and `pnpm dev` fails
     loudly rather than silently falling back to 5174 if 5173 is occupied.

3. Create `apps/web/index.html` at the package root:
   - `<!doctype html>`, `<html lang="en">`, a `<div id="root">`, viewport meta
     tag `width=device-width, initial-scale=1` (mobile-first is a non-negotiable;
     the viewport tag is the minimum even for a skeleton), `<title>Picks Leagues</title>`,
     and `<script type="module" src="/src/main.tsx">`.

4. Create `apps/web/src/main.tsx` (PascalCase not required — this is the entry
   module, kebab/lower is fine; use `main.tsx` per Vite convention):
   - Import the generated `routeTree.gen` (path: `./routeTree.gen`, no extension
     per import rules).
   - `createRouter({ routeTree })`.
   - Register the router type via the `@tanstack/react-router`
     `declare module "@tanstack/react-router" { interface Register { router: ... } }`
     block so route params/links are typed.
   - `createRoot(document.getElementById("root")!)` — `createRoot` (from
     `react-dom/client`) is unchanged in React 19, so this stays as-is. The
     non-null assertion on a guaranteed-present root element is the one acceptable
     assertion here (it is a framework guarantee, not a type-error bypass;
     code-standards bans `as` casts, not the well-known root `!`). If review
     prefers, guard with an explicit `if (!rootEl) throw new Error(...)` instead —
     implementer picks one, no `any`.
   - Render `<RouterProvider router={router} />` inside `<React.StrictMode>`
     (`StrictMode` is unchanged in React 19).

5. Create the route tree root: `apps/web/src/routes/__root.tsx`:
   - `createRootRoute({ component: RootComponent })`.
   - `RootComponent` renders `<Outlet />` plus `<TanStackRouterDevtools />`
     gated so it only mounts in dev (`import.meta.env.DEV`). No app chrome, no
     nav — that is later tickets.

6. Create the index route: `apps/web/src/routes/index.tsx`:
   - `createFileRoute("/")({ component: IndexComponent })`.
   - `IndexComponent` returns a minimal element rendering the text
     `Hello Picks Leagues`. No Tailwind classes (FND-010), no shadcn. Plain
     semantic HTML (`<main><h1>…</h1></main>`) is acceptable for a skeleton; the
     "compose shadcn, not raw HTML" rule applies once shadcn exists (FND-010).

7. Create `apps/web/src/vite-env.d.ts` with
   `/// <reference types="vite/client" />` so `import.meta.env` (used by
   `src/lib/api.ts` and the devtools gate) is typed. This is the typing the
   existing `api.ts` comment said would "land in FND-009".

8. Generated file: `apps/web/src/routeTree.gen.ts` is produced by the TanStack
   Router Vite plugin on first dev/build. It is already ESLint-ignored. Per
   Decision 1, **commit it** (consistent with how `apps/web/src/lib/api-client/`
   generated output is committed and gated). Add it to the "Files touched
   (generated)" list; do not hand-edit it.

9. Update `apps/web/package.json` scripts:
   - `dev`: `pnpm run gen:api && vite` — keep the `gen:api` prerequisite that
     FND-008 wired (web dev still needs a fresh typed client), then start Vite.
     Root `pnpm dev` (`pnpm -r --parallel dev`) then runs API `tsx watch` (3000)
     and web Vite (5173) together. **No root package.json change needed.**
   - `build`: `pnpm run gen:api && tsc --noEmit -p tsconfig.json && vite build`
     (replace the echo placeholder; typecheck stays part of build). Per Decision 3,
     wiring this now keeps `pnpm -r build` whole.
   - `typecheck`: keep `tsc --noEmit -p tsconfig.json` (already correct).
   - Leave `test`/`lint` as-is (no web tests this ticket — skeleton has no
     non-trivial interactive component per code-standards § Frontend testing).

10. Adjust `apps/web/tsconfig.json` minimally:
    - Add `"vite.config.ts"` and the root-level `.tsx`/`.ts` to the type-checked
      set, OR keep `include: ["src"]` and add a separate `tsconfig.node.json` for
      `vite.config.ts` (standard Vite split). **Recommend the tsconfig.node.json
      split** to keep DOM libs out of the Vite config's Node context, mirroring
      the API package's existing `tsconfig.json` + `tsconfig.tooling.json` split.
    - Drop `noEmit: true` from the leaf tsconfig only if it conflicts with
      `vite build`; Vite does its own transpile, so `noEmit` can stay and
      `tsc` remains typecheck-only. Keep `noEmit: true`.
    - Ensure the generated `routeTree.gen.ts` is inside `src` so it is
      type-checked (it imports the route modules; we want type errors caught).

11. Delete `apps/web/src/index.ts` (its own comment declares it replaced by this
    ticket). Verify nothing imports `APP_NAME` first (Grep). If something does,
    keep it and revisit — but expectation is it is unused now that FND-008's
    `api.ts` is the real entry surface. See Risks for the `knip` interaction.

## Risks

- **TanStack Router devtools package name/version churn.** Recent TanStack
  releases reorganized devtools packaging (`@tanstack/react-router-devtools`
  vs. the unified `@tanstack/react-devtools`). The implementer must resolve the
  exact current package against the registry/Context7 at install time rather
  than pinning blind. Flagged, not blocking.
- **React 19 peer ranges.** Low risk — React 19 has been stable since Dec 2024
  and `@vitejs/plugin-react`, the TanStack Router packages, and `@types/react@^19`
  all support it. The "confirm exact versions at install time" note in Approach
  step 1 covers verifying the peer ranges resolve cleanly. Flagged, not blocking.
- **`knip` / dead-export gate vs. deleting `src/index.ts`.** code-standards
  bans dead exports and CI runs `knip`/`ts-prune`. Deleting `src/index.ts`
  removes a dangling export (good). But after this ticket `src/lib/api.ts`'s
  `apiClient` export still has no caller until FND-011. That export pre-exists
  this ticket and is out of scope to fix here — note for the reviewer so it is
  not attributed to FND-009.
