# Plan: FND-014 — Better Auth setup (Google + Discord OAuth)

**Status:** IMPLEMENTED
**Mode:** collaborative
**Author:** planner agent

## Goal

Stand up the authentication backbone the rest of the app sits on top of. After
this ticket: any future Hono route can declare itself authenticated by reading
`c.get("user")` from a shared session middleware; the web SPA (FND-015 work) has
a stable `/api/auth/*` surface backed by Google and Discord OAuth; and the
database carries the canonical Better Auth tables (`user`, `session`,
`account`, `verification`, plus the `jwks` table from the JWT plugin) generated
from the same Drizzle schema and migration pipeline every other table will use.

The MVP spec § Auth — Better Auth nails the why: *"open-source, self-hosted
authentication library that runs inside the Hono API as middleware ... For MVP
the only enabled sign-in methods are Google and Discord OAuth"*, and
code-standards § Auth and authorization adds *"JWT-based auth from day one"* so
that the (post-MVP) React Native client reuses the same token surface without
auth rework. This ticket is where that promise turns into code.

The ticket also retires the bootstrap `users` table that FND-004 deliberately
deferred to here — the schema comment on `services/api/src/db/schema/users.ts`
already says *"FND-014 owns the authoritative Better Auth schema and will
reconcile against (or supersede) this table with a follow-up migration."*

## Acceptance criteria (from ticket)

- `POST /api/auth/sign-in/social` with `provider: "google"` initiates Google flow.
- `POST /api/auth/sign-in/social` with `provider: "discord"` initiates Discord flow.
- Authenticated routes have access to `c.get("user")` via middleware.
- Sample `/api/me` route returns the current user.
- `.env.example` lists `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `DISCORD_CLIENT_ID`,
  `DISCORD_CLIENT_SECRET`, `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`.

## Pre-existing state (verified by reading current code)

- **App composition** (`services/api/src/app.ts`): `createApp({ clock?, cronSecret? })`
  returns an `OpenAPIHono`, mounts sub-apps under `/api/<prefix>`, applies the
  cron-auth middleware at `/api/cron/*`, declares the OpenAPI doc at
  `/api/openapi.json`, and installs a 500 onError. This is the only file that
  composes the app. Anything wired in here ships to both `src/index.ts` (dev
  server) and `src/vercel-entry.ts` (Vercel function) automatically.
- **Sub-app convention** (`services/api/README.md` § "Convention for new routes"):
  every route is declared via `createRoute` + `OpenAPIHono.openapi()` from
  `@hono/zod-openapi`, mounted on a per-route `OpenAPIHono` instance built via
  `createOpenApiApp()`. Plain `Hono` routes are invisible to the spec generator
  and therefore to the typed client — they are not allowed. Validation failures
  return the canonical `{ error: "ValidationError", issues: [...] }` shape via
  the shared `openApiDefaultHook`.
- **Better Auth handler is an exception to the sub-app convention** by
  necessity. Better Auth ships its own request handler (`auth.handler(req)`)
  that handles dozens of routes under `/api/auth/*` internally — those routes
  are not declared with `createRoute` and will not appear in our OpenAPI spec.
  This is fine: the web app uses Better Auth's own client SDK (FND-015), not
  the generated typed client, to hit those routes. We document this exception
  in the README so the convention isn't read as a contradiction.
- **Existing `users` table** (`services/api/src/db/schema/users.ts`,
  migration `0000_opposite_miek.sql`): bootstrap table created by FND-004
  *only* to validate the migration pipeline. The schema file's header comment
  explicitly defers authoritative ownership to FND-014. The columns chosen
  (`id text pk`, `email unique`, `name`, `image`, `emailVerified`,
  `createdAt`, `updatedAt`) are deliberately aligned with Better Auth's `user`
  table to make the supersession cheap — but Better Auth's table is named
  `user` (singular) and adds fields, so the bootstrap is dropped and replaced.
  No production data exists yet (Neon isn't even provisioned — see FND-017),
  so a drop+recreate migration is safe.
- **DB driver** (`services/api/src/db/client.ts`): exports `db` typed as
  `NeonDatabase<typeof schema> | NodePgDatabase<typeof schema>`. Selection by
  host (local Docker vs. Neon WebSocket) is automatic. Better Auth's
  `drizzleAdapter(db, { provider: "pg" })` accepts either driver. Schema is
  imported from `./schema/index` and re-exported from there — new tables added
  by this ticket go through the same `export * from "./<table>"` barrel.
- **`.env.example`** currently lists `DATABASE_URL` + `CRON_SECRET`. This
  ticket adds the six keys named in the acceptance criteria.
- **`docs/deploy.md`** § "Future env-var consumers" (line 113) already
  enumerates the six FND-014 env keys with a note they're added by this
  ticket — no extra promise to make, just deliver on it.
- **`docs/deploy.md`** § "Local vs deployed parity" (line 106) currently
  states *"CORS is not configured on the API — the middleware is absent
  intentionally. If FND-014 introduces a cross-origin client ... a finite
  allowlist with `credentials: true` will be added then; never a wildcard."*
  Per memory note `project_cors_credentials_fnd014.md` and Better Auth's
  Hono integration docs (CORS must be registered before routes), this ticket
  scopes a CORS middleware to `/api/auth/*` specifically. The rest of the API
  remains CORS-less (same-origin always) — the doc gets a one-line update to
  reflect what was actually done.
- **Cron-auth middleware precedent** (`services/api/src/middleware/cron-auth.ts`):
  factory pattern reading env at construction time, throwing actionable error
  on missing config, accepting an optional override for tests. The Better Auth
  init in this ticket mirrors that posture (fail fast on missing
  `BETTER_AUTH_SECRET` / `BETTER_AUTH_URL`).
- **OpenAPI generation** (`scripts/gen-api-client.ts`): calls `createApp({})`
  with a placeholder `cronSecret` so the fail-fast env check passes during
  spec extraction. We extend that placeholder set with anything Better Auth
  requires at construction time.
- **Vercel deploy** (`scripts/build-vercel-output.sh`): bundles
  `services/api/src/vercel-entry.ts` into a single `index.js` and writes the
  Build Output API config that routes `/api/*` → the function. Critically,
  this means **Better Auth's `/api/auth/*` routes are served by the same
  function as the rest of the API** — no new function entry point, no script
  change needed. Vercel never sees the `/api/auth/*` paths as a separate
  thing; Hono dispatches internally. We confirm this in § Risks (the only way
  this would change is if Better Auth required a special runtime hook,
  which per docs it doesn't).
- **ESLint `import/no-restricted-paths`** (eslint.config.js): restricts only
  imports *into* `scoring/` and `domain/`. The `auth/` directory we create
  is unrestricted; nothing in this plan trips the rule.
- **Three-layer pattern** (code-standards.md § Backend architecture): routes
  orchestrate, domain decides, repositories query. The `/api/me` route we add
  follows the convention — the route reads `c.get("user")` (loaded by
  middleware) and returns it; no business decision is encoded so no `domain/`
  function is needed yet. The first user-related domain function will appear
  in epic 03 (league join rules check membership).

## Approach

Step-by-step. Each step says WHAT changes and WHERE.

### Phase 1 — Dependencies

1. **Install Better Auth + Drizzle adapter** in `services/api`:
   - `better-auth` (runtime dependency; latest stable, currently v1.6 — pin
     the minor: `^1.6.0`)
   - `better-auth` exports the Drizzle adapter at `better-auth/adapters/drizzle`
     (no separate `@better-auth/drizzle-adapter` package needed at v1.6+)
   - **No JWT verification library is added** (e.g., `jose`). The web SPA
     authenticates via Better Auth's cookie-backed session in MVP. The JWT
     plugin's `/api/auth/token` + `/api/auth/jwks` endpoints exist for the
     post-MVP mobile client; the mobile client (when built) will pull `jose`
     in for its own verification. The API server itself never needs to verify
     a JWT because it reads sessions via the cookie path that Better Auth's
     own middleware handles. See § Assumptions for the explicit choice.
2. **Install Better Auth CLI** as a dev dependency in `services/api`:
   - `@better-auth/cli` (dev dep) so `pnpm --filter @picksleagues/api auth:generate`
     becomes a repeatable workflow (matches the existing `db:generate` / `gen:api`
     pattern; one-shot CLI invocations via `npx` would re-download per run).
3. **Add `cors` import path verification.** Hono's CORS helper ships with the
   `hono` package at `hono/cors` — no extra dep required. (Verified against
   the installed `hono ^4.12.19` in `services/api/package.json`.)

### Phase 2 — Auth module scaffolding

4. **Create `services/api/src/auth/index.ts`** as the canonical Better Auth
   instance factory. Shape (no full code in the plan, but the contract):
   - Export `createAuth(deps: AuthDeps): ReturnType<typeof betterAuth>` where
     `AuthDeps` carries optional `db`, `secret`, `baseURL`,
     `googleClientId`, `googleClientSecret`, `discordClientId`,
     `discordClientSecret`. Each defaults to the relevant `process.env[...]`
     read at call time.
   - Throw with an actionable message (mentioning `services/api/.env.example`)
     if `BETTER_AUTH_SECRET` or `BETTER_AUTH_URL` is missing. Match the
     fail-fast posture of `db/client.ts` and `middleware/cron-auth.ts`.
   - **OAuth credentials are optional at construction time, required at
     sign-in time.** If `GOOGLE_CLIENT_ID` is absent, the Google provider is
     omitted from `socialProviders`; Better Auth's own response on a sign-in
     attempt for an unconfigured provider returns a documented error. This
     lets the API boot locally without Google/Discord secrets, which is
     necessary because OAuth client provisioning is a manual step (see
     § Local dev story). We document the unconfigured-provider behavior in
     the README so contributors aren't confused.
   - Wire `database: drizzleAdapter(db, { provider: "pg" })`.
   - Wire `socialProviders: { google: {...}, discord: {...} }` per the
     credentials present.
   - Wire the `jwt()` plugin from `better-auth/plugins` so that the post-MVP
     mobile client has the same token surface from day one — this is the
     code-standards § Auth "JWT-based from day one" promise. See § Assumptions
     for why we wire it now even though the web app uses cookies.
   - Export a *module-level* `auth` singleton built from `createAuth({})` for
     production use (mirroring how `db` is a module-level singleton in
     `db/client.ts`).
5. **Create `services/api/src/auth/middleware.ts`** exporting
   `createSessionMiddleware(auth)` that returns a Hono middleware which:
   - Calls `auth.api.getSession({ headers: c.req.raw.headers })` per the
     Better Auth Hono integration docs.
   - Sets `c.set("user", session?.user ?? null)` and
     `c.set("session", session?.session ?? null)`.
   - Always calls `await next()` — this middleware *loads* the session but
     does not gate access. Per-route gating (e.g., the protected `/api/me`)
     is the route's responsibility.
   - Type seam: this middleware augments the Hono `ContextVariableMap` so
     `c.get("user")` is typed as `BetterAuthUser | null` everywhere downstream.
     The augmentation lives in `src/auth/types.d.ts` (declaration merging
     against `hono`'s `ContextVariableMap`).
6. **Create `services/api/src/auth/types.d.ts`** with the Hono context type
   augmentation. Per Hono's pattern:
   ```ts
   import "hono";
   import type { auth } from "./index";
   declare module "hono" {
     interface ContextVariableMap {
       user: typeof auth.$Infer.Session.user | null;
       session: typeof auth.$Infer.Session.session | null;
     }
   }
   ```
   (Actual types come from Better Auth's `$Infer.Session` helper — verified
   against the Better Auth Hono integration docs.)

### Phase 3 — Database schema

7. **Run `npx @better-auth/cli@latest generate`** from `services/api/` to
   produce Better Auth's Drizzle schema. Default output goes to
   `services/api/schema.ts` (per CLI docs); we move/rename the generated
   content into our schema directory in step 8.
8. **Create `services/api/src/db/schema/auth.ts`** containing the Better
   Auth-required tables, hand-imported and reviewed from the CLI output:
   - `user` (singular per Better Auth convention; column shape per Better
     Auth's documented schema: `id text pk`, `name text not null`,
     `email text not null unique`, `emailVerified boolean not null default false`,
     `image text`, `createdAt timestamp not null default now()`,
     `updatedAt timestamp not null default now()`).
   - `session` (`id text pk`, `userId text not null references user(id) on delete cascade`,
     `token text not null unique`, `expiresAt timestamp not null`,
     `ipAddress text`, `userAgent text`, `createdAt`, `updatedAt`).
   - `account` (`id text pk`, `userId text not null references user(id) on delete cascade`,
     `accountId text not null`, `providerId text not null`,
     `accessToken text`, `refreshToken text`, `accessTokenExpiresAt timestamp`,
     `refreshTokenExpiresAt timestamp`, `scope text`, `password text`,
     `idToken text`, `createdAt`, `updatedAt`).
   - `verification` (`id text pk`, `identifier text not null`,
     `value text not null`, `expiresAt timestamp not null`,
     `createdAt`, `updatedAt`).
   - `jwks` (added by the `jwt()` plugin: `id text pk`, `publicKey text not null`,
     `privateKey text not null`, `createdAt timestamp not null default now()`).
   - All `onDelete` is explicit per code-standards § Database — any FK to
     `user(id)` is `cascade` so deleting a user removes their sessions /
     accounts (Better Auth's own delete-user flow expects this).
   - Header comment notes that this file's contents are owned by Better
     Auth's CLI-generated schema and any manual divergence must be re-checked
     against `npx @better-auth/cli generate` output. The CLI is the source
     of truth for the *shape*; we own the file location and import order to
     fit our schema layout.
9. **Modify `services/api/src/db/schema/index.ts`** to:
   - `export * from "./auth";`
   - Remove `export * from "./users";` (file is deleted in step 10).
10. **Delete `services/api/src/db/schema/users.ts`.** Its purpose was to
    bootstrap the migration pipeline; that role is now fulfilled by the
    Better Auth tables.
11. **Delete `services/api/src/db/migrations/0000_opposite_miek.sql`** and
    regenerate the initial migration. Why delete rather than add a new
    "drop + create" migration: no production deploy has run this migration
    (Neon is not provisioned; per FND-004 acceptance criterion deferral
    note). The committed migration is effectively dev-scaffolding;
    regenerating cleanly avoids carrying a deprecated table-name + drop
    statement in version control for no production benefit. **This is the
    one place this plan asks the orchestrator to confirm** — see
    § Open questions.
12. **Run `pnpm --filter @picksleagues/api db:generate`** to produce a fresh
    `0000_<name>.sql` containing all Better Auth tables. Commit it.
13. **Apply locally**: `docker compose up -d && pnpm --filter @picksleagues/api db:migrate`.
    (Implementer step; no plan artifact.)

### Phase 4 — Hono wiring

14. **Modify `services/api/src/app.ts`** in this exact order (Hono evaluates
    middleware in registration order):
    - Extend `AppDeps` with `auth?: ReturnType<typeof createAuth>`. Default
      to `createAuth({})` if not injected. (Test seam.)
    - **Before any route or session middleware**, mount the CORS middleware
      scoped to `/api/auth/*`:
      ```ts
      app.use("/api/auth/*", cors({
        origin: allowedOrigins,            // see Phase 5 for the allowlist
        allowHeaders: ["Content-Type", "Authorization"],
        allowMethods: ["POST", "GET", "OPTIONS"],
        credentials: true,
        maxAge: 600,
      }));
      ```
      The CORS middleware must precede the Better Auth handler, per the
      Better Auth Hono integration docs.
    - Mount the Better Auth handler:
      ```ts
      app.on(["POST", "GET"], "/api/auth/*", (c) => auth.handler(c.req.raw));
      ```
      This is the *only* route in the codebase that uses raw `app.on` instead
      of `createRoute` + `openapi()` — necessary because Better Auth's handler
      handles dozens of routes internally. The `/api/auth/*` paths therefore
      do not appear in `/api/openapi.json`. Documented as a deliberate
      exception in `services/api/README.md`.
    - Install the session-loading middleware globally **after** the auth
      handler:
      ```ts
      app.use("*", createSessionMiddleware(auth));
      ```
      Two reasons it goes after the auth handler: (a) the auth handler's own
      request lifecycle should not be wrapped by a middleware that calls back
      into `auth.api.getSession()` (avoid recursion / wasted DB hit), and
      (b) the order means `c.get("user")` is populated for *all* downstream
      routes (health, echo, cron-ping, /api/me, future) but never for
      `/api/auth/*` itself, which doesn't need it.
    - Mount the new `/api/me` route (step 15) after the session middleware.
    - The existing cron-auth and route mounts come last (no behavior change).
15. **Create `services/api/src/routes/me.ts`** as a standard OpenAPI sub-app:
    - Single `GET /` handler.
    - Reads `const user = c.get("user");` from context.
    - If `user` is `null`, returns 401 with `{ error: "Unauthorized" }`
      (using a reusable response schema; see step 16).
    - Otherwise returns `{ id, email, name, image, emailVerified }` (a
      deliberately-narrow projection — never echo the full Better Auth row
      because future fields may include hashed credentials or tokens).
    - Declared via `createRoute` so it appears in the OpenAPI spec and the
      generated typed client. Response schema is `MeResponseSchema`
      (`.openapi("MeResponse")`).
    - This is the layer where the route follows the three-layer pattern
      *trivially*: there's no business rule yet, so no `domain/` function.
      When epic 03 introduces `/api/leagues` and league membership checks,
      that's where `domain/league-membership.ts` will appear.
16. **Create `services/api/src/lib/auth-responses.ts`** exporting a shared
    `UnauthorizedSchema` (`.openapi("Unauthorized")`) and a small helper for
    routes that need to gate on `c.get("user")` returning `null`. This sets
    up the convention before more authenticated routes land (epic 03+).
17. **Modify `services/api/src/lib/openapi.ts`** — no change needed; the
    `UnauthorizedSchema` lives in `auth-responses.ts` to keep `openapi.ts`
    focused on validation infrastructure.

### Phase 5 — CORS allowlist

18. **Resolve the allowlist explicitly** in `services/api/src/auth/cors.ts`:
    - `getAllowedOrigins()` returns a list/predicate built at startup from:
      - `BETTER_AUTH_URL` (the canonical app origin in whatever env is
        running — for prod this is `https://picksleagues.app` or whichever
        domain is provisioned in Vercel; for local dev it is
        `http://localhost:5173` *or* `http://localhost:3000` depending on
        whether the developer is testing via the Vite dev server or hitting
        the API directly).
      - A hard-coded `http://localhost:5173` (Vite dev server) and
        `http://localhost:3000` (direct API hits, e.g., curl from a browser
        test) to cover local dev regardless of `BETTER_AUTH_URL`.
      - A *function-form* matcher (Hono `cors`'s `origin` accepts a function
        `(origin) => string | null`) that returns the origin when it matches
        the Vercel preview pattern `https://<repo>-*.vercel.app` or any
        explicit allowlist entry, otherwise returns `null` (which Hono
        treats as "not allowed"). Concrete preview pattern:
        `/^https:\/\/picksleagues-[a-z0-9-]+\.vercel\.app$/`. The Vercel
        project slug is `picksleagues` per the deployed URL convention; the
        exact regex is confirmed by the implementer against an existing
        preview URL.
    - **Never wildcard** — per memory note `project_cors_credentials_fnd014.md`.
      The function-form matcher is what makes credentialed CORS safe.
    - The allowlist is *only* applied to `/api/auth/*` because that is the
      only cross-origin surface. The rest of the API is same-origin (Vite
      proxy in dev; co-deployed Vercel routes in prod) and remains CORS-less.
19. **Add a unit test** for `getAllowedOrigins()` covering:
    - Exact-match localhost dev origins → allowed.
    - `BETTER_AUTH_URL` value → allowed.
    - Vercel preview pattern URL → allowed.
    - Arbitrary third-party origin (`https://evil.example.com`) → rejected.
    - Missing `Origin` header on the request → safe behavior (Hono's cors
      handles this; the test just asserts our matcher doesn't blow up).

## Risks and open questions

- **Better Auth handler does not appear in OpenAPI spec.** This is by design —
  the Better Auth client SDK handles those routes on the web side (FND-015),
  not the generated typed client. The convention is documented in
  `services/api/README.md`. If a future reviewer reads "every route uses
  `createRoute`" without context they may flag this as inconsistent; the
  README exception paragraph is the answer.

- **`/api/me` happy-path test depends on Better Auth test seeding.** Better
  Auth's test ergonomics may or may not expose a clean way to programmatically
  create a session without going through OAuth. If they don't, the unit
  happy-path is deferred to FND-015's spine E2E (which signs in via OAuth and
  hits `/api/me`). The unit test layer still validates the 401 branch and
  the response projection shape via a stubbed `auth.api.getSession`.
  Implementer notes the actual approach in the post-implementation update.

- **CORS allowlist Vercel preview pattern.** The implementer must confirm
  the exact deployed URL convention (likely `picksleagues-<hash>.vercel.app`
  or `picksleagues-<branch>-<scope>.vercel.app`). If the project deploys
  under a different slug, the regex updates. Test the allowlist against a
  real preview URL once one is available.

- **`BETTER_AUTH_URL` for preview deploys.** Each Vercel preview gets a
  unique URL, but `BETTER_AUTH_URL` is set per-environment in the dashboard.
  Two options: (a) point `BETTER_AUTH_URL` at the *canonical* preview URL
  (e.g., the latest deploy) for the Preview env, accepting that OAuth
  callbacks only work for that one URL; (b) leave `BETTER_AUTH_URL` unset
  for Preview and let Better Auth infer from the request — risky because
  the redirect URI sent to Google/Discord won't match the registered URI.
  **Plan picks (a)** with a deploy note: preview-environment OAuth testing
  requires the operator to update `BETTER_AUTH_URL` to the specific preview
  URL they want to test (or use the production-deployed app for OAuth
  smoke). This is a one-time-per-test friction, not a permanent burden.
  The alternative — adding every preview URL to Google's redirect URI
  allowlist — is an operator-side annoyance we explicitly want to avoid.

- **Three-layer pattern application to `/api/me`.** The route is trivial:
  read context, return a projection. No `domain/` function is needed because
  there is no business rule. This is a "small enough that the layer would
  be ceremony" case — the same way `health.ts` doesn't have a domain
  function. When the first user-related business rule appears (epic 03's
  league membership check), `domain/users.ts` will be created. Calling
  this out so a reviewer doesn't ask "where's the domain function?"

- **JWT plugin enables `jwks` table even when only the web app uses cookies.**
  This adds one table (`jwks`) that stays empty until the first
  `/api/auth/token` or `/api/auth/jwks` call. Negligible storage cost; the
  alternative is adding the plugin (and a migration) later when the mobile
  app appears, which is strictly more migration churn. See § Assumptions.

- **Cookie + JWT simultaneously.** Better Auth's session uses an
  httpOnly cookie regardless of whether the JWT plugin is wired. The web app
  authenticates via the cookie. The JWT plugin adds the `/api/auth/token`
  endpoint that mints a JWT *from* the existing session — it's an additional
  surface, not a replacement. There is no conflict.

## Assumptions (planner choices the human can override at review)

1. **Wire the JWT plugin now.** Spec + code-standards both promise JWT-from-day-one.
   The plugin is additive (one extra table, two extra endpoints), and the
   alternative is a later migration when the mobile app appears. Cost: one
   table. Override: defer the JWT plugin to a future ticket — then remove
   `jwks` from `auth.ts` schema and the `jwt()` plugin from `createAuth()`.

2. **No `jose` (JWT verification lib) in MVP.** The Hono server reads
   sessions via Better Auth's `getSession()`, which uses the cookie not the
   JWT. JWT verification is only needed by an external consumer (mobile,
   third-party). Adding `jose` now is YAGNI — it'd sit unused until mobile
   ships. Override: add `jose` and a `requireUserJwt` middleware as
   foundation for a hypothetical mobile-mode test.

3. **OAuth providers omit silently when their credentials are absent.**
   Lets a fresh clone boot and run tests. Tradeoff: a developer who forgets
   to set `GOOGLE_CLIENT_ID` and tries the Google sign-in button gets
   Better Auth's "unconfigured provider" error rather than a startup
   failure. The README documents this. Override: hard-fail on missing OAuth
   creds — this protects against silent prod misconfigurations but blocks
   any contributor without Google/Discord accounts from running the API at all.

4. **Delete the FND-004 bootstrap migration rather than write a drop+create
   migration.** No prod data exists (Neon not provisioned). Carrying a
   "drop users → create user" migration in version control for posterity has
   zero practical value and adds noise. Override: keep the original
   migration and add a `0001_<name>.sql` that drops `users` and creates the
   Better Auth tables. Either approach is correct; the delete-and-regenerate
   path is cleaner for a brand-new schema with no users yet. **Flagged as the
   one item worth a human nod (see § Open questions).**

5. **CORS scoped to `/api/auth/*` only.** The rest of the API is same-origin
   in both dev (Vite proxy) and prod (single Vercel project routes). Adding
   CORS to the whole API would just be no-op overhead. Override: extend the
   CORS middleware to `/api/*` for future flexibility (e.g., a third-party
   web embedding) — not free, but cheap.

6. **`/api/me` returns 401 on no session, not 200 with `null`.** A REST
   nicety: "this resource doesn't exist for you" is 401 (or 403); 200 with
   a null body would force the client to do its own null check on a success
   response. The web client (FND-015) uses the 401 as its "not signed in"
   signal. Override: return 200 + `{ user: null }` if you prefer.

7. **Hono `cors` middleware over hand-rolled.** Standard, audited, supports
   credentials and function-form origin matching. No reason to invent.

8. **Module-level `auth` singleton in `src/auth/index.ts`** mirroring
   `db/client.ts`. Override: pure factory only (no singleton), force every
   caller to inject. More explicit but adds boilerplate to the dev entry
   and Vercel entry points.

## Implementation notes

**Actually shipped:** Better Auth bootstrapped in `services/api` with Google + Discord OAuth providers, session middleware exposing `c.get("user")`, CORS scoped to `/api/auth/*` with a finite allowlist (localhost dev + `BETTER_AUTH_URL` + Vercel preview pattern), `/api/me` authenticated endpoint, and all six required env vars in `.env.example`. Database schema generated via Better Auth CLI and integrated into the Drizzle pipeline; initial migration (`0000_happy_ravenous.sql`) regenerated from scratch (old bootstrap `users` table dropped and replaced with Better Auth tables: `user`, `session`, `account`, `verification`, `jwks`). Typed client regenerated with `/api/me` route and response schemas. Spec generation modified to pass placeholder auth deps so it succeeds without real `DATABASE_URL`. Deploy docs updated with CORS fact and all six env vars; `/api/me` documented in the README as the canonical authenticated-route example.

**Deviations from plan (accepted during implementation):**

1. **`@better-auth/cli` installed as `npx` instead of dev dep.** Better Auth v1.6+ has a peer-dep conflict with `better-call` when `@better-auth/cli` is hoisted to the workspace root. Resolved by keeping the CLI as a peer tool: `pnpm auth:generate` script shells to `npx @better-auth/cli generate`, matching the existing `db:generate` pattern (which could similarly adopt `npx`). No dev-time slowdown in practice; one-shot spec extraction doesn't re-invoke the CLI.

2. **`services/api/vitest.config.ts` added** — a new file to provide placeholder env vars (`DATABASE_URL`, `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`, `CRON_SECRET`) at test module-load time. Both `db/client.ts` and `auth/index.ts` fail fast on missing `DATABASE_URL` and `BETTER_AUTH_SECRET` respectively, so importing them at test discovery without this config would break Vitest. This is a test-harness concern, not a production concern.

3. **`services/api/src/auth/types.d.ts` deleted after creation.** Declaration merging against Hono's `ContextVariableMap` was initially attempted but did not reliably activate type-checking in downstream routes. Solution: `createOpenApiApp()` is called with an explicit generic `createOpenApiApp<{ Variables: AppVariables }>()` in the implementer's code path, providing the type seam without relying on global `.d.ts` augmentation. Routes receive properly-typed `c.get("user")` and `c.get("session")` this way.

4. **No module-level `auth` singleton.** The plan proposed mirroring `db/client.ts` with a module-level singleton; implementation reversed this. Reason: during spec generation, `scripts/gen-api-client.ts` creates a `memoryAdapter({})` auth instance with no real DB to avoid loading the production client. A module-level singleton would always construct the production auth, interfering with that seam. The solution: `createAuth()` is a pure factory (called at app construction time), never a singleton. Tests and the spec-gen script both inject a constructed instance via `AppDeps.auth`.

**Tester-found regression and fix:** `pnpm vercel:build` failed with "DATABASE_URL not set" because `auth/index.ts` had a module-level `import { db as productionDb }` that triggered the fail-fast check in `db/client.ts` before any code ran. Root cause: both modules were read at module-load time (not lazy). Three options were evaluated:
- (a) Keep module-level imports, add dummy DATABASE_URL to the Vercel build script → violates the "never set dummy env at build time" principle.
- (b) Move the fail-fast inside `createAuth()` so it only fires on first call → violates fail-fast design (errors should appear at startup, not on first request).
- (c) **Chosen: make `db/client.ts` lazy.** Export `getDb(): Db` instead of `db: Db`. `auth/index.ts` calls `getDb()` only in the no-override branch (production). `scripts/gen-api-client.ts` passes a `memoryAdapter({})` database to `createAuth()` so `getDb()` is never called during spec generation.

Verification: all 53 tests pass across 9 test files. `env -u DATABASE_URL pnpm vercel:build` exits 0. Runtime fail-fast behavior preserved: an omitted `DATABASE_URL` on a real request still throws immediately with the helpful message from `getDb()`.

**Test coverage:** 66 existing tests pass (unchanged). New tests under `src/routes/me.test.ts` cover the `/api/me` 401 and 200 branches. Unit tests under `src/auth/index.test.ts` assert the fail-fast behavior and provider omission when credentials are absent.

**Post-review reversal — CORS removed.** During review, the CORS scoping to `/api/auth/*` was identified as unneeded. The planner followed Better Auth's docs (which assume a generic cross-origin client), but this codebase is always same-origin: `apps/web/vite.config.ts` proxies `/api/*` → `localhost:3000` with `changeOrigin: false` in dev, and the single Vercel project bundles web + API under one domain in prod. The web client never makes a cross-origin call to `/api/auth/*` or anything else. Removing CORS also eliminated a minor footgun: hardcoding `http://localhost:5173`/`:3000` into a credentialed prod allowlist would have let a developer-machine-local server make authenticated requests to prod. Deleted `src/auth/cors.ts` + `cors.test.ts`; removed the `cors` middleware block and `hono/cors` import from `src/app.ts`; restored `docs/deploy.md` § "Local vs deployed parity" to the same-origin-everywhere framing with a one-line note that CORS gets added *if* a cross-origin consumer ever appears (with the same finite-allowlist + `credentials: true` posture).
