## Goal

Wire TanStack Query into the web app so all server state is fetched, cached, and
displayed through Query (a CLAUDE.md non-negotiable: "Server state lives in
TanStack Query. No `useState` for fetched data; no manual fetch-then-setState").
This ticket establishes the QueryClient at the app root, makes the Query
Devtools available in dev, documents the default query options, and proves the
pipe end-to-end by fetching and rendering the `/api/health` response on `/`
through the existing generated typed client. This is foundation plumbing every
later data-driven screen (leagues, picks, standings) depends on.

## Approach

2. **Create `apps/web/src/lib/query-client.ts`** â€” a single factory
   `createQueryClient(): QueryClient` that constructs the `QueryClient` with the
   documented default options:
   - `queries.staleTime`: `30_000` (30s). Rationale: this app's data is
     event-driven and short-lived (scores, picks); 30s is the upper bound of the
     "polling during game windows: 15â€“30 seconds" rule in code-standards, so the
     global default never causes a refetch storm while per-route configs can
     tighten polling. Non-game data tolerates 30s staleness comfortably.
   - `queries.retry`: `2`. Rationale: tolerate transient network/cold-start
     blips (Vercel serverless) without masking real failures or hammering the
     API. Mutations default to `retry: 0` (TanStack default â€” leave it; retrying
     non-idempotent writes is unsafe).
   - `queries.refetchOnWindowFocus`: `true` (TanStack default â€” left explicit
     in a comment-free, self-documenting options object; documented in the doc
     block / standards note below). Acceptable for a "halftime check" product:
     returning to the tab should show fresh data.
   - A top-of-file one-line comment is allowed here under code-standards
     ("hidden constraints / non-obvious WHY") to justify the staleTimeâ†”polling
     coupling; keep it to one line.
   - Explicit return type on the exported factory (code-standards: exported
     functions have explicit return types).

## Assumptions (collaborative mode â€” recorded, not blocking)

These are non-obvious choices made from the docs as authority; reviewer/human
can override at review time:

1. **staleTime = 30s, query retry = 2, mutation retry = 0,
   refetchOnWindowFocus = on.** Chosen to align the global default with the
   code-standards "15â€“30s polling during game windows, configured per route"
   rule (global default is the loose upper bound; routes tighten it). If you
   want a different global default (e.g., `staleTime: 0` and rely entirely on
   per-route config), say so and the factory changes only.
2. **QueryClient created in `main.tsx` and threaded into router `context`**
   (not created in `__root.tsx`). Enables future loader-based prefetch without
   re-plumbing; matches the TanStack Router + Query idiom.
3. **Devtools shipped as a normal dependency, dev-gated at render** via
   `import.meta.env.DEV`, mirroring the existing `TanStackRouterDevtools`
   treatment in `__root.tsx`.
4. **Health has no empty state** â€” a 200 is always `{status:"ok"}`; the route
   designs loading/error/happy and explicitly documents "no empty state."
5. **Router context plumbing is added but no route loader is converted** in
   this ticket (keeps scope to the AC; loader migration is per-feature later).
