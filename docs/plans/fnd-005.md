# Plan: FND-005 — Hono API skeleton with health endpoint

**Status:** IMPLEMENTED
**Mode:** collaborative
**Author:** planner agent

## Goal

Stand up the `services/api` Hono application so the backend has a real HTTP
surface and a place to receive route modules. The deliverable is a running app
that serves `GET /api/health` returning `{ status: "ok", time: <iso> }`,
runnable locally via `pnpm dev` and exposing a Vercel Node serverless entry at
`services/api/api/index.ts`. This is pure plumbing — it unblocks the rest of
the foundation epic (Zod validation FND-006, OpenAPI export FND-007, typed
client FND-008, Better Auth FND-014) by establishing the Hono app object, the
routes directory convention, the `clock.now()` helper, and the Vercel handler
shape. No DB access, no auth, no business logic.

## Acceptance criteria (from ticket)

- `pnpm dev` in `services/api` serves the health endpoint locally
- `curl localhost:3000/api/health` returns JSON
- App structured to receive route modules under `src/routes/`

## Approach

The current `services/api/src/index.ts` is a throwaway bootstrap
(`export const API_NAME`) explicitly marked "replaced entirely by FND-005". It
gets removed in favor of a real app structure.

1. **Add Hono runtime dependencies** to `services/api/package.json`:
   - `hono` (dependency)
   - `@hono/node-server` (dependency) — local dev server adapter; the Vercel
     entry uses Hono's Web-standard `fetch` handler via `hono/vercel`, which
     ships with `hono` (no extra dependency).
   - `vitest` (devDependency) — first API test runner; see Tests.
   `tsx` is already present and runs the dev server.

2. **Create `services/api/src/lib/clock.ts`** — the single `clock.now()` helper
   mandated by CLAUDE.md / code-standards.md § "Server time is the source of
   truth". Export an object `clock` with a `now(): Date` method backed by
   `new Date()`. This is introduced here (not deferred) because the health
   endpoint's `time` field is the first server-time read and must go through
   the helper so nothing in the codebase ever calls `new Date()`/`Date.now()`
   directly for a decision/output. The fake-clock swap is added when the
   simulator lands (out of scope here); the shape is `{ now: () => Date }`.

3. **Create `services/api/src/routes/health.ts`** — a Hono sub-app (router
   module) exporting the health route. Handler returns
   `c.json({ status: "ok", time: clock.now().toISOString() })` with an explicit
   return type (`code-standards.md` § TypeScript: exported functions and route
   handlers have explicit return types). This file establishes the per-module
   route convention every later route follows: each module is its own
   `new Hono()` instance, exported and mounted by the app factory.

4. **Create `services/api/src/app.ts`** — the app factory. Builds and returns
   the root `Hono` instance, mounts route modules under their base paths
   (`app.route("/api/health", healthRoute)`), and registers a minimal
   `onError` handler that converts thrown errors to a JSON 500 (the HTTP layer
   is the only place that maps errors to status codes — code-standards.md
   § "Error handling"). Keeping the factory separate from the entry points lets
   FND-006/FND-007 and tests import a fresh app without binding a port. The
   factory accepts an optional dependency bag `createApp(deps?: { clock?: Clock })`
   defaulting to the real clock, so the clock is injectable for the test
   (minimal injection seam — no DI framework). The `routes/` directory is the
   documented extension point: new tickets add a module and one
   `app.route(...)` line here.

5. **Rewrite `services/api/src/index.ts`** — the local dev entry. Imports the
   app factory and starts `@hono/node-server` on `process.env.PORT ?? 3000`.
   This is what `pnpm dev` runs. Keep it thin: build app, serve, log the bound
   URL.

6. **Create `services/api/api/index.ts`** — the Vercel Node serverless entry.
   Per the MVP spec the API runs on Vercel's Node runtime. Use Hono's
   documented Vercel pattern: `export default handle(app)` via `hono/vercel`
   (no extra dependency — `hono/vercel` ships with `hono`). The `api/`
   directory at the package root is the Vercel filesystem-routing convention;
   `api/index.ts` is the function entry that all `/api/*` traffic will map to.
   This entry is created here because the ticket explicitly names it; the
   surrounding deploy config (`vercel.json`, runtime declaration, rewrites,
   env vars, PR previews) is **deferred to FND-012** (deployment ticket). The
   entry compiles and typechecks standalone — it does not depend on any deploy
   config existing.

7. **Update `services/api/package.json` scripts**:
   - `"dev": "tsx watch src/index.ts"` (replaces the `echo` placeholder)
   - `"test": "vitest run"` (replaces the `echo` placeholder) — see Tests.
   - Leave `build` as-is (Vercel builds the function; no local build step is
     part of this ticket's acceptance).

8. **Update `services/api/tsconfig.json`** `include` to cover the new `api/`
   directory (currently only `["src"]`) so `pnpm typecheck` checks the Vercel
   entry. Ensure `"types": ["node"]` is present (the dev entry and Vercel
   entry use Node globals); mirror the existing `tsconfig.tooling.json` which
   already uses `["node"]`.

9. **Add `services/api/vitest.config.ts`** only if Vitest needs explicit
   config for the TS/ESM setup in this package (Node environment, no DOM).
   Prefer convention/zero-config; add the file only if the test does not run
   without it. The health test uses Hono's in-memory `app.request()` helper —
   no DOM, no jsdom, Node environment is sufficient.

10. **Update `services/api/README.md`** — add an "API server" section: how to
    run (`pnpm --filter @picksleagues/api dev`), the health endpoint contract,
    the Vercel entry (`api/index.ts`, with a note that deploy config lands in
    FND-012), and the `src/routes/` module convention (one Hono sub-app per
    module, mounted in `src/app.ts`). This satisfies "app structured to
    receive route modules" being a documented, discoverable convention.

## Tests

Per code-standards.md, route handlers are thin orchestrators and DB-touching
paths are covered by spine E2E later — but `/api/health` touches no DB and is
pure, so a fast Vitest test is appropriate and gives the epic its first green
API test. Vitest is wired as part of this ticket (devDependency + `test`
script).

`services/api/src/routes/health.test.ts` (Vitest, no DB, no network, no socket):

- **`GET /api/health` returns 200** — assert status 200.
- **`GET /api/health` returns the documented JSON shape** — body equals
  `{ status: "ok", time: <string> }`; assert `status === "ok"` and `time`
  parses as a valid ISO-8601 date.
- **`time` reflects the injected clock** — construct the app via
  `createApp({ clock: stubClock })` where `stubClock.now()` returns a fixed
  `Date`; assert the response `time` equals that fixed instant's ISO string.
  This proves the endpoint reads `clock.now()` and not `new Date()` directly.

Tested via Hono's `app.request("/api/health")` in-memory test helper — no port,
no HTTP socket, no DB.

## Risks and open questions

- **Port.** Acceptance says `localhost:3000`. `@hono/node-server` defaults to
  3000; the dev entry uses `process.env.PORT ?? 3000` so the curl in the
  acceptance criteria works verbatim.
- **`vitest.config.ts` necessity.** Whether an explicit config file is needed
  depends on the package's TS/ESM resolution. Implementer should prefer
  zero-config and only add the file if the test does not run otherwise — not a
  blocker either way.

## Decisions (resolved clarifying questions)

These were raised in collaborative mode and answered by the human:

1. **DB connectivity scope — RESOLVED: stay DB-free.** FND-005 touches no
   database. The health endpoint deliberately does not read the DB. Neon
   provisioning and `pnpm db:migrate` / `pnpm db:studio` validation remain
   deferred to **FND-014** (Better Auth needs a live DB and is where those
   commands get exercised end to end). The FND-004 `db:migrate`/`db:studio`
   deferral is **explicitly NOT discharged by FND-005** — it carries forward
   to FND-014. No local-Docker `db:migrate` validation is in scope here.

2. **`vercel.json` — RESOLVED: defer deploy config.** Create only
   `services/api/api/index.ts` (the Vercel Node serverless entry, named in the
   ticket). Do **not** author `vercel.json` or any other deploy config in
   FND-005. All deploy config (runtime declaration, `/api/*` rewrites, env var
   management, PR-preview wiring) is owned by **FND-012** (deployment ticket).
   Handoff note: FND-012 must add `vercel.json` and wire routing/rewrites so
   `/api/*` reaches `api/index.ts`; FND-005 leaves the entry file
   self-contained and typechecked but not deploy-wired.

3. **Vitest — RESOLVED: wire now.** Add `vitest` as a `services/api`
   devDependency, wire `"test": "vitest run"`, and ship the pure health-route
   test (no DB, no socket) as a deliverable of this ticket. This is the first
   API test and establishes the runner for the epic.
