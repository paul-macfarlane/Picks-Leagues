# Plan: FND-016 — CI pipeline (typecheck, lint, test, build, format:check)

**Status:** IMPLEMENTED
**Mode:** collaborative
**Author:** planner agent

## Goal

Stand up a GitHub Actions workflow (`.github/workflows/ci.yml`) that runs on
every PR and on push-to-`main`, gating merges on five parallel checks across
both workspace packages: typecheck, lint, test, build, and format:check.
Vercel continues to own preview deploys; this workflow only validates the
diff. The pipeline must finish in under five minutes on a no-change PR so it
never feels like the bottleneck on small changes.

## Acceptance criteria (from ticket, with clarifications)

- PR opens → CI runs all five jobs (typecheck, lint, test, build,
  format:check) in parallel for both packages.
  - "Both packages" is satisfied because `pnpm -r` already runs the script in
    every workspace package, and the root-level `eslint` / `prettier`
    invocations cover both packages from one entry point. No matrix-per-
    package is needed — running root scripts is the equivalent and matches
    how a developer runs them locally.
- Failures block merge (branch protection rule set on `main`).
  - The branch protection rule is a GitHub UI step the human applies once
    after the workflow has run successfully and registered the check names.
    Listed under § Verification.
- CI completes in under 5 minutes on a no-change PR.
  - Measurable on the Actions tab after the first PR runs. If the bound is
    exceeded, the most likely culprits are pnpm install time (mitigated by
    `cache: "pnpm"`) and the web `build` step (which runs `tsc --noEmit` +
    `vite build`); both are bounded by the codebase size at MVP.

## Approach / Design

### Workflow shape

A single new workflow file `.github/workflows/ci.yml`, distinct from
`api-client-check.yml`, with five jobs that run in parallel. Each job:

1. Checks out the repo.
2. Installs pnpm at version `10.33.4`.
3. Sets up Node `22` with `cache: "pnpm"`.
4. Runs `pnpm install --frozen-lockfile`.
5. Runs its one root-level pnpm script.

No shared "install" job. GitHub Actions cannot easily share a `node_modules`
tree between jobs, and the pnpm store cache (keyed on the lockfile) makes
re-installing in each job fast in practice. This is also the shape the
existing `api-client-check.yml` uses — staying consistent makes the workflows
easy to read side-by-side.

### Triggers

```yaml
on:
  pull_request:
  push:
    branches:
      - main
```

- `pull_request` covers the gating contract.
- `push: main` is the post-merge sanity net — catches the rare case where a
  merge commit produces a different result than the PR diff did (e.g., two
  PRs both pass on their own but break together after merge). Mirrors
  `api-client-check.yml`.

### Concurrency

```yaml
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true
```

- Same group key (`github.ref`) cancels in-flight runs when a new push lands
  on the same PR branch. Saves wall-clock time on rapidly-iterated PRs.
- Does not affect `main` (each push to `main` has a unique ref-sha pair under
  normal use; cancellation only applies within the same group/ref).

### Jobs

Each job runs `runs-on: ubuntu-latest` and shares the first four steps
(checkout, pnpm setup, node setup with cache, install). Pseudocode for one
job (full YAML is the implementer's output):

```
jobs:
  typecheck:
    name: Typecheck
    steps: [...shared..., run: pnpm typecheck]

  lint:
    name: Lint
    steps: [...shared..., run: pnpm lint]

  test:
    name: Test
    steps: [...shared..., run: pnpm test]

  build:
    name: Build
    steps: [...shared..., run: pnpm build]

  format-check:
    name: Format check
    steps: [...shared..., run: pnpm format:check]
```

| Job key        | `name:` (GitHub check name) | Script run            |
| -------------- | --------------------------- | --------------------- |
| `typecheck`    | Typecheck                   | `pnpm typecheck`      |
| `lint`         | Lint                        | `pnpm lint`           |
| `test`         | Test                        | `pnpm test`           |
| `build`        | Build                       | `pnpm build`          |
| `format-check` | Format check                | `pnpm format:check`   |

The five `name:` values become the GitHub status-check names. They are what
the human selects when configuring branch protection (see § Verification).

### What the workflow does NOT do

- No deploy step. Vercel's GitHub App handles preview deploys.
- No `gen:api:check` step. The existing `api-client-check.yml` workflow
  owns that concern and stays separate (decision confirmed).
- No knip / ts-prune dead-export check. Code-standards mentions running it in
  CI; not in the ticket's acceptance criteria. Out of scope here (could be a
  follow-up ticket, similar to how FND-019 owns the drift check).
- No matrix per package. Root scripts already cover both packages via
  `pnpm -r` / the root eslint and prettier configs.
- No artifact uploads. There is no consumer (Vercel handles its own build).
- No environment secrets. Tests use the env values injected by
  `services/api/vitest.config.ts`; nothing else needs a real secret.

## Verification (how we'll know it works)

1. **First-run check.** Push the branch and open the PR (or push directly to
   the branch the PR is open on, if already opened). All five jobs should
   appear in the PR's "Checks" tab and pass on a no-op codebase (current
   `main` already passes all five locally — confirm `pnpm format:check` is
   green before opening the PR).
2. **Timing check.** Open the Actions run; note the wall-clock duration of
   the slowest job. Target ≤ 5 minutes from job start to finish. If exceeded,
   the most likely fixes are (a) confirming `cache: "pnpm"` is hitting (look
   for "Cache restored" log lines), (b) tightening the `build` job if Vite's
   build is the long pole.
3. **Negative test (optional, recommended).** A throwaway commit that
   introduces a deliberate typecheck error, lint error, failing test, build
   break, or unformatted file — verify CI fails the matching job and that
   the other four still report status. Revert before merge. (The human can
   do this on the FND-016 PR itself or skip if confident the wiring is
   correct from inspection.)
4. **Branch protection.** Once the workflow has run at least once on
   `main` (i.e., post-merge), the human goes to **Settings → Branches →
   Branch protection rules → `main`** and:
   - Enables "Require status checks to pass before merging."
   - Adds the following six check names as required (decision confirmed):
     - **Typecheck**
     - **Lint**
     - **Test**
     - **Build**
     - **Format check**
     - **Verify generated API client is up to date** (from the existing
       `api-client-check.yml` workflow)
   - Keeps "Require branches to be up to date before merging" enabled (per
     existing repo convention; verify).
   - Leaves "Restrict who can push to matching branches" as-is.
   This step is a GitHub UI configuration, not a code change. Document the
   action in the PR description so it isn't forgotten at merge time.

## Risks

- **Wall-clock on the `build` job.** Web `build` runs `gen:api` (a tsx
  invocation that starts the API in-process) + `tsc --noEmit` + `vite build`.
  At MVP code size this should finish well under 2 minutes; at later epics
  it could grow. If we hit 5 minutes, splitting `build` into web-build /
  api-build (the latter being a no-op today) won't help — the long pole is
  the web build. Mitigation, if it ever matters: turbo/nx-style remote
  caching. Out of scope.
- **pnpm-store cache hit-rate.** First CI run will be a cold cache and slow
  (3-5 min just for install). Subsequent runs reuse the lockfile-keyed
  store and the install step drops to ~10-20s. The under-5-minute
  acceptance criterion should be measured on a warm-cache run, not the
  first run. Note in the PR description.
- **Concurrency group cancellation on `main`.** `cancel-in-progress: true`
  applied uniformly will cancel an in-flight `main` push if a new push
  lands on `main` while it's running. In MVP this is extremely rare (small
  team, squash-merge cadence) and the post-merge run on `main` is a
  sanity net — losing it occasionally is fine. If this becomes a problem
  we can scope the cancellation to PRs by keying on `github.head_ref`
  instead of `github.ref`. Recorded as a known minor tradeoff.
- **`api-client-check.yml` and `ci.yml` cohabiting.** Two workflows means
  two sets of "Install dependencies" steps per PR run. This is wasteful in
  cache-miss scenarios but cheap on warm cache. The ticket explicitly
  allows folding the staleness check into FND-016; the human's decision
  is to keep them separate for now (see § Decisions). If consolidation
  ever becomes worth the churn, it's a follow-up ticket.
- **Branch protection requires a human UI step.** The first time the
  workflow runs, the new check names don't exist in GitHub's protection
  picker. The human must let the workflow run on `main` once (after merge
  of this PR) before they can add the required-checks. Documented under
  § Verification; not a code concern.

## Decisions (confirmed by the human)

1. **Include `pnpm format:check` as the fifth parallel job.** Aligns with
   code-standards § Linting and formatting ("`pnpm format:check` runs in
   CI and blocks merge"). Adds a cheap (~30s warm-cache) job that closes
   the "developer ran a different Prettier version" footgun.
2. **Branch-protection required-checks list includes all five new jobs
   plus the existing `Verify generated API client is up to date` check
   from `api-client-check.yml`.** Six required checks total. The
   api-client check already exists and gates a real failure class;
   elevating it to required is a natural co-action even though it's
   outside FND-016's strict acceptance criteria.
3. **Keep `api-client-check.yml` as a separate workflow file — do not
   fold it into `ci.yml`.** It pre-dates FND-016, is known-green, and
   covers a different concern (artifact staleness, not source
   validation). Folding it in would reshape a working file without
   strong benefit. Any future consolidation is a separate small
   cleanup ticket.

## Assumptions (planner choices, override at review)

1. **Single `ci.yml` with five parallel jobs, not five separate workflow
   files.** Better at-a-glance — one CI file to read. The existing
   `api-client-check.yml` stays separate per § Decisions.
2. **`pnpm install --frozen-lockfile`** in every job, matching
   `api-client-check.yml`. Standard CI hygiene; refuses to install if
   `pnpm-lock.yaml` is out of sync with `package.json`.
3. **`cancel-in-progress: true` on the concurrency group.** Tradeoff
   accepted (occasional cancelled `main` run) per § Risks.
4. **No artifact uploads.** Vercel handles deploys; nothing else consumes
   build output.
5. **No `if` conditions on jobs.** Every PR runs every job. Path-filtered
   skipping (e.g., "skip the API tests if only `apps/web/**` changed") is
   premature optimization at MVP size and risks missing real cross-package
   regressions.
6. **`name:` field on each job.** Improves readability of GitHub's check
   list and pins the check names the human will reference in branch
   protection.
7. **Script name is `format:check`.** Matches code-standards § Linting and
   formatting. Implementer should verify the exact script name in root
   `package.json` and adjust the workflow `run:` line if it differs (e.g.,
   `format-check` vs `format:check`).
