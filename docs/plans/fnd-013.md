## Goal

Stand up the authorization guard that every future cron handler (epic 02's
`sync-schedule`, `sync-scores`, `process-elimination-deadlines`,
`send-pre-game-reminders`) will sit behind. After this ticket, any route mounted
under `/api/cron/*` is unreachable without the shared `CRON_SECRET` â€”
matching Vercel Cron's `Authorization: Bearer <secret>` convention so that when
epic 02 declares cron schedules in `vercel.json`, the platform-issued requests
authenticate against our handler with no extra wiring.

The ticket also drops a tiny `/api/cron/ping` route â€” both to satisfy the
acceptance criteria and to give epic 02 a known-good guarded route to compare
against. It stays in the codebase as a permanent smoke endpoint.

Why this matters per the MVP spec Â§ "Background Work â€” Vercel Cron + Idempotent
Handlers": *"All cron endpoints are protected â€” Vercel Cron sends a secret in
the `Authorization` header that the handler verifies. Unauthenticated requests
are rejected with 401. This prevents arbitrary external invocation of jobs that
mutate state."* This ticket establishes that contract before any state-mutating
job exists.

## Acceptance criteria (from ticket)

- Unauthenticated request to a sample `/api/cron/ping` returns 401.
- Authenticated request (correct secret) returns 200.
- Secret read from env, with a startup check that fails fast if missing.

## Approach

Step-by-step. Each step says WHAT changes and WHERE.

1. **Create `services/api/src/middleware/cron-auth.ts`** exporting a factory
   `createCronAuth({ secret?: string })` that returns a Hono middleware
   function. Behavior:
   - Read the `Authorization` header from the request.
   - Reject (`return c.json({ error: "Unauthorized" }, 401)`) when:
     - header is missing, OR
     - header does not start with `Bearer ` (case-sensitive prefix per RFC
       6750), OR
     - the token after `Bearer ` does not equal the configured secret under
       constant-time comparison.
   - On success, call `await next()`.
   - The factory takes an *optional* `secret` override so tests can inject a
     known value without touching `process.env`. When not provided, it reads
     `process.env["CRON_SECRET"]` once at factory-call time. If the value is
     missing/empty at that moment, the factory `throw`s with the same
     actionable message style as `db/client.ts` (mentions `.env.example`).
   - Constant-time comparison uses `crypto.timingSafeEqual` on equal-length
     Buffers. If the candidate's byte length differs from the secret's, return
     401 without calling `timingSafeEqual` (which throws on length mismatch).
     This is a real concern even on the server side â€” see Â§ Risks.
   - No `c.set('cronAuthenticated', true)` or similar context plumbing;
     "the handler ran" already proves auth passed. Keep it minimal.
2. **Create `services/api/src/middleware/cron-auth.test.ts`** with Vitest
   unit tests. Cases:
   - missing `Authorization` header â†’ 401, body `{ error: "Unauthorized" }`,
     and `next` not called.
   - header present but not `Bearer ...` (e.g., `"Basic xyz"`) â†’ 401.
   - `Bearer` with wrong secret â†’ 401.
   - `Bearer` with secret of different length than configured â†’ 401 (covers
     the `timingSafeEqual` length-guard branch).
   - `Bearer ` with empty token â†’ 401.
   - `Bearer <correct-secret>` â†’ `next()` invoked, downstream handler reached,
     200 returned.
   - Factory throws when `CRON_SECRET` is missing at construction time and no
     explicit secret is passed.
   - Tests use a minimal `Hono` app wrapping the middleware + a trivial handler,
     dispatched via `app.request(...)`. No need to spin up `createApp()` for
     these unit tests.
3. **Create `services/api/src/routes/cron-ping.ts`** containing a
   `createCronPingRoute()` factory that returns an `OpenAPIHono` sub-app
   created via `createOpenApiApp()`. It declares a single `GET /` route with
   `createRoute` returning `{ status: "ok", time: <iso8601> }` â€” the same
   shape as health, so the OpenAPI spec describes it for the typed client. The
   route accepts an injected `Clock` (default `clock` from `src/lib/clock`),
   matching the established pattern in `routes/health.ts`. The route does
   **not** apply the cron-auth middleware itself â€” the wiring happens in
   `createApp()` so the prefix-level guard is impossible to bypass by mounting
   the sub-app elsewhere.
4. **Create `services/api/src/routes/cron-ping.test.ts`** with an integration-
   shaped test that drives the full `createApp()` to confirm the guard is
   wired at the app layer (not just unit-tested in isolation). Cases:
   - `GET /api/cron/ping` with no auth header â†’ 401.
   - `GET /api/cron/ping` with wrong bearer â†’ 401.
   - `GET /api/cron/ping` with correct bearer â†’ 200, body has `status: "ok"`
     and an ISO timestamp.
   - `createApp` accepts an injected secret for testing (see step 5).
5. **Modify `services/api/src/app.ts`** to:
   - Extend `AppDeps` with an optional `cronSecret?: string` (test seam,
     same pattern as the existing `clock?: Clock`).
   - Build the cron-auth middleware via `createCronAuth({ secret: deps.cronSecret })`.
   - Apply it as a prefix guard: `app.use("/api/cron/*", cronAuth)`. This wires
     it on the parent app so it runs *before* any sub-app under `/api/cron/`
     dispatches â€” matching how Hono's pattern-scoped `app.use` is intended.
   - Mount the ping route: `app.route("/api/cron/ping", createCronPingRoute({ clock: resolvedClock }))`.
   - Order in the file: `app.use("/api/cron/*", ...)` must come *before*
     `app.route("/api/cron/ping", ...)` (Hono evaluates middleware/route
     registrations in registration order against the matched path).
6. **Modify `services/api/.env.example`** to add a `CRON_SECRET` entry with a
   short comment explaining that it's a long random string shared with Vercel
   Cron, and that locally any non-empty value works.
7. **Modify `services/api/README.md`** to add a brief "Cron endpoints" section
   under "API server" describing:
   - The `/api/cron/*` prefix is protected by `CRON_SECRET`.
   - Format: `Authorization: Bearer <CRON_SECRET>`.
   - Vercel Cron uses this header automatically when schedules are declared
     in `vercel.json` (epic 02).
   - How to hit `/api/cron/ping` locally with `curl -H "Authorization: Bearer $CRON_SECRET" http://localhost:3000/api/cron/ping`.
8. **Update `docs/backlog/01-foundation.md`** â€” flip FND-013's `**Status:**`
   line from `IN_PROGRESS` to `IN_REVIEW` once implementation lands. (This is

## Risks and open questions

- **Constant-time compare in Node.** `crypto.timingSafeEqual` requires both
  arguments to be `Buffer`/`Uint8Array` of identical length and throws otherwise.
  The middleware must check length first and short-circuit to 401 â€” leaking
  "wrong length" via a different error path is acceptable here because length
  is a single byte of information and not the secret itself, but the throw-vs-
  return distinction matters for code correctness. The plan handles this
  explicitly in step 1.
- **Why this guard is at the route layer, not in `domain/`.** Per
  `code-standards.md` Â§ "Backend architecture: domain â†’ repositories â†’ routes":
  authentication is a route-layer concern. Domain functions operate on already-
  authorized inputs. Putting cron-auth in `middleware/` (which routes consume)
  is the correct layer. Domain isolation is unaffected â€” `middleware/` is not
  one of the restricted-from paths.
- **Header case-sensitivity.** Hono's `c.req.header("Authorization")` is
  case-insensitive on the header *name* per Fetch API semantics. The middleware
  treats the `Bearer ` *value* prefix as case-sensitive per RFC 6750. Vercel
  Cron sends `Bearer ` with that exact casing; this is consistent.
- **No `c.set()` of an "isCron" flag.** Other route handlers don't need to
  know "this came from cron" â€” they're mounted under `/api/cron/*` and that
  fact is the marker. If a future ticket needs to differentiate further, that
  ticket can add the context value; speculatively adding it now is YAGNI.

## Assumptions (autonomous mode only â€” flagged for human review even though
this is collaborative mode)

These are the choices made without explicit instruction beyond the orchestrator
brief; the human can override any of them at plan-review time:

1. **`/api/cron/ping` stays in the codebase permanently** as a smoke test, not
   removed at the end of the ticket. The orchestrator hinted at this; I'm
   committing to it explicitly. (Override: remove the file + its test + the
   `app.route(...)` line.)
2. **Ping route response shape mirrors `/api/health`** (`{ status: "ok", time }`)
   because there is already client-side type machinery (TanStack Query, OpenAPI
   client) that benefits from a familiar response. Easy to change to a literal
   `{ pong: true }` if preferred.
3. **`CRON_SECRET` read happens inside the middleware factory (`createCronAuth`)
   at construction time**, called from `createApp()`. This means the throw fires
   when `createApp()` is called â€” which is at module load for the dev server
   (`src/index.ts`) and for the Vercel function entry (`src/vercel-entry.ts`).
   Effectively, the API process refuses to start without `CRON_SECRET`. That is
   the "fail fast" the ticket wants, and it matches `db/client.ts`'s posture for
   `DATABASE_URL`. (Override: defer the env read to first request, with a
   lazy-init guard. I do *not* recommend this â€” the ticket explicitly says
   "fails fast if missing.")
4. **No CI env-var assertion added in this ticket.** The local + production
   deploys are what enforce the env presence. CI runs unit tests with secrets
   injected by the test (no real env required). If we later want CI to assert
   that production env vars exist, that's a separate concern. (Override: add an
   `.env.example`-vs-code drift check; not in scope per the ticket.)
5. **The middleware is *not* declared via `@hono/zod-openapi` `createRoute`.**
   Middleware doesn't appear in OpenAPI; only the routes it guards do. The
   `cron-ping` route *is* declared via `createRoute`, and its 401 response can
   optionally be documented in its OpenAPI `responses` block. The plan does *not*
   add the 401 to the OpenAPI doc because the guard is a prefix-level concern
   that applies to many future routes â€” documenting it once on every route is
   noise. (Override: add `401: { description: "Missing or invalid cron secret",
   content: { "application/json": { schema: <UnauthorizedSchema> } } }` to the
   `createRoute` call and create a shared `UnauthorizedSchema` in `src/lib/openapi.ts`.)
6. **Test injection point is `createApp({ cronSecret })`**, mirroring the
   existing `createApp({ clock })`. The middleware factory itself also accepts a
   secret directly, used by the unit tests. (Override: use `process.env` mutation
   in tests via `vi.stubEnv`. I find the explicit-injection pattern cleaner and
   it matches the existing convention.)
