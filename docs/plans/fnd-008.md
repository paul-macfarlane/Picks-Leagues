# Plan: FND-008 — Typed API client generation

**Status:** IMPLEMENTED
**Mode:** collaborative
**Author:** planner agent

## Goal

Give the web app a single, fully-typed door into the API. FND-007 already serves a
live OpenAPI 3.0 document at `GET /api/openapi.json` (generated in-process by
`@hono/zod-openapi`, no static artifact). This ticket adds a `pnpm gen:api` script
that consumes that spec and emits a typed TypeScript client into
`apps/web/src/lib/api-client/`, wires it into web `pnpm dev`, and adds a CI gate
that fails when the checked-in client drifts from the spec. This is the mechanism
behind the code-standards rule "All API calls go through the generated typed
client. No raw `fetch` in feature code" — every later epic's frontend work depends
on it. FND-011 (TanStack Query demo against `/api/health`) is the first real
consumer and is the reason "typed functions for health and echo" is the acceptance
bar here.

## Acceptance criteria (from ticket)

- `pnpm gen:api` regenerates the client from the running API.
- Client exposes typed functions for health and echo.
- CI fails if the checked-in client is stale relative to the spec.

## Pre-existing state (verified, not assumed)

- Spec is **live-only**: `app.doc("/api/openapi.json", ...)` in
  `services/api/src/app.ts`. There is no spec file on disk and FND-007 deliberately
  added no build step (`services/api/README.md` § OpenAPI spec).
- `createApp()` (`services/api/src/app.ts`) builds the whole Hono app in-process and
  is already used by route tests via `app.fetch(...)` — the spec can be obtained
  without binding a TCP port.
- Routes in spec today: `GET /api/health` (`HealthResponse`), `POST /api/echo`
  (`EchoHeaders`/`EchoQuery`/`EchoRequest`/`EchoResponse` + `ValidationError`).
- `apps/web/src/lib/api-client/**` is **already** ignored by ESLint
  (`eslint.config.js` global ignores) and Prettier (`.prettierignore`). The
  generated-client location is therefore a settled team decision — generate there.
- `apps/web` is still a bootstrap stub (`src/index.ts` only); Vite/React arrives in
  FND-009. The client must typecheck under `apps/web/tsconfig.json`
  (`moduleResolution: "Bundler"`, `strict`, `noUncheckedIndexedAccess`, `types: []`,
  `lib` includes DOM) without a running web app.
- No `.github/` directory exists yet. The full CI pipeline is FND-016; this ticket
  must stand up the minimal CI gate for staleness and shape it so FND-016 absorbs it.
- Root scripts run package scripts via `pnpm -r` / `--filter`; `gen:api` is a new
  root script. Tooling-only TS (e.g. `drizzle.config.ts`) is typechecked via a
  separate `tsconfig.tooling.json` — follow that precedent for the generator script.

## Approach

### 1. Generator + dependency choice

Use **`openapi-typescript`** (spec → `types.ts`) plus **`openapi-fetch`** (tiny
typed runtime client over `fetch`). Rationale:

- Ticket explicitly allows `openapi-typescript` or `orval`.
- `openapi-fetch` gives typed call sites (`client.GET("/api/health")`,
  `client.POST("/api/echo", { body, params })`) with full request/response/error
  typing pulled from the generated `paths` — this satisfies "typed functions for
  health and echo" without generating per-operation wrappers we'd have to maintain.
- No React/TanStack-Query coupling now (that integration is FND-011); `openapi-fetch`
  has a separate `openapi-react-query` adapter FND-011 can add later without
  regenerating anything.
- `openapi-typescript` output is deterministic given the same spec → reliable
  staleness diff. `orval` pulls a heavier dependency tree and config surface we
  don't need for two routes.

Add as **devDependencies of `apps/web`**: `openapi-typescript`. Add `openapi-fetch`
as a **runtime dependency of `apps/web`** (it ships in the browser bundle). Pin
exact versions (no `^`) so the generated output is reproducible across machines and
CI — staleness checks are only meaningful with a pinned generator.

### 2. Spec-export helper in the API package

Create `services/api/src/lib/openapi-document.ts`:

- Export `getOpenApiDocument()` that builds the app via `createApp()` and returns
  the OpenAPI JSON object by calling the in-process route
  (`await createApp().fetch(new Request("http://local/api/openapi.json"))` then
  `.json()`), or by factoring the doc config so it can be produced without HTTP.
  Prefer the `app.fetch` path — it exercises the exact same code path as production
  and keeps a single source of truth for the doc metadata in `app.ts`.
- Explicit return type (code-standards: exported functions have explicit return
  types). Type the return as `unknown` and let the consuming script treat it as the
  JSON to serialize — we do not need a precise OpenAPI type here.

This lives in `lib/`, imports only `./app`, and stays out of `domain/`/`scoring/`
so the isolation ESLint zones are unaffected.

### 3. Generation script

Create `services/api/scripts/gen-api-client.ts` (tooling script, run with `tsx`):

1. Call `getOpenApiDocument()`.
2. Write the spec to `apps/web/src/lib/api-client/openapi.json` (committed — it is
   the diff anchor for the staleness check and lets the generator and CI run
   without booting the API).
3. Invoke `openapi-typescript` against that JSON, emitting
   `apps/web/src/lib/api-client/types.gen.ts`.
4. The hand-authored client-construction module is **not** part of generation and
   **not** written by this script. It lives at `apps/web/src/lib/api.ts` —
   **outside** the ESLint/Prettier-ignored `api-client/` directory so it is linted
   and formatted like normal source — and imports the generated `paths` type from
   the ignored `apps/web/src/lib/api-client/types.gen.ts` (see step 4). No
   `index.ts` is created inside `api-client/`; only truly generated files
   (`openapi.json`, `types.gen.ts`) live in the ignored directory. (Decided — see
   Decisions, Q1.)
5. Resolve all paths from the script file via `import.meta.url` / `node:path` so
   `pnpm gen:api` works regardless of cwd (harness resets cwd between calls).

The script must be a no-network operation: it imports `createApp` directly and uses
`app.fetch`. It never requires `pnpm dev` to be running. (The ticket phrase
"regenerates the client from the running API" is satisfied by running the real app
in-process — this is strictly more reliable in CI than curling a dev server and is
consistent with how route tests already consume the app.)

Add to `apps/web/scripts/`? No — the script imports from `services/api`, so it
belongs in `services/api/scripts/` and is owned by the API package's tooling
tsconfig.

### 4. The committed client surface

Generated (committed) files in `apps/web/src/lib/api-client/`:

- `openapi.json` — committed spec snapshot (generated; diff anchor).
- `types.gen.ts` — generated by `openapi-typescript` (`paths`, `components`).

Hand-authored client-construction module `apps/web/src/lib/api.ts` (linted,
formatted — lives outside the ignored `api-client/` directory):

  - `import createClient from "openapi-fetch"; import type { paths } from "./api-client/types.gen";`
  - Export `apiClient = createClient<paths>({ baseUrl })`.
  - `baseUrl` comes from `import.meta.env.VITE_API_BASE_URL` with a localhost
    default (`http://localhost:3000`). `import.meta.env` typing is a Vite concern
    that lands in FND-009; until then guard the read so `apps/web` typecheck
    (`tsc -p apps/web/tsconfig.json`, `types: []`, no Vite client types) still
    passes. Concretely: read via `import.meta.env?.VITE_API_BASE_URL` is not
    type-safe under bare tsc — instead define a minimal local
    `declare global { interface ImportMeta { env?: Record<string,string|undefined> } }`
    ambient in this dir, OR fall back to a plain `const baseUrl = "http://localhost:3000"`
    constant now and let FND-009/FND-012 introduce env-driven config. **Decision:
    ship the constant now**, leave a one-line WHY comment, and let FND-012 (deploy
    config) own env-driven base URL. This keeps FND-008 self-contained and
    typecheck-clean.
  - Re-export the generated `paths`/`components` types for consumers (so feature
    code imports types from `lib/api.ts`, never reaching into the ignored dir).

"Typed functions for health and echo" acceptance is met because
`apiClient.GET("/api/health")` and `apiClient.POST("/api/echo", ...)` are fully
typed from `paths`. **Decision (Q2): expose the typed `apiClient` only — no named
`getHealth()` / `postEcho()` wrappers.** FND-011 will surface the real ergonomic
need; wrappers are trivial to add later if wanted.

### 5. Root + package scripts

- `services/api/package.json`: add

- `services/api/package.json`: add `"gen:api:check"` that runs `gen:api` then fails

  the check command in the API package (it owns the generator) and expose a root
  alias `"gen:api:check"` for CI ergonomics.

### 6. Hook into web `pnpm dev`

`apps/web/package.json` `dev` is currently a placeholder echo (real dev server is
FND-009). To satisfy "Hook into web `pnpm dev`" without prematurely owning FND-009's
Vite setup:

- Change `apps/web` `dev` to `"pnpm run gen:api && echo '@picksleagues/web: no dev server yet (FND-009)'"`,
  and add `apps/web` script `"gen:api": "pnpm --filter @picksleagues/api gen:api"`
  (or call the root script). This regenerates the client whenever the web dev
  process starts.
- Document in the plan handoff that FND-009 must preserve a `predev`/`dev` hook that
  runs `gen:api` before Vite (`"predev": "pnpm gen:api"` is the natural shape once
  Vite exists). Leave a one-line note in `services/api/README.md`'s FND-008 section
  pointing at this so FND-009's implementer doesn't drop it.

Note the root `pnpm dev` is `pnpm -r --parallel dev`; the web package's own `dev`
running `gen:api` first is the correct seam (per-package, not global), matching
code-standards "configured per route/package, not globally" philosophy.

### 7. CI staleness gate

No `.github/workflows` exists. Create
`.github/workflows/api-client-check.yml`:

- Trigger: `pull_request` (and `push` to `main` as a safety net).
- Steps: checkout → setup pnpm + Node 22 (matches root `engines`) → `pnpm install
  --frozen-lockfile` → `pnpm gen:api` → `git diff --exit-code -- apps/web/src/lib/api-client`.
  A non-empty diff means the committed client is stale → job fails with a message
  telling the dev to run `pnpm gen:api` and commit.
- Keep this workflow **small and single-purpose** so FND-016 (full CI pipeline) can
  either fold this job into its matrix or keep it as a standalone gate. Add a
  comment header in the YAML noting FND-016 may absorb it.

Equivalent local command: root `package.json`
`"gen:api:check": "pnpm gen:api && git diff --exit-code -- apps/web/src/lib/api-client"`
so the same gate is runnable locally and reused by the workflow.

## Tests

Code-standards: repositories/tooling are not unit-tested; business logic is. The
generator is tooling glue with no business rules, so no Vitest suite for the script
itself. Coverage strategy:

- **`services/api/src/lib/openapi-document.test.ts`** (Vitest, pure, no DB/HTTP
  server — uses `createApp().fetch` like existing route tests):
  - `getOpenApiDocument() returns an OpenAPI 3.0 document with info.title "Picks Leagues API"`
  - `document contains GET /api/health path with HealthResponse 200 schema`
  - `document contains POST /api/echo path with EchoRequest body and ValidationError 400`
  - `document.components.schemas includes HealthResponse, EchoResponse, ValidationError`

  This is the meaningful regression net: if a future route breaks spec generation,
  this test fails before the stale-client CI gate would.
- The **CI staleness workflow itself** is the functional test for "client matches
  spec"; no extra unit test reproduces it.
- No test added under `apps/web` (still a stub; React Testing Library setup is
  FND-009/FND-010). The generated client's correctness is a compile-time guarantee
  via `apps/web` `tsc --noEmit`, which already runs in the per-package `typecheck`.

Manual verification checklist for the implementer/human (collaborative-mode test
phase):

- `pnpm gen:api` succeeds from repo root with no API server running.
- Re-running `pnpm gen:api` produces a clean `git diff` (idempotent / deterministic).
- `pnpm --filter @picksleagues/web typecheck` passes with the new client.
- `apps/web/src/lib/api.ts` is linted/formatted (it is outside the ignored
  `api-client/` dir) — `pnpm lint` and Prettier cover it; the generated files in
  `api-client/` remain ignored.
- `apiClient.GET("/api/health")` and `apiClient.POST("/api/echo", ...)` typecheck
  with full request/response/error inference; no `getHealth()`/`postEcho()`
  wrappers exist (apiClient-only surface, per Q2).
- `pnpm gen:api:check` exits 0 on a clean tree, non-zero after editing a route
  schema without regenerating.

## Risks and open questions

No open questions — Q1 and Q2 are resolved (see Decisions). Remaining risks:

- **Generator/runtime split.** `openapi-typescript` is dev-only; `openapi-fetch`
  ships to the browser. Both belong to `apps/web` (the consumer), not `services/api`,
  even though the *script* runs from `services/api`. The script invokes
  `openapi-typescript` — it must resolve from `apps/web`'s node_modules; in a pnpm
  workspace a root-level or web-level devDep is reachable. Implementer should invoke
  the generator via its Node API (`import { astToString } ...` / programmatic) or
  via `pnpm --filter @picksleagues/web exec openapi-typescript` to keep resolution
  unambiguous. Mild risk; verify resolution during implementation.
- **CI before FND-016.** This ticket creates the first GitHub Actions workflow.
  Branch protection / required-checks wiring is FND-016's job; this workflow will
  run but won't *block* merge until FND-016 (or the human) marks it required. The
  acceptance criterion "CI fails if stale" is met (the job fails); enforcement as a
  merge gate is FND-016. Called out so it isn't mistaken for incomplete work.
- **`import.meta.env` typing.** Plan ships a hardcoded localhost `baseUrl` constant
  now to keep `apps/web` typecheck clean without Vite types; env-driven base URL is
  deferred to FND-012. If the reviewer wants env config in this ticket, that pulls
  in ambient typing work that's cleaner once Vite exists (FND-009).
