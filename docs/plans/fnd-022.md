# Plan: FND-022 — Per-PR preview environments (Neon branches + OAuth)

**Status:** IMPLEMENTED
**Mode:** collaborative
**Author:** planner agent

## Goal
Make every PR's Vercel preview deploy a fully usable, isolated environment instead of just a
static render. Each PR gets its own Neon database branch (forked from a baseline, migrated by the
existing deploy-time migrate step, and deleted when the PR closes), so reviewers can exercise sign-in,
league creation, and picks destructively without ever touching production data. OAuth completes on a
single stable, shared preview origin with the session cookie scoped where the SPA can see it
(the same `:5173` cookie-scoping lesson from local dev, applied to the shared preview origin). This
unblocks reviewers testing real flows on a PR before merge and is a prerequisite for preview-based
smoke tests (POL-003).

## Acceptance criteria (from ticket)
- On PR open/update, a Neon branch is created (forked from a baseline branch) and migrations run against it.
- The preview deploy's `DATABASE_URL` points at that PR's Neon branch; no preview ever reads/writes production data.
- Neon branch is deleted when the PR is closed/merged (no orphaned branches).
- Google OAuth callback registered for the preview origin; sign-in completes on the shared preview origin and the session cookie is visible to the SPA.
- `BETTER_AUTH_URL` (and any origin-dependent env) resolves to the preview origin on previews.
- Env-var provisioning runs through the existing hand-rolled GHA→Vercel path (no Vercel GitHub integration button).
- Documented runbook: how a reviewer signs in and exercises a preview, and how branches are cleaned up.

## How this repo ACTUALLY deploys — three tensions to reconcile up front

I investigated `.github/workflows/` (`ci.yml`, `schema-drift-check.yml`, `api-client-check.yml`),
`vercel.json`, `scripts/build-vercel-output.sh`, `services/api/src/db/migrate.ts`,
`services/api/src/auth/index.ts`, `docs/deploy.md`, and project memory. Findings:

### Tension 1 — "the existing hand-rolled GHA→Vercel deploy path" does not exist as described
There is **no GHA workflow that deploys to Vercel**. All three workflows trigger on `pull_request`
and only run lint/typecheck/test/build/format/drift checks. **Deploys are driven by Vercel's own Git
integration** (`docs/deploy.md` §"Deploy workflow": a PR triggers a Vercel preview build; merge to
`main` triggers a production build). The "hand-rolled" memory refers to the *build* being hand-rolled
(Build Output API in `scripts/build-vercel-output.sh`), **not** a GHA-orchestrated deploy. FND-020's
plan flagged exactly this. So the acceptance criterion "env-var provisioning runs through the existing
hand-rolled GHA→Vercel path" describes a path that isn't there.

**Recommendation (flagged):** Build a **new GitHub Actions workflow pair** that owns the per-PR Neon
lifecycle and pushes per-deploy env vars to Vercel via the Vercel CLI/REST API. The Vercel Git
integration still does the actual deploy; our workflow runs *before* that deploy and sets the env vars
the deploy will read. This is the realistic interpretation of "hand-rolled GHA→Vercel path" — we are
hand-rolling the env-var provisioning in GHA rather than clicking Vercel's GitHub-integration env UI.
We do **not** introduce a parallel GHA→Vercel *deploy* job (that would duplicate the Git integration and
fight FND-020's build-time migrate gate). I recommend updating the ticket wording from "the existing
hand-rolled GHA→Vercel deploy path" to "a GHA workflow that provisions per-preview env vars before
Vercel's Git-integration build."

### Tension 2 — Vercel env vars are project/branch-scoped, not per-deploy, when set via dashboard
Vercel's standard env-var model scopes a variable to an environment (Preview/Production) or to a
specific git branch — **not to an individual deploy.** A single shared "Preview" `DATABASE_URL` can't
give PR-A and PR-B different branch URLs. With Option B (chosen) only `DATABASE_URL` needs to be per-PR;
`BETTER_AUTH_URL` is a single fixed preview-scoped value (see Tension 4). The mechanism for per-PR
`DATABASE_URL`:

- **(Chosen) Git-branch-scoped env var via the Vercel REST API.** Vercel lets an env var be scoped to
  a specific git branch (`gitBranch` field on the env-var API). Our GHA workflow, on PR open/sync,
  creates the Neon branch and then `POST`s **only `DATABASE_URL`** to the Vercel project scoped to that
  PR's head branch (`target: ["preview"]`, `gitBranch: <pr-head-ref>`). When Vercel's Git integration
  then builds the preview for that branch, it injects that branch-scoped `DATABASE_URL`. Cleanup deletes
  both the Neon branch and the branch-scoped `DATABASE_URL` env var on PR close.

- **(Rejected for `DATABASE_URL`) Deriving everything at build time inside `build-vercel-output.sh`.**
  We can't mint a Neon branch URL from inside the Vercel build without a Neon token in the Vercel build
  env and branch-name→URL logic there; that puts the Neon admin token into Vercel's build environment
  and couples branch lifecycle to build success. Keeping branch lifecycle in GHA (which already has the
  PR open/close events) is cleaner and keeps the Neon token in GitHub secrets only.

### Tension 3 — `BETTER_AUTH_URL` precedence and the runtime `VERCEL_URL` fallback
`services/api/src/auth/index.ts` reads `process.env["BETTER_AUTH_URL"]` and passes it to Better Auth as
`baseURL`, which Better Auth uses to build OAuth callback URLs and to scope the session cookie. Vercel
injects `VERCEL_URL` (the deploy's own host) and `VERCEL_ENV` into **both the build and the function
runtime**, so the per-deploy origin is available *inside the deploy*.

**Recommendation (with Option B):** keep a runtime `VERCEL_URL` fallback in `resolveAuthBaseURL`, but the
fixed preview-scoped `BETTER_AUTH_URL` (the shared stable preview origin) **wins** on previews. The
`VERCEL_URL` fallback only applies to deploys reached at their raw per-deploy host for non-OAuth
verification. Precedence (preserved exactly):

`explicit deps.baseURL` → `process.env["BETTER_AUTH_URL"]` → `https://${VERCEL_URL}` → **throw**.

This keeps the throw for local/prod where neither is set, and re-applies the local `:5173` cookie-scoping
lesson (cookie scoped to the origin the SPA runs on — here, the shared preview origin). See Tension 4 for
why OAuth specifically depends on the shared origin.

### Tension 4 — dynamic preview URL ↔ Google's exact-match redirect URIs (the hard constraint)
Google OAuth requires **exact** authorized redirect URIs — no wildcards, no `*.vercel.app`. Better Auth's
Google callback is `<baseURL>/api/auth/callback/google`. If `baseURL` were the per-deploy random host, the
callback URI would change every deploy and could never be pre-registered in the Google console. This is the
central conflict in this ticket. Options:

- **(B — CHOSEN) One shared, stable preview origin with a single registered OAuth redirect URI.** There is
  ONE stable preview origin (a fixed Vercel preview subdomain / alias) registered ONCE in the Google and
  Discord consoles as `<stable-preview-origin>/api/auth/callback/google` (and `/discord`). The Preview
  environment's `BETTER_AUTH_URL` is a single fixed preview-scoped Vercel env var equal to that stable
  origin (NOT per-branch). Simplest for OAuth (one console entry, minimal maintenance). Loses per-PR
  isolation of the *frontend* origin — fine here because this is a solo project and reviewers test one PR
  at a time. **Per-PR isolation is at the DATABASE layer (the Neon branch), not the frontend origin.**
- **(A — considered, rejected) A stable per-branch preview alias.** Provision a Vercel branch-scoped domain
  alias per active PR and set a branch-scoped `BETTER_AUTH_URL` to that alias. Gives each PR its own working
  OAuth origin, but multiplies console/alias maintenance per PR. **Rejected for maintenance simplicity**
  given a solo dev testing one PR at a time.
- **(C) Register the random URL per deploy via the Google Admin SDK.** Technically possible but requires a
  GCP service account with directory/admin scope to mutate OAuth client redirect URIs programmatically,
  is brittle, rate-limited, and a security footgun. **Rejected.**

**Decision (locked):** Option **B — one shared, stable preview origin** is the IMPLEMENTED approach.
Concretely:
- ONE stable preview origin (fixed Vercel preview subdomain / alias) is registered ONCE in the Google and
  Discord consoles: `<stable-preview-origin>/api/auth/callback/google` and `.../discord`.
- `BETTER_AUTH_URL` for the **Preview** environment is a single fixed preview-scoped Vercel env var equal to
  that stable origin. It is **NOT** pushed per-branch by `preview-env.yml`; `preview-env.yml` pushes only
  the per-branch `DATABASE_URL`.
- `resolveAuthBaseURL` keeps the runtime `VERCEL_URL` fallback, but explicit `BETTER_AUTH_URL` still wins —
  so on previews the fixed preview value wins, and `VERCEL_URL` is only a fallback for deploys reached at
  their raw per-deploy host for non-OAuth verification. Precedence preserved:
  `explicit deps.baseURL` → `BETTER_AUTH_URL` env → `https://${VERCEL_URL}` → throw.
- **Constraint to document:** because the frontend origin is shared, OAuth sign-in is exercised one-PR-at-a-
  time through the shared preview origin; per-PR isolation is at the DATABASE layer (Neon branch), not the
  frontend origin. The reviewer runbook says "sign in via the shared preview origin."
- **Implementation detail to resolve + document:** the implementer must determine and document the exact
  mechanism for the stable preview subdomain (e.g. a Vercel preview alias assigned to the branch under test,
  or a dedicated stable preview domain). Flag that this needs the **human to pick/confirm the actual
  hostname and register it in the Google + Discord OAuth consoles** before live verification.

### Finding — no Neon automation exists today
There is **zero Neon branching tooling** in the repo: no Neon API token, no project-ID config, no
`@neondatabase/api-client`, no GHA step, no `scripts/` for branch create/delete. The only Neon coupling is
the runtime driver selection in `services/api/src/db/client.ts` / `migrate.ts` (`isLocalDatabaseUrl`).
Everything Neon-branch-related in this ticket is net-new and depends on human-provisioned secrets.

2. **Vercel API token + IDs.** Create a Vercel access token. Capture `VERCEL_ORG_ID` and `VERCEL_PROJECT_ID`.

1. **Modify `services/api/src/auth/index.ts`** so `baseURL` resolution becomes:

2. **Consider `trustedOrigins`** for Better Auth so that the shared preview origin is trusted even when a

3. **Update `services/api/.env.example`** comment block to explain the new precedence (`VERCEL_URL`

### Phase 2 — CI: per-PR Neon branch lifecycle + Vercel `DATABASE_URL` provisioning
4. **Create `.github/workflows/preview-env.yml`** triggered on `pull_request`
   (`types: [opened, reopened, synchronize]`). Steps:
   - Guard: if the required secrets (`NEON_API_KEY`, `NEON_PROJECT_ID`, `NEON_PARENT_BRANCH`, `VERCEL_TOKEN`,
     `VERCEL_ORG_ID`, `VERCEL_PROJECT_ID`) are absent, **fail clearly** (or short-circuit with an explicit
     "secrets not provisioned yet — see runbook" message) rather than mis-provisioning. Until the human
     provisions secrets post-merge, this workflow is expected to fail/skip — acceptable for this PR.
   - Compute a deterministic branch name: `preview/pr-${{ github.event.number }}`.
   - Create-or-reuse the Neon branch via the Neon API (recommended: the official `neondatabase/create-branch-action`,
     or a small `scripts/neon-branch.ts` calling `@neondatabase/api-client` with `NEON_API_KEY` /
     `NEON_PROJECT_ID`, parent = `NEON_PARENT_BRANCH`). Output the branch's **pooled** connection string.
   - Push **only `DATABASE_URL`** (the new branch's pooled URL) to Vercel via the Vercel REST API
     (`POST /v10/projects/{id}/env`), scoped `target: ["preview"]`, `gitBranch: <pr head ref>`, `type: encrypted`.
     Use upsert semantics (delete-then-create or PATCH by id) so PR `synchronize` events don't error on duplicate keys.
     **Do NOT push a per-branch `BETTER_AUTH_URL`** — under Option B it is a single fixed preview-scoped value
     set once in Vercel (Phase 0 step 5).
   - **Do not run migrations here.** Migrations run inside Vercel's build via the FND-020 hook
     (`scripts/build-vercel-output.sh` → `db:migrate`), which will now see the per-branch `DATABASE_URL`.
     This is the integration point the ticket calls out: the migrate-on-deploy hook automatically targets
     the PR's branch because we set the branch-scoped `DATABASE_URL` *before* the Vercel build runs.
     (Sequencing caveat in Risks: ensure env vars land before Vercel starts the build.)
5. **Create `.github/workflows/preview-cleanup.yml`** triggered on `pull_request` `types: [closed]`

   - Same missing-secret guard / clear-fail behavior as `preview-env.yml`.
   - Delete the Neon branch `preview/pr-${{ github.event.number }}` via the Neon API (idempotent — ignore
     404).
   - Delete the branch-scoped Vercel `DATABASE_URL` env var for that git branch. (No per-branch
     `BETTER_AUTH_URL` to delete under Option B.)

## Re-enabling Vercel preview deploys (explicit, tracked step)

**Current state:** previews are TEMPORARILY DISABLED by the user. I cannot read Vercel dashboard state from
the repo, so the plan documents the exact toggles and asks the implementer/human to confirm which one was used.
The likely place(s) previews get disabled:

- **Vercel → Project → Settings → Git → "Preview Deployments" / Ignored Build Step.** Previews can be
  suppressed via (a) toggling off preview deployments for non-production branches, or (b) a
  **"Ignored Build Step"** command that exits 0 to skip the build, or (c) Project → Git → only deploying the
  production branch. Re-enable by: ensure non-production branch deployments are ON, and clear any Ignored
  Build Step override (set it back to "Automatic" / empty).
- **Deploy Hooks / GitHub integration paused.** Vercel → Settings → Git → ensure the GitHub connection is
  active and "Create Deployments for all branches/PRs" is enabled.

**Likely reason it was disabled:** cost/noise control (a preview build per push) or to avoid broken preview
builds while FND-020's migrate-on-deploy hook had no isolated DB to migrate (every preview was migrating a
shared preview DB). FND-022 removes that reason by giving each PR its own branch.

**Runbook step (tracked post-merge; the human checks this off):**
1. Vercel → Project → Settings → Git: confirm GitHub repo connected and preview deployments enabled for PRs.
2. Settings → Git → Ignored Build Step: set to "Automatic" (clear any custom skip command).
3. Open a throwaway PR and confirm a preview build is triggered and `preview-env.yml` runs first.

The implementer should record in the plan's implementation notes **which** toggle was actually used to
disable previews (the human will confirm), so the re-enable instruction is exact rather than a menu.

## Risks and open questions
1. **GHA → Vercel env-var sequencing race (highest risk).** Vercel's Git integration may start the preview
   build the instant the PR push lands, *before* `preview-env.yml` finishes setting the branch-scoped
   `DATABASE_URL`. If so, the first build of a brand-new PR could build against a stale/absent env var, and
   FND-020's migrate step could fail or hit the wrong DB. Mitigations to evaluate during implementation:
   (a) use an **Ignored Build Step** that waits for / checks env readiness, (b) have `preview-env.yml`
   trigger the Vercel deploy itself via the Vercel CLI *after* provisioning (deploy hook), accepting more
   coupling, or (c) document that the first push may need a re-trigger. This is the riskiest unknown; flag
   for the reviewer.
2. **OAuth shared-origin constraint (Tension 4, Option B).** Because the preview frontend origin is shared,
   only one PR at a time is exercised through it for OAuth sign-in; per-PR isolation is at the Neon-branch
   (DB) layer, not the frontend origin. Acceptable for a solo dev testing one PR at a time. The exact stable
   preview hostname must be picked/confirmed by the human and registered in the Google + Discord consoles.
3. **Neon branch limits / cost.** Neon free/launch tiers cap concurrent branches and compute hours. Many open
   PRs → many branches. Cleanup-on-close mitigates orphans, but stale long-lived PRs accumulate. Recommend a
   scheduled sweeper (out of scope; note as follow-up) and forking from a small `preview-baseline`. User is
   cost-conscious: each preview branch consumes Neon compute when its serverless function queries it — fine at
   solo scale, monitor.
4. **GitHub Actions cost.** Repo is public → Actions are free/unlimited (per project memory), so the two new
   workflows add no billing concern. The cost lever is Neon branch compute, not CI minutes.
5. **Pooled vs. direct Neon URL.** `db:migrate` and the runtime both expect the **pooled** connection string
   for Neon (matches `client.ts` driver selection). Ensure the branch-create step outputs the pooled URL.
6. **Secret exposure on public repo PRs.** `pull_request` workflows from forks do NOT get repo secrets by
   GitHub default — good (prevents secret theft) but means fork PRs won't get preview envs. Since this is a
   solo project (branches pushed to the same repo, not forks), acceptable; document the limitation.
7. **Workflows committed before secrets exist.** Both workflows are committed in this PR but cannot run green
   until the human provisions secrets post-merge. This is expected and acceptable; the workflows must
   fail-clearly / be guarded on missing secrets rather than mis-provision. Live verification is a tracked
   runbook checklist, not a blocker for this PR.
8. **Stale doc references.** `docs/deploy.md` still cites FND-017 for Neon branch wiring (that ID doesn't match
   this ticket). Fixed as part of Phase 4.

## Implementation notes

> **AMENDMENT (2026-05-30) — per-PR automation descoped to persistent staging.** After the workflows were built and reviewed, the human chose a **single persistent staging preview environment** over per-PR ephemeral environments. Reason: Google/Discord forbid `*.vercel.app` wildcard OAuth redirect URIs and a Vercel domain binds to one branch at a time, so per-PR previews would require re-registering/reassigning the OAuth origin per PR — high friction for a solo/small team. The persistent-staging model uses one long-lived `staging` git branch bound to `preview.picksleagues.com`, one persistent Neon `staging` branch, and Preview-scoped `DATABASE_URL`/`BETTER_AUTH_URL`. **`.github/workflows/preview-env.yml` and `preview-cleanup.yml` were removed**, along with their GitHub-secret requirements — the persistent flow is entirely Vercel/Neon dashboard config. The `resolveAuthBaseURL` auth code (and its 7 tests) stays — it powers the fixed-`BETTER_AUTH_URL`-with-`VERCEL_URL`-fallback precedence the staging origin relies on. The authoritative runbook is now `docs/deploy.md` § "Preview environment (persistent staging)". Per-PR isolation can be revisited under POL-003 if automated preview smoke tests need it. The notes below describe the originally-built (now partly-removed) workflow design and are retained for history.

**Actually shipped:** All code + CI workflows + documentation completed. **Auth runtime origin derivation:** new pure helper `services/api/src/auth/resolve-auth-base-url.ts` with 7 table-tests covering explicit override, BETTER_AUTH_URL precedence, VERCEL_URL fallback, scheme-handling, and error case. Wired into `services/api/src/auth/index.ts` with `trustedOrigins` populated from raw `VERCEL_URL` to trust the deploy's own host alongside the stable preview origin. **.env.example updated** to document the resolution precedence (items 1–4) and explain why OAuth uses Option B (shared stable origin). **CI: `.github/workflows/preview-env.yml`** (PR open/sync) with required-secrets guard that fails clearly on missing credentials; creates/reuses Neon branch `preview/pr-<n>` via `neondatabase/create-branch-action@v5`; upserts `gitBranch`-scoped `DATABASE_URL` to Vercel REST API. **Critical fix applied:** the `database:` input now uses `${{ secrets.NEON_DB_NAME || 'neondb' }}` to fallback gracefully when NEON_DB_NAME is unset (avoids passing empty string to neonctl). **CI: `.github/workflows/preview-cleanup.yml`** (PR close) idempotently deletes Neon branch + branch-scoped env var, both guarded against missing secrets. **`docs/deploy.md` sections added:** "Per-PR preview environments" explaining the Neon branch lifecycle, the per-branch DATABASE_URL provisioning (via REST API + gitBranch scope), Option B OAuth shared-origin constraint, the runtime BETTER_AUTH_URL precedence, and the reviewer runbook (step-by-step how to sign in via the shared origin and exercise the preview). Fixed stale FND-017 references (should be FND-022). Added "Re-enabling Vercel previews" runbook step with instructions to enable Git integration + clear Ignored Build Step. Added "Required secrets" table documenting all GitHub secrets + where to find them (Neon console, Vercel dashboard). All tests (97 Vitest) pass; workflows validate as correct YAML.

**Decisions made during implementation:**
- **OAuth locked to Option B** (one shared, stable preview origin registered once in OAuth consoles): frontend origin is shared across all open PRs; per-PR isolation is at the Neon database layer, not the frontend. `BETTER_AUTH_URL` is a single fixed preview-scoped Vercel env var, NOT per-branch. Acceptance criteria in the ticket were updated to reflect this (the old per-deploy-`*.vercel.app` wording was replaced with the shared-origin constraint).
- **Build scope shipped; secrets + console registration + re-enable are post-merge runbook items.** Code + workflows committed; they will fail clearly on missing GitHub secrets (acceptable and expected until the human provisions them post-merge). Secrets, Vercel preview re-enable, and OAuth console registration are tracked in `docs/deploy.md` as post-merge checklist items, not blockers for this PR.
- **`database:` input fallback fix.** The Neon action's `database` input now uses `${{ secrets.NEON_DB_NAME || 'neondb' }}` instead of `${{ secrets.NEON_DB_NAME }}`, ensuring it defaults to `neondb` when the secret is unset (avoiding empty-string rejection by neonctl).
