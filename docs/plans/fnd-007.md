# Plan: FND-007 — OpenAPI spec export

**Status:** IMPLEMENTED
**Mode:** collaborative
**Author:** planner agent

## Goal

Expose a machine-readable OpenAPI 3 document at `GET /api/openapi.json` that
accurately describes every API route, starting with `health` and `echo`. This
is the contract source for FND-008's typed client generation: the web (and
later mobile) client is generated from this spec rather than hand-writing fetch
calls (per `picks-leagues-mvp-spec.md` §"API client"). Getting the export shape
right now — one stable document, schemas named, validation error contract
preserved — keeps the generated client clean and stable as routes grow across
later epics.

## Acceptance criteria (from ticket)

- `/api/openapi.json` returns a valid OpenAPI 3 doc
- Health and echo routes present with correct schemas
- README note explains how to regenerate the spec

## Context / current state

- FND-005 established plain `Hono` sub-apps mounted in `src/app.ts`
  (`createHealthRoute`, `createEchoRoute`).
- FND-006 added Zod validation at the HTTP boundary via `@hono/zod-validator`
  helpers in `src/lib/validation.ts` (`zBody`/`zQuery`/`zHeader`/`zParam`),
  which all emit a **canonical 400 `ValidationError` body** documented in
  `services/api/README.md` and asserted by `echo.test.ts`.
- `app.ts` wires an `onError` 500 handler and injects a `Clock` into health.
- The Vercel entry (`api/index.ts`) and node entry (`src/index.ts`) both call
  `createApp()` and must keep working unchanged.

### Key design tension

`@hono/zod-openapi` only derives spec entries from routes declared with its
`createRoute()` + `OpenAPIHono.openapi()` API. Plain `Hono` sub-apps using the
`@hono/zod-validator` helpers contribute **nothing** to the document. So
"health and echo present with correct schemas" requires migrating those two
routes to the `OpenAPIHono` style. The migration must **preserve the existing
behavior and the canonical 400 validation error contract** (README + tests),
not just bolt on a spec.

`@hono/zod-openapi` re-exports a Zod instance with an `.openapi()` augmentation;
its bundled validator drives both runtime validation and schema generation, so
the per-route `@hono/zod-validator` helper calls for migrated routes are
replaced by the `request` block of `createRoute`. Validation error formatting
moves to `OpenAPIHono`'s `defaultHook`, which will reproduce the exact
`{ error: "ValidationError", issues: [{ path, code, message }] }` shape and
`path` prefixing (`body`/`query`/`header`/`param`) currently produced by the
helpers.

## Approach

1. **Add dependency.** Add `@hono/zod-openapi` to `services/api/package.json`
   `dependencies` (pin to the current major compatible with `hono ^4.12` and
   `zod ^4.0`; implementer resolves the exact version at install). It bundles a
   compatible `@hono/zod-validator`. Keep the existing `@hono/zod-validator`
   dependency for now — `zParam` etc. remain exported for not-yet-migrated
   future routes; do not remove `src/lib/validation.ts`.

2. **Add a shared OpenAPI validation hook.** Create
   `services/api/src/lib/openapi.ts` exporting:
   - `formatZodIssues(target, error)` — extracts the existing per-target
     formatting logic (`path` = `[target, ...issue.path].join(".")`, `code`,
     `message`) so it is the single source of truth shared with
     `src/lib/validation.ts` (refactor `validation.ts` to import this helper so
     the contract cannot drift between the two route styles).
   - `openApiDefaultHook` — an `OpenAPIHono` `defaultHook` that, on validation
     failure, derives the target from `result.target`
     (`json`→`body`, `query`, `header`, `param`/`cookie` mapped accordingly)
     and returns `c.json({ error: "ValidationError", issues }, 400)` — byte-for-byte
     the canonical shape.
   - A `createOpenApiApp()` factory returning a configured
     `new OpenAPIHono({ defaultHook: openApiDefaultHook })` so `app.ts` and
     sub-apps construct instances consistently.

3. **Migrate the health route** (`src/routes/health.ts`):
   - Keep `createHealthRoute(deps)` signature and `Clock` injection.
   - Return an `OpenAPIHono` instead of `Hono`.
   - Define a `HealthResponseSchema` with `z` from `@hono/zod-openapi`:
     `{ status: z.literal("ok"), time: z.string().datetime() }`, named via
     `.openapi("HealthResponse")`.
   - Declare the route with `createRoute({ method: "get", path: "/", responses:
     { 200: { description, content: { "application/json": { schema } } } } })`
     and register the handler with `route.openapi(...)`. Handler body unchanged
     (`deps.clock.now().toISOString()`).

4. **Migrate the echo route** (`src/routes/echo.ts`):
   - Return an `OpenAPIHono`.
   - Reuse the existing `headerSchema`/`querySchema`/`bodySchema` shapes (move
     `z` import to `@hono/zod-openapi`'s `z`; the shapes are unchanged). Name
     the request/response schemas via `.openapi(...)`.
   - Declare `createRoute({ method: "post", path: "/", request: { headers,
     query, body: { content: { "application/json": { schema: bodySchema } } } },
     responses: { 200: {...EchoResponseSchema}, 400: {...ValidationErrorSchema} } })`.
   - Handler body unchanged (reads `c.req.valid("query")` / `valid("json")`).
   - Define a shared `ValidationErrorSchema`
     (`{ error: z.literal("ValidationError"), issues: array of { path, code,
     message } }`) in `src/lib/openapi.ts`, named `.openapi("ValidationError")`,
     and reference it in the 400 response so the generated client types errors.

5. **Convert the root app** (`src/app.ts`):
   - `createApp` builds `const app = createOpenApiApp()` (an `OpenAPIHono`).
   - Mount sub-apps exactly as today: `app.route("/api/health", ...)`,
     `app.route("/api/echo", ...)`. `OpenAPIHono` merges direct sub-app
     definitions, so mounted paths appear as `/api/health` and `/api/echo` in
     the doc.
   - Add `app.doc("/api/openapi.json", { openapi: "3.0.0", info: { title:
     "Picks Leagues API", version: "0.0.0" } })`. (Use `app.doc`, not
     `doc31`/Swagger UI — only the raw JSON is in scope; version string can read
     from `package.json` is out of scope, hardcode `"0.0.0"` to match package.)
   - Keep the existing `onError` 500 handler unchanged.
   - `createApp`'s return type widens from `Hono` to the `OpenAPIHono` instance
     type; `src/index.ts` and `api/index.ts` use `app.fetch`/`handle(app)`
     which `OpenAPIHono` supports — no change needed there, but verify
     typecheck passes (both `tsconfig.json` and `tsconfig.tooling.json`).

6. **README.** Add a "## OpenAPI spec" section to `services/api/README.md`
   explaining: the spec is generated from route definitions (no separate build
   step), served live at `GET /api/openapi.json` while the dev server runs, how
   to fetch it (`curl http://localhost:3000/api/openapi.json`), and that
   FND-008's `pnpm gen:api` consumes this endpoint. Note the convention: new
   routes must be declared via `createRoute`/`OpenAPIHono.openapi()` to appear
   in the spec, and update the existing "Adding new routes" / "Request
   validation" sections to reflect the `@hono/zod-openapi` route style (the
   colocated-one-schema-set-per-route convention is preserved; only the
   declaration mechanism changes).

## Risks and open questions

- **Route-style migration is the real work.** The ticket reads like "add an
  export" but `@hono/zod-openapi` cannot document plain `Hono` sub-apps, so
  health and echo must move to `createRoute`. This is unavoidable to satisfy
  AC #2 ("present with correct schemas"). Flagging because the diff is larger
  than the one-line ticket implies. The behavior contract is fully preserved
  and guarded by the existing unmodified tests.
- **Zod v4 + `@hono/zod-openapi` compatibility.** Repo uses `zod ^4.0`. Most
  recent `@hono/zod-openapi` majors support Zod v4, but the implementer must
  pick a version that does; if the current release only supports Zod v3 this
  becomes a blocker requiring either a Zod pin discussion or the alternative
  library — see assumption below.
- **Two validator deps coexist.** `@hono/zod-openapi` bundles its own
  `@hono/zod-validator`; the standalone one stays for `zParam` and future
  non-migrated helpers. Acceptable short-term; a later ticket could fully
  standardize. Out of scope here.
- **`createApp` return type change** ripples to `src/index.ts` /
  `api/index.ts`. Expected to be source-compatible (`OpenAPIHono extends Hono`),
  but typecheck across both tsconfigs must be verified.

## Assumptions (autonomous mode only)

(Collaborative mode — see clarifying questions; these are the conforming
defaults I will use absent other direction.)

- Use `@hono/zod-openapi` (the ticket's first suggestion and the
  `createRoute`-based style that gives the cleanest generated types for
  FND-008) rather than `hono-openapi`.
- Endpoint path is exactly `/api/openapi.json` (matches ticket and the `/api`
  prefix convention). No Swagger UI / no `/api/doc` HTML — raw JSON only.
- `info.version` hardcoded to `"0.0.0"` to match `package.json`; reading it
  dynamically is out of scope.

## Out of scope

- FND-008 client generation (`pnpm gen:api`, `apps/web` client, CI staleness
  check) — depends on this ticket but is separate work.
- Swagger UI / Scalar / any HTML documentation viewer.
- Auth/JWT security schemes in the spec (no auth routes exist yet).
- Migrating or adding any routes beyond health and echo.
- Removing `@hono/zod-validator` or deleting `src/lib/validation.ts`.
- Dynamic version sourcing from `package.json`.

## Implementation notes

**Actually shipped:** Added `@hono/zod-openapi@1.4.0` (compatible with `hono >=4.10` and `zod ^4.0` — Zod v4 blocker did not materialize). Migrated health and echo routes from plain Hono to OpenAPIHono via `createRoute()` and `.openapi()` declaration. Created `services/api/src/lib/openapi.ts` with a shared `formatZodIssues` helper extracted from validation.ts to keep the canonical 400 ValidationError contract consistent across both route styles. Added `GET /api/openapi.json` serving OpenAPI 3.0.0 spec. All existing health.test.ts and echo.test.ts pass unchanged; new openapi.test.ts covers spec structure and regression scenarios (19/19 suite green, lint + typecheck clean). README updated with OpenAPI section and revised "Adding new routes" / "Request validation" sections per the new `@hono/zod-openapi` style.

**Decisions made during implementation:**
- EchoQuery and EchoHeaders declared as OpenAPI `parameters` array rather than inlined in `components.schemas` (correct OpenAPI 3 convention for query/header params); test asserts parameters array structure.
- `createOpenApiApp()` returns non-generic `OpenAPIHono` for source compatibility with entry files (generics unused; avoids cascading type changes in app.ts).
- One justified `as` cast in openapi.ts for `result.target` (runtime field absent from Hook generic type) with WHY comment and fallback; avoids unsafe `any`.
- `ValidationTarget` includes forward-looking `cookie`/`form` prefixes for future routes; noted in README as ready when routes exercise those targets.
