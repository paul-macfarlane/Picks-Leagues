## Design decisions

1. **Flat config (`eslint.config.js`), not legacy `.eslintrc`.** The repo is
   ESM-only and on a modern toolchain; `typescript-eslint`'s current
   recommended setup is flat config via the `typescript-eslint` helper
   package. Legacy `.eslintrc` is deprecated in ESLint 9. Flat config is also
   strictly better for a monorepo: one root array can carry per-path overrides
   (web vs api, fixtures, generated files) without `overrides` glob gymnastics.

2. **Shared root config, thin per-package wrappers.** A single
   `eslint.config.js` and a single `prettier.config.js` at the repo root.
   - ESLint: root config is the source of truth. Each package's `lint` script
     runs ESLint pointed at its own `src` using the root config. This keeps
     React-specific rules scoped to `apps/web` via a `files` override and
     avoids duplicated rule blocks. A package-local `eslint.config.js` is *not*
     created — ESLint 9 walks up to the nearest config and the root is found.
   - Prettier: one root config; root-level `format` / `format:check` scripts
     run across the whole repo. Per-package format scripts are unnecessary and
     would fragment the config.
   - Rationale: code-standards.md treats lint/format as repo-wide gates run in
     CI (FND-016). One config = one place to change a rule.

3. **`import/no-restricted-paths` scaffolded but inert.** The
   `scoring/`/`domain/` directories do not exist yet, so an active rule has
   nothing to restrict. We add the `eslint-plugin-import-x` plugin and define
   the `import-x/no-restricted-paths` rule with the zones **written out and
   commented with the exact target shape**, set to `'off'` (or zones array
   present but directories not yet created — see Open Questions Q1). A single
   exported constant `RESTRICTED_PATH_ZONES` holds the zone definitions so a
   future ticket flips one value / uncomments one line. This satisfies AC
   "configured in a way that's easy to add new rules to later."

   Chosen plugin: **`eslint-plugin-import-x`** (the maintained, flat-config-
   native, faster fork of `eslint-plugin-import`). It exposes the same
   `no-restricted-paths` rule. If parent prefers the original
   `eslint-plugin-import`, see Open Questions Q2 — it works too but has
   rougher flat-config + TS-resolver ergonomics.

4. **ESLint = correctness, Prettier = style/imports.** Per code-standards.md:
   ESLint does NOT format and does NOT sort imports. We use
   `eslint-config-prettier` to turn off any stylistic rules that could fight
   Prettier. Import ordering and unused-import removal are Prettier's job via
   `@ianvs/prettier-plugin-sort-imports`.

5. **`no-explicit-any` is an error, not a warning.** The FND-002 ticket text
   says "warning"; code-standards.md § Dead code says unused-vars is an
   *error*. For `no-explicit-any` we follow `tseslint.configs.recommended`
   default, which makes it an **error**. The fixture only asserts it "fires" —
   error satisfies that. See Open Questions Q3 for confirmation.

### 2. Create `eslint.config.js` at repo root

Flat config array built with `tseslint.config(...)`. Structure (described, not
coded):

1. **Global ignores** block: `**/dist/**`, `**/node_modules/**`,
   `**/coverage/**`, `apps/web/src/routeTree.gen.ts`,
   `apps/web/src/lib/api-client/**` (generated by FND-008),
   `**/*.config.js` (the config files themselves), and
   `services/api/src/__lint-fixtures__/**` is NOT ignored (the renamed
   `any-usage.ts` must be lintable — see step 7).
2. **Base JS**: `@eslint/js` `configs.recommended`.
3. **TypeScript**: `tseslint.configs.recommended` applied to
   `**/*.{ts,tsx}`. Decide on type-checked rules — see Open Questions Q4. Plan
   default: use **non-type-checked `recommended`** for FND-003 (empty packages,
   no `tsconfig` project graph wired for linting yet), and defer
   `recommendedTypeChecked` + `no-floating-promises` / `no-misused-promises`
   to when real async API code exists (track as a follow-up note in the plan's
   "Out of scope"). This keeps `pnpm lint` green on empty packages without a
   `parserOptions.project` setup that has nothing to typecheck.
4. **import-x rules** block (applies to `**/*.{ts,tsx}`):
   - Register `eslint-plugin-import-x` plugin.
   - Configure the TypeScript resolver settings
     (`settings['import-x/resolver']`) so both package tsconfigs are picked up.
   - `import-x/no-restricted-paths`: configured from an exported
     `RESTRICTED_PATH_ZONES` constant defined at the top of the file. The
     constant contains the **final intended zones** (target =
     `services/api/src/scoring/**` and `services/api/src/domain/**`, from =
     `db`, `repositories`, `routes`, `cron`, `providers`, `notifications`),
     with a clear one-line WHY comment citing the non-negotiable. The rule is
     activated but the zones reference directories that do not yet exist, so it
     is a no-op today and becomes enforcing the moment those directories are
     created — OR set to `'off'` with the zones ready (Open Questions Q1
     decides which; default below).
     **Default decision (autonomous fallback if no answer):** keep the rule
     `'error'` with the full zone list present. ESLint's `no-restricted-paths`
     simply matches nothing when target dirs are absent, so `pnpm lint` stays
     green on empty packages AND the boundary auto-engages when epic 02 adds
     the first `scoring/` or `domain/` file — zero config change needed. This
     is the strongest interpretation of "easy to add new rules to later."
5. **React override** block (`files: ['apps/web/**/*.{ts,tsx}']`):
   `eslint-plugin-react-hooks` `recommended` (rules-of-hooks error,
   exhaustive-deps warn per code-standards.md). Scoped so API code isn't
   touched.
6. **Language options**: `globals.node` for `services/api/**`,
   `globals.browser` for `apps/web/**`; `ecmaVersion: 2022`,
   `sourceType: 'module'`.
7. **`eslint-config-prettier`** applied LAST so it wins and disables stylistic
   conflicts. (Imported as a flat config object and spread at the end of the
   array.)

`@typescript-eslint/no-unused-vars` is set to `error` with
`argsIgnorePattern: '^_'` and `varsIgnorePattern: '^_'` per code-standards.md
§ Dead code (overriding the recommended preset's `warn`).

### 3. Create `prettier.config.js` at repo root

ESM module exporting the config object. Per code-standards.md, do NOT override
formatting defaults to settle bikesheds — only configure the plugins:

- `plugins`: `['@ianvs/prettier-plugin-sort-imports',
  'prettier-plugin-tailwindcss']`. Order matters — sort-imports first, tailwind
  last (tailwind plugin must be last per its docs).
- `importOrder`: encode code-standards.md § Imports order:
  external → internal alias (`@/`) → relative. Concretely:
  `["<BUILT_IN_MODULES>", "<THIRD_PARTY_MODULES>", "", "^@/(.*)$", "",
  "^[.]"]` (blank strings = newline separators between groups).
- `importOrderParserPlugins`: `["typescript", "jsx", "decorators-legacy"]`.
- No `tailwindStylesheet`/`tailwindConfig` yet (Tailwind config arrives
  FND-010); plugin is a no-op until then — harmless.

Note: `@ianvs/prettier-plugin-sort-imports` does NOT remove unused imports.
code-standards.md § Imports line 26 claims unused imports are auto-removed by
the Prettier plugin. This is **factually not what that plugin does** — see
Open Questions Q5. Plan default: rely on `@typescript-eslint/no-unused-vars`
as the enforcement (it's already an error), and flag the doc inaccuracy as a
risk rather than silently shipping a config that doesn't deliver the documented
behavior.
