13. **Add a "Transactions" subsection to `docs/code-standards.md` Â§ Database.**
    This is a **docs-only** deliverable (no enforcement tooling this ticket).
    The implementer adds it as a subsection at the end of the existing
    `## Database (Drizzle + Neon)` section, matching the document's terse,
    enforceable bullet voice and consistent with Â§ Error handling and Â§
    "Backend architecture: domain â†’ repositories â†’ routes". It must establish
    at minimum:
    - Any operation performing **more than one write** that must succeed or
      fail together MUST run inside a single `db.transaction()`.
    - Any **read-then-write where the write depends on the read for
      correctness** (check-then-act, invariant enforcement, uniqueness/quota
      checks) MUST occur in one transaction so the read is consistent with the
      write. A check outside the transaction is a race, not a check.
    - The transaction callback receives a `tx` handle. Every query in that unit
      of work MUST use `tx`, never the outer `db`. Capturing `db` inside a
      transaction silently escapes it and is a correctness bug.
    - Repository functions accept an optional executor parameter (the `tx`
      handle, defaulting to the shared `db`) so a domain-orchestrated unit of
      work can compose multiple repository calls in one transaction. The
      transaction boundary is **owned by the orchestration layer** (the route
      handler / orchestration described in Â§ Backend architecture), opened
      around a sequence of repository calls â€” never opened or hidden inside a
      single repository function. Repositories stay dumb; they just thread the
      executor through.
    - Keep transactions short. Do **not** perform external/network I/O inside a
      transaction (e.g., ESPN/`SportsProvider` calls, HTTP, queueing). Load
      external data first, then open the transaction for the DB writes.
    - One-line rationale that the chosen Neon serverless WebSocket **pooled**
      driver (`drizzle-orm/neon-serverless` + `Pool`) supports interactive
      transactions; the HTTP driver (`neon-http`) does not â€” so this is why the
      pooled driver is mandated and must not be "optimized" back to `neon-http`.
    - Note that this is enforced by review for now. A future ESLint rule could
      enforce "no `db` reference inside a `db.transaction(async (tx) => ...)`
      callback" â€” flagged as a possible future tightening, **out of scope for
      this ticket**.
    The exact wording is the implementer's to draft within these constraints;
    additional load-bearing rules consistent with the above are welcome (e.g.,
    no nested-transaction surprises, prefer the smallest viable isolation, do
    not swallow a transaction rollback error â€” rethrow per Â§ Error handling).

## Risks and open questions

- **Deliberate deviation from the MVP spec's `neon-http` mention (accepted).**
  The MVP spec references the Neon serverless HTTP driver. This plan instead
  mandates the Neon serverless **WebSocket pooled** driver
  (`drizzle-orm/neon-serverless` + `Pool` from `@neondatabase/serverless`,
  `neonConfig.webSocketConstructor = ws`). Rationale: the human requires the DB
  layer to support interactive transactions, and `drizzle-orm/neon-http`
  **cannot** do interactive transactions â€” `db.transaction(async (tx) => ...)`
  throws "No transactions support in neon-http driver". The WebSocket pooled
  driver is Neon's recommended approach when transactions are needed and is
  fully compatible with the Vercel Node serverless runtime (the only added
  requirement is the `ws` package, since Node serverless has no native
  WebSocket). This is the single intentional `neon-http` reference in this
  plan and exists only to document why we did not use it. FND-005 should treat
  the first live `db:migrate` as the integration checkpoint for this driver.
- **Neon not provisioned this ticket (resolved).** Per human decision, the two
  DB-dependent acceptance criteria are deferred to FND-005/FND-014, not run
  during this ticket's testing pause. Risk: a latent driver/migrator
  incompatibility (now including WebSocket pool behavior) won't surface until
  Neon exists. Mitigation: generation determinism is verified now;
  `migrate.ts`/`client.ts` typecheck/lint without a live DB; first real run in
  FND-005/FND-014 is the integration checkpoint.
- **`users` schema overlap with Better Auth (FND-014) (resolved).** Minimal
  bootstrap table now; FND-014 owns the authoritative Better Auth schema and
  reconciles (adopts via schema mapping or supersedes with a follow-up
  migration). Documented in the schema file header so FND-014 has the context.
- **Migration directory location.** Plan puts migrations in
  `src/db/migrations` (inside the package `include`, one reviewable DB unit),
  consistent with the MVP spec's "Drizzle schema + migrations" under `db/` and
  code-standards.md. Some Drizzle examples default to a top-level `drizzle/`
  dir; this plan deliberately keeps it under `src/db/`.
- **Transactions standard is docs-only this ticket.** No ESLint/tooling
  enforcement is added now. An ESLint rule forbidding the outer `db` reference
  inside a `db.transaction()` callback is feasible and noted in the standard as
  a future tightening, but is **out of scope** here. Until then the rule is
  review-enforced.
