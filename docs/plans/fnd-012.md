# Plan: FND-012 — Vercel deployment config

**Status:** IMPLEMENTED
**Mode:** collaborative
**Author:** planner agent

## Goal

Take the local-only monorepo and make it deploy to a single Vercel project,
hosting the Vite SPA at `/` and the Hono API as Node.js serverless functions
at `/api/*`. After this ticket, every PR gets a working preview URL where the
web app loads, calls the deployed API health endpoint, and renders the result —
satisfying the "Sunday afternoon with friends" experience from `product-vision.md`
end-to-end on the real internet for the first time.

This is foundation plumbing every later epic (auth callbacks, cron jobs,
sports-data syncs) depends on; once the deploy works for the spine, FND-013
(cron secret), FND-014 (Better Auth), and FND-017 (Neon per-PR branch) layer on
top without re-litigating the deploy topology.

## Acceptance criteria (from ticket)

- A preview deploy succeeds end-to-end (web + API).
- Deployed API health endpoint responds.
- Deployed web app fetches the deployed API successfully.

## Approach

The eight steps below are the implementer's checklist. Order matters: the API
entry move (step 2) and base-URL switch (step 4) must land together with the
`vercel.json` (step 1) so the repo is consistent at every commit.

### 1. Add `vercel.json` at the repo root

Create `vercel.json` at the repo root with this exact content. Each property
is justified after the block.

```json
{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "buildCommand": "pnpm --filter @picksleagues/web build",
  "installCommand": "pnpm install --frozen-lockfile",
  "outputDirectory": "apps/web/dist",
  "framework": null,
  "rewrites": [
    { "source": "/((?!api/).*)", "destination": "/index.html" }
  ]
}
```

**Why each field:**

- `buildCommand`: explicitly builds only the web workspace; the web build's
  `prebuild` already runs `gen:api` (see `apps/web/package.json`). pnpm
  workspaces honor `--filter` from the repo root, which is where Vercel runs
  `installCommand`.
- `installCommand`: `--frozen-lockfile` makes builds deterministic and
  reproduces what CI does. Without this, Vercel may use a non-frozen install.
- `outputDirectory: "apps/web/dist"`: where Vite writes its build output. SPA
  static files served by Vercel CDN.
- `framework: null`: tells Vercel "Other" — do not auto-detect Vite (we are
  composing a custom build that includes the workspace API). The auto-detector
  expects a single-package Vite app at root, which we are not.
- `rewrites`: SPA fallback — any path that is NOT `/api/*` and is not a static
  asset is rewritten to `/index.html` so TanStack Router handles it
  client-side. Static files (anything with an extension) take filesystem
  precedence per Vercel's
  [rewrite docs](https://vercel.com/docs/project-configuration/vercel-json#rewrites)
  ("The `source` property should NOT be a file because precedence is given to
  the filesystem prior to rewrites being applied"). The negative-lookahead
  `(?!api/)` keeps `/api/*` traffic going to the function — Vercel resolves
  `/api/*` to the function at `api/index.ts` before falling through to the
  rewrite. We do NOT need an explicit `/api/*` rewrite.

**Not included, with rationale:**

- `functions.api/index.ts.runtime` / `nodejs22.x`: Vercel reads
  `engines.node` from the root `package.json` (`>=22 <23`). No explicit pin
  needed here. If Vercel's auto-selection regresses, add
  `"functions": { "api/index.ts": { "runtime": "nodejs22.x" } }` then.
- `crons`: epic 02 owns cron schedules; out of scope here.
- `regions`: leave at default (`iad1`). Neon's default region matters more
  than ours; revisit when we pick a Neon region in FND-017.

### 2. Move the API entry to the repo root `/api/index.ts`

Vercel's Node.js function builder, when used with a non-Next.js project,
discovers serverless functions at the **deployment-root** `api/` directory.
Because we deploy from the repo root (Root Directory unset / `.`), the function
file must live at `/api/index.ts`, not `/services/api/api/index.ts`.

Two sub-steps:

**(a) Create `/api/index.ts`** at the repo root with this content (port of the
existing `services/api/api/index.ts`, just relocated):

```ts
import { handle } from "hono/vercel";

import { createApp } from "../services/api/src/app";

const app = createApp();

export default handle(app);
```

Notes on the import path:

- A relative import (`../services/api/src/app`) is the simplest thing that
  works — no `exports` field gymnastics on the API package, no path alias
  changes. Vercel's Node builder traces dependencies via require/import
  resolution and bundles them; `services/api/src/app` and everything it
  transitively imports (Hono, Zod, our routes) get included.
- Per code-standards § Imports, "No file extensions on relative imports."
  This file follows that rule.

**(b) Delete `/services/api/api/index.ts`** and remove `"api"` from the
`include` array in `services/api/tsconfig.json` (it referenced the now-gone
folder). This is a deliberate FND-005 drift — the entry has moved from the
service subdirectory to the deployment root. Add a one-line revision-log entry
to `docs/picks-leagues-mvp-spec.md` § Revision log per code-standards
§ Documentation and spec sync.

**(c) Update `services/api/README.md`** if it references `services/api/api/`
as the Vercel entry — replace with a pointer to the root `api/index.ts` and a
short rationale ("we deploy as a single Vercel project; entry lives at the
deployment root").

**(d) Add a root `tsconfig.json` include for the new file** — the existing
root tsconfig (none) / setup needs the new `api/index.ts` to be typechecked.
Two options:
- Add the root `/api` directory to `services/api/tsconfig.json` `include` so
  `pnpm --filter @picksleagues/api typecheck` covers it (it imports from the
  api package), OR
- Create a minimal `tsconfig.json` at the repo root that includes `api/`. The
  Vercel builder uses its own bundler and does not require this; it's purely
  for local typecheck/editor support.

Recommendation: add a thin `/tsconfig.json` at the repo root extending
`./tsconfig.base.json` with `"include": ["api"]` and `"compilerOptions": {
"types": ["node"], "noEmit": true }`, and add an `api:typecheck` script to the
root package.json that runs `tsc -p tsconfig.json --noEmit`. Wire it into the
root `pnpm typecheck` aggregate so CI catches breakage. Keep this minimal —
the root tsconfig is JUST for the `api/` entry file, not for anything else.

### 3. Configure local dev to mirror production (same-origin via Vite proxy)

Same-origin in production means the web calls `/api/health` (relative) and
Vercel routes it to the function. Locally, the Vite dev server at `:5173`
needs to forward `/api/*` to the API dev server at `:3000`.

Modify `apps/web/vite.config.ts` to add a `server.proxy` block:

```ts
server: {
  port: 5173,
  strictPort: true,
  proxy: {
    "/api": {
      target: "http://localhost:3000",
      changeOrigin: false,
    },
  },
},
```

**Why proxy over `VITE_API_BASE_URL`-with-an-absolute-localhost-URL:** the dev
experience now matches production exactly (relative URLs everywhere; CORS
never engaged from the browser). This is a one-line config and avoids per-env
URL juggling in `apps/web/src/lib/api.ts`.

### 4. Switch the web client to a relative (same-origin) base URL

- Change `const baseUrl = "http://localhost:3000"` to read from a single

  default means "same origin" — `openapi-fetch` then issues requests like

  origin) and dev (Vite proxies it).
- Remove the FND-009/FND-011 placeholder comment now that FND-012 has wired
  this. Replace with one line stating: relative same-origin by default; an
  explicit override is supported via `VITE_API_BASE_URL` for cases we don't

Add `VITE_API_BASE_URL` declaration to `apps/web/src/vite-env.d.ts`:

```ts
}
```

## Risks and open questions

- **Vercel project creation is human-owned.** The implementer ships the code
  config; the human must create the Vercel project, point it at the repo, and
  set Root Directory to `.`. The plan documents this in `docs/deploy.md`. The
  AC "preview deploy succeeds end-to-end" cannot be closed until the human
  does this — that's a hand-off, not a defect.
- **Function bundle size.** Bundling the full Hono app at `api/index.ts`
  brings in Drizzle, Neon driver, all routes. Bundle size has a Vercel limit
  (~50 MB unzipped on Hobby/Pro). Modern Vercel Node builder tree-shakes;
  the current app is tiny (echo + health + openapi). Flag for revisit if we
  ever approach the limit; not a concern today.
- **`engines.node` enforcement.** Root `package.json` says `>=22 <23`. Vercel
  honors `engines.node` for the Node runtime version. If Vercel drops Node 22
  support before we update the engines pin, the deploy will fail with a clear
  message. Acceptable risk.
- **Dev-only CORS middleware lingers in production code paths.** It is
  inert in same-origin production (browser never sends an `Origin` that
  doesn't match the route origin from a same-origin fetch). But it's noise.
  Removing it cleanly is best done when FND-014 settles the credentials/CORS
  story for real — left for that ticket.

## Assumptions (collaborative mode — recorded, not blocking)

Recorded so the reviewer/human can override at review time:

1. **One Vercel project hosts BOTH the SPA and the API** (per ticket
   description's "Confirm both deploy to a single Vercel project"). The
   alternative — separate Web and API projects, each with its own Root
   Directory — is more idiomatic for Vercel monorepos per
   [Vercel's monorepo docs](https://vercel.com/docs/monorepos) but would
   require CORS in production and two preview URLs per PR. The ticket and
   `docs/picks-leagues-mvp-spec.md` § Deployment Topology both say "Same
   platform... one place for env vars and PR previews" — read as single
   project.
2. **Same-origin via co-deployed function, no rewrite for `/api/*`.** Vercel
   routes `/api/*` to the function at `api/index.ts` automatically when both
   are deployed together; only the SPA fallback rewrite is needed. This is
   the simplest correct configuration and side-steps CORS entirely (memory:
   `project_cors_credentials_fnd014` — same-origin makes the future
   credentials-CORS question moot for now; flagged to revisit in FND-014).
3. **Web base URL is relative (`""`) by default.** `VITE_API_BASE_URL` is
   declared and supported but unset in normal use. This keeps the deployed
   and dev experiences identical (Vite dev proxy mirrors the production
   routing). If you prefer the explicit absolute-URL pattern instead,
   `apps/web/src/lib/api.ts` becomes a one-line change.
4. **Move `services/api/api/index.ts` to root `api/index.ts`** rather than
   adding a second adapter file or using `functions` glob remap. Reasons:
   keeps one canonical Vercel function entry; matches Vercel's discovery
   convention exactly; avoids ambiguity. This is a deliberate drift from
   FND-005's file layout — logged in the spec's Revision log per
   code-standards § Documentation and spec sync.
5. **No explicit Node runtime pin in `vercel.json`.** Root
   `engines.node: ">=22 <23"` is the source of truth; Vercel reads it. If
   that turns out not to be honored on first deploy, add
   `"functions": { "api/index.ts": { "runtime": "nodejs22.x" } }` then.
6. **Add a minimal root `tsconfig.json`** to make the new `api/index.ts`
   typecheck without conflating it with the API workspace's tsconfig. Wire
   into `pnpm typecheck` so CI catches breakage. Alternative considered: add
   `"../../api"` to `services/api/tsconfig.json` `include` — rejected because
   it confuses scope (the API workspace shouldn't own a file outside its
   directory).

## Implementation notes

**Actually shipped:** The Vercel deployment config is now live and verified on preview deploy. The SPA and API co-deploy to a single Vercel project. The web app loads, fetches the deployed API successfully, and displays the health card with server time — all three acceptance criteria met. All deviations from the original plan are documented below because Vercel's TypeScript compilation constraints and filesystem routing behavior forced four non-trivial changes to the implementation path.

**Decisions made during implementation:**

- **Deviation 1: API entry is bundled, not deployed as TS source.**
  Plan said: deploy `api/index.ts` directly at the repo root using Vercel's `@vercel/node` builder. Reality: Vercel's `@vercel/node` builder hard-codes `moduleResolution: nodenext` (for Node ESM runtime alignment), which conflicts with our `moduleResolution: bundler` tsconfig. The `nodenext` resolver rejects relative imports without `.js` extensions on every import in our codebase. Vercel silently skipped emitting the function but marked the build "Ready" because the SPA build succeeded. Fix: source lives at `services/api/src/vercel-entry.ts` (typecheck-included in normal `services/api` builds). The root `tsconfig.json` and root `hono` devDep from the original plan were not needed. New `pnpm build:api` script runs esbuild to bundle to a single self-contained ESM file (~700 KB). `vercel.json` `buildCommand` chains web build + api bundle. `/api/` directory is gitignored — it's a build artifact.

- **Deviation 2: Bundle output filename uses Vercel's optional catch-all syntax.**
  Plan said: function at `api/index.ts` (then `api/index.js` after the bundle deviation). Reality: `api/index.js` only matches the literal URL `/api` in Vercel's filesystem routing; any `/api/*` subpath returned 404 before Hono could dispatch. Fix: bundle output is `api/[[...slug]].js`. The `[[...slug]]` optional catch-all tells Vercel to route `/api` and all subpaths to the same function, which receives the full original URL for Hono's internal routing.

- **Deviation 3: Vercel Node runtime adapter via `@hono/node-server`, not `hono/vercel`.**
  Plan said: `import { handle } from "hono/vercel"; export default handle(app)`. Reality: `hono/vercel`'s `handle()` returns `app.fetch`, which expects Fetch API `Request`. Vercel's generic Node runtime invokes the function with Node `IncomingMessage`/`ServerResponse` — different API, so `c.req.header(...)` in CORS middleware threw `TypeError: this.raw.headers.get is not a function` at runtime. Setting `NODEJS_HELPERS=0` (Next.js Pages Router-specific) did not help. Fix: use `getRequestListener` from `@hono/node-server` (already a dep). It bridges Node `IncomingMessage`/`ServerResponse` → Fetch `Request` → `app.fetch` → Fetch `Response` → Node response.

- **Deviation 4: ESLint ignore for the build artifact.**
  The bundled `api/[[...slug]].js` is minified third-party code; ESLint reported ~158 violations. Added `api/**` to `eslint.config.js` global ignores.

**What did NOT change from the plan:** single Vercel project hosting both SPA and API; same-origin routing in production (`/api/*` relative URLs, no CORS); Vite dev proxy mirrors production routing; `apps/web/src/lib/api.ts` reads `VITE_API_BASE_URL` (defaults to empty string for same-origin); `docs/deploy.md` env var table and setup steps; required dev experience (`pnpm dev` at `:5173` + `:3000` with proxy).

**Follow-ups (if any):** none
