# Plan: FND-018 — Dependabot dependency + security updates

**Status:** IMPLEMENTED
**Mode:** collaborative
**Author:** planner agent

## Goal
Add a committed `.github/dependabot.yml` so the repo automatically receives PRs for
GitHub security advisories (opened immediately) and routine version bumps (grouped into a
low-noise weekly PR). This keeps the dependency surface current and patched without manual
auditing, which matters because the project commits to being independent and long-lived
(`product-vision.md`) — unmaintained dependencies are how an indie project rots. FND-016's
CI pipeline (DONE) already runs on every `pull_request`, so every Dependabot PR is
auto-validated (typecheck, lint, test, build, format:check, plus the API-client staleness
gate) before a human merges it. No infrastructure or GitHub-UI configuration is required;
Dependabot reads the committed file automatically once it lands on the default branch.

## Acceptance criteria (from ticket)
- `dependabot.yml` covers npm (all workspace manifests) + github-actions.
- Security-advisory PRs open automatically; routine bumps are grouped weekly.
- A test bump PR is created and passes CI (FND-016 now exists, so "passes CI" is the bar,
  not the pre-FND-016 "builds locally" fallback).

## Current state (verified)
- Three npm manifests: `/package.json` (root, dev tooling only), `/apps/web/package.json`,
  `/services/api/package.json`.
- Workspace globs: `apps/*`, `services/*` (`pnpm-workspace.yaml`).
- Single root `pnpm-lock.yaml` covering all three manifests (pnpm workspace).

## Approach

### 1. npm ecosystem — directory strategy
Use a **single npm entry rooted at `/` with `directories: ["/**/*"]`**, not three separate
per-package entries.

Justification:
- This is a pnpm workspace with one shared root `pnpm-lock.yaml`. Dependabot resolves npm
  updates against that single lockfile. The `directories` glob `"/**/*"` tells Dependabot to
  discover every `package.json` in the tree (root, `apps/web`, `services/api`) and update
  them as one cohesive unit while writing the single resulting lockfile change.
- Per-package entries (`directory: "/"`, `directory: "/apps/web"`, `directory: "/services/api"`)
  would fragment a workspace that shares one lockfile, risking competing PRs that each try to
  rewrite the same `pnpm-lock.yaml` and conflict. One entry keeps the lockfile authority in
  one place and keeps PR noise down — directly serving the ticket's "keep noise low" goal.
- `directories` (plural, glob-capable) is the documented mechanism for matching multiple
  manifest directories from one config block, which is exactly the monorepo shape here.

If the reviewer/implementer finds that the installed Dependabot version on this repo does not
honor `directories: ["/**/*"]` for nested workspace manifests, the fallback is a single entry
with `directory: "/"` (Dependabot's npm updater is workspace-aware and follows
`pnpm-workspace.yaml` from the root). Either way the principle holds: **one npm entry, root-anchored.**

### 2. Grouping + schedule for routine bumps
Within the npm entry:
- `schedule.interval: "weekly"` (one routine PR window per week).
- A `groups` block that batches non-major updates into a single PR. Use a catch-all group
  (e.g. named `npm-minor-and-patch`) with `update-types: ["minor", "patch"]` and
  `patterns: ["*"]`. Major bumps fall outside the group and open as individual PRs (majors
  carry breaking-change risk and deserve isolated review + isolated CI signal).
- Set `open-pull-requests-limit` modestly (e.g. `5`) to cap outstanding routine PRs.

Security behavior: Dependabot security updates are independent of `schedule` — they open
immediately when an advisory matches a dependency, regardless of the weekly cadence. No extra
key is required to "enable" this beyond having the ecosystem configured; the weekly schedule
only governs version (non-security) updates. The plan notes this explicitly so the reviewer
doesn't expect a separate stanza.

### 3. github-actions ecosystem
Add a second `package-ecosystem: "github-actions"` entry, `directory: "/"` (Dependabot scans
`.github/workflows/` from the repo root for action versions). Same `schedule.interval: "weekly"`
and a grouping block batching action bumps into one weekly PR (group name e.g.
`github-actions`, `patterns: ["*"]`). This keeps `actions/checkout`, `pnpm/action-setup`, and
`actions/setup-node` current across both workflow files without a PR per action.

### 4. Conventional Commits alignment
Set `commit-message.prefix` per ecosystem so Dependabot PRs match repo commit conventions
(CLAUDE.md → Conventional Commits scoped by epic). Proposed:
- npm entry: `prefix: "chore(deps)"` (and Dependabot auto-appends `(deps-dev)` semantics via
  `prefix-development` if desired — optional).
- github-actions entry: `prefix: "ci(deps)"` to reflect that action bumps are CI-surface
  changes.
This keeps the pre-commit/Conventional-Commit expectations satisfied without a human editing
each Dependabot commit message.

### 5. Test bump PR (acceptance criterion 3)
This is a human/GitHub-side verification, not a code change in this repo:
- After the file merges to `main`, trigger Dependabot manually from the repo's
  **Insights → Dependency graph → Dependabot** tab ("Check for updates"), or simply wait for
  the first weekly run. The first run will open the grouped npm PR and/or the github-actions
  PR.
- Confirm the resulting PR runs `ci.yml` (5 jobs) + `api-client-check.yml` and goes green.
- The plan flags this as the one step that happens outside committed files. It does **not**
  require pre-configuring anything in the UI for Dependabot to function — the committed YAML
  is sufficient. The "Check for updates" click is only to avoid waiting up to a week to
  observe the test PR.

## Risks and open questions
- **`directories` glob support.** The single-entry `directories: ["/**/*"]` approach is the
  documented monorepo pattern, but if this repo's Dependabot version mishandles nested pnpm
  workspace manifests, fall back to a single `directory: "/"` npm entry (Dependabot's npm
  updater follows `pnpm-workspace.yaml`). Either keeps the "one root-anchored npm entry"
  principle. Flagged so the implementer verifies via the first real run rather than assuming.
- **Grouped-PR lockfile churn.** A weekly grouped npm PR may touch many lines of
  `pnpm-lock.yaml`. That is expected and acceptable (it is exactly the noise-reduction
  trade: one big PR beats many small ones). CI validates it regardless.
- **No auto-merge in this ticket.** Auto-merging green Dependabot PRs is intentionally out of
  scope (see below); review stays human for now.

## Implementation notes

**Actually shipped:** `.github/dependabot.yml` (v2) was created with two ecosystems. The npm ecosystem uses a single root-anchored entry with `directories: ["/**/*"]` to cover all three workspace manifests (root, `apps/web`, `services/api`) without fragmenting the shared lockfile. Weekly schedule with a `npm-minor-and-patch` group batches non-major updates into one PR; security advisories open immediately per Dependabot's default behavior. The `github-actions` ecosystem uses `directory: "/"` with a weekly schedule and grouping to keep action versions current. Both commit prefixes are conventional (`chore(deps)` and `ci(deps)` respectively). The config was validated by the repo's existing CI (FND-016) on the merge PR.

**Decisions made during implementation:**
- Reviewer caught that the initial npm entry had both `directory` and `directories` keys (mutually exclusive in Dependabot v2). Fixed by dropping `directory` and keeping `directories: ["/**/*"]`.
- Acceptance criterion #3 (a test bump PR passes CI) is deferred to a post-merge follow-up per the user's explicit decision. Dependabot only acts on the config once it's on the default branch, so the first run cannot be verified until after merge. This is expected and does not block rollout.
