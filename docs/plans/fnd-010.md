# Plan: FND-010 — Tailwind + shadcn install with theme

**Status:** IMPLEMENTED
**Mode:** collaborative
**Author:** planner agent

## Goal

Establish the visual foundation for the web app: Tailwind CSS, shadcn/ui
primitives, the stone-base + amber-accent theme from `docs/ui-design-standards.md`,
Inter typography, and a working system/light/dark theme toggle. This unblocks
every subsequent UI ticket — FND-011 (TanStack Query demo on `/`) and all of
epics 03–09 build on these primitives and CSS variables. The user-visible
outcome is that `/` stops being unstyled HTML and renders a properly themed
shadcn `Button` with a working theme switcher, matching the "Sunday afternoon
with friends, not spreadsheet at the office" character.

## Acceptance criteria (from ticket)

- `/` renders a Button using shadcn primitives
- Theme toggle works (system / light / dark)
- Colors match the standards (stone base, amber primary)
- Inter font loads

## Context found during research

- `apps/web` is a Vite 8 + React 19 + TanStack Router app (FND-009, DONE). No
  Tailwind, no CSS file, no `index.css` import. `index.html` has no font links.
- Root `prettier.config.js` already loads `prettier-plugin-tailwindcss` and
  `package.json` already has it as a devDependency — Tailwind class sorting is
  expected to work the moment Tailwind is present.
- `tsconfig.base.json` is strict: `verbatimModuleSyntax: true`,
  `noUnusedLocals: true`, `noUnusedParameters: true`,
  `noUncheckedIndexedAccess: true`. shadcn-generated files must typecheck under
  these (notably: `import type` for type-only imports; no unused params).
- `apps/web/tsconfig.json` already defines the `@/*` path alias and
  `vite.config.ts` already defines the matching `@` resolve alias — shadcn's
  init alias step is effectively already satisfied; no change needed there.
- ESLint flat config (`eslint.config.js`) globally ignores
  `apps/web/src/lib/api-client/**` and `routeTree.gen.ts` but NOT
  `src/components/ui/**`. shadcn-generated components must pass our ESLint
  (`no-unused-vars` as error) and `tsc`.
- The standards doc predates Tailwind v4. It names `tailwind.config.ts` and
  `apps/web/src/styles/globals.css`. Current shadcn + Tailwind v4 convention is
  CSS-first config (`@import "tailwindcss"` + `@theme inline` in a single CSS
  file, no `tailwind.config.ts`, `@tailwindcss/vite` plugin). See "Risks and
  open questions" — this plan uses Tailwind v4 and updates the standards doc to
  match (per code-standards.md § Documentation and spec sync).

## Approach

### 1. Install dependencies

Add to `apps/web/package.json` (exact versions resolved by the implementer to
current stable; ranges shown):

- `dependencies`: `tailwindcss`, `@tailwindcss/vite`, `class-variance-authority`,
  `clsx`, `tailwind-merge`, `lucide-react`, `tw-animate-css`, `sonner`,
  `next-themes` is **not** used (Vite has no next-themes; use shadcn's own
  context provider per shadcn Vite dark-mode docs), `@fontsource-variable/inter`
  (variable Inter; standards say `@fontsource/inter` — variable package is the
  current fontsource convention and lighter; documented as an assumption).
- Form stack: `react-hook-form`, `@hookform/resolvers`, `zod` (Zod is already a
  backend dep; the web app needs its own copy for the Form primitive).
- Radix primitives are pulled transitively by the shadcn CLI per component; do
  not hand-add them.
- `devDependencies`: `@types/node` (shadcn Vite guide uses `path` in
  `vite.config.ts`; we already use `node:url`/`fileURLToPath` so this may be
  unnecessary — implementer keeps the existing `fileURLToPath` alias and skips
  `@types/node` unless a config needs it).

### 2. Wire Tailwind v4 into Vite

- Modify `apps/web/vite.config.ts`: add `@tailwindcss/vite` plugin to the
  `plugins` array (alongside existing `tanstackRouter` and `react`). Keep the
  existing `fileURLToPath`-based `@` alias as-is.

### 3. Create the global stylesheet with the theme

- Create `apps/web/src/styles/globals.css` (keep the standards-documented path).
  Contents:
  - `@import "tailwindcss";`
  - `@import "tw-animate-css";` (animation utilities shadcn references)
  - `@custom-variant dark (&:is(.dark *));` (Tailwind v4 class-based dark mode)
  - `@fontsource-variable/inter` import (or import in `main.tsx` — see step 4;
    pick one, not both).
  - `:root { ... }` block defining shadcn base vars (`--background`,
    `--foreground`, `--card`, `--popover`, `--primary`, `--primary-foreground`,
    `--secondary`, `--muted`, `--muted-foreground`, `--accent`, `--destructive`,
    `--border`, `--input`, `--ring`, `--radius`) using **Stone** values for
    neutrals and **amber-500** for `--primary`.
  - `.dark { ... }` block with the dark values (stone-950/900 backgrounds,
    stone-50 foregrounds, amber-500 primary tuned for dark contrast).
  - Our domain-semantic vars in BOTH `:root` and `.dark`, exactly per the
    standards table:
    - `--success` / `--success-foreground` — light `emerald-600`/white, dark
      `emerald-400`/`stone-950`
    - `--error` / `--error-foreground` — light `red-600`/white, dark
      `red-400`/`stone-950`
    - `--push` / `--push-foreground` — light `stone-500`/white, dark
      `stone-400`/`stone-950`
    - `--warning` / `--warning-foreground` — light `amber-600`/white, dark
      `amber-400`/`stone-950`
  - `--radius: 0.5rem` (standards: `rounded-lg`).
  - `@theme inline { ... }` block mapping every var (shadcn base + our four
    semantic pairs) to Tailwind theme tokens so utilities like `bg-primary`,
    `text-success`, `bg-push`, `border-warning`, `text-muted-foreground` exist.
    This is the Tailwind v4 replacement for the standards-doc's
    `tailwind.config.ts` mapping.
  - Set the default sans font stack to Inter variable + system fallback via
    `--font-sans` in `@theme` and a base `body` rule, with
    `font-feature-settings` enabling `tabular-nums` opt-in via the
    `tabular-nums` utility (no global override).
  - Base layer: `* { border-color: var(--color-border); }` and
    `body { background: var(--color-background); color: var(--color-foreground); }`
    per shadcn convention.
- Color values: use OKLCH (shadcn v4 default) or hex/HSL — implementer matches
  shadcn's current `init` output format for the base vars and hand-authors the
  four semantic pairs from Tailwind's published Stone/emerald/red/amber values.

### 4. Load the stylesheet and font

- Modify `apps/web/src/main.tsx`: add `import "./styles/globals.css";` and the
  `@fontsource-variable/inter` import (single source of truth — do not also
  import the font in CSS).
- No change to `index.html` (no Google Fonts link; fontsource is bundled, which
  the standards mandate over CDN fonts).

### 5. shadcn config

- Create `apps/web/components.json` describing: style `new-york` (shadcn
  current default; document choice), RSC `false`, TSX `true`, Tailwind config
  `""` (v4 — no config file), CSS path `src/styles/globals.css`, base color
  `stone`, CSS variables `true`, aliases (`@/components`, `@/lib/utils`,
  `@/components/ui`, `@/hooks`), icon library `lucide`.
- Create `apps/web/src/lib/utils.ts` with the standard `cn()` helper
  (`clsx` + `tailwind-merge`). Explicit return type `string` per
  code-standards.md (exported function).

### 6. Add the starter shadcn component set

Generate into `apps/web/src/components/ui/` via the shadcn CLI (implementer runs
`pnpm dlx shadcn@latest add ...`):

- Ticket-specified: `button`, `card`, `input`, `label`, `form`, `dialog`,
  `sonner`, `skeleton`, `sheet`, `table`, `badge`.
- **Additional, required to satisfy the "theme toggle works" criterion:**
  `dropdown-menu` (the shadcn Vite ModeToggle pattern is a Button +
  DropdownMenu). Flagged in Risks as a deliberate scope addition driven by an
  acceptance criterion, not scope creep.
- After generation, review each file for our standards: `import type` for
  type-only imports (`verbatimModuleSyntax`), no unused vars, explicit return
  types on exported components/functions where shadcn omits them, no `any`
  without inline comment. Fix in place (shadcn files are owned, per
  code-standards.md § shadcn/ui).

### 7. Theme provider + mode toggle

- Create `apps/web/src/components/theme-provider.tsx`: a context provider with
  theme state `"system" | "light" | "dark"`, default `"system"` (standards:
  "system-preference default" — overrides shadcn doc's `"dark"` default),
  persisted to `localStorage` under key `picksleagues-ui-theme`, applying
  `light`/`dark` class to `document.documentElement`, and reacting to
  `window.matchMedia("(prefers-color-scheme: dark)")` changes while in
  `system` mode. Hooks ordered state → derived → effects → handlers per
  code-standards.md § Frontend. Export a `useTheme` hook with explicit return
  type.
- Create `apps/web/src/components/mode-toggle.tsx`: Button (`variant="outline"`,
  `size="icon"`) + DropdownMenu with System / Light / Dark items, animated
  Sun/Moon Lucide icons, `aria-label` per accessibility standards
  (color/icon not the only signal — items have text labels).
- Modify `apps/web/src/routes/__root.tsx`: wrap `<Outlet />` in
  `<ThemeProvider>`, and render `<Toaster />` (shadcn Sonner wrapper) once at
  the root so toasts work app-wide.

### 8. Demo on `/`

- Modify `apps/web/src/routes/index.tsx`: render a small themed surface proving
  the acceptance criteria — a `Card` containing a heading, a shadcn `Button`,
  and the `ModeToggle`. Mobile-first layout (centered, `p-4`/`p-6`, no
  horizontal scroll at 375px), uses semantic classes only
  (`bg-background`, `text-foreground`, `bg-primary`), no raw
  `bg-amber-500`/`text-stone-*` per standards § Color. Keep it minimal — this
  is a foundation demo, not a real screen.

### 9. Update the UI standards doc (required by code-standards.md § Doc sync)

- Modify `docs/ui-design-standards.md`: the doc currently states the theme is
  "Mapped into the Tailwind theme via `tailwind.config.ts`" and font via

  `@theme inline`, no `tailwind.config.ts`) and `@fontsource-variable/inter`.
  Fix the prose in place to state the new decision + one-line rationale, and
  add a dated entry to the doc's Revision log. (If no Revision log section

## Tests

Per code-standards.md § Frontend: "Static display components don't need tests."
This ticket adds no business logic. The substantive interactive unit is the
theme provider.

- Add `apps/web/src/components/theme-provider.test.tsx` (Vitest + React Testing
  Library) covering:
  - default theme is `system` when nothing in localStorage
  - `setTheme("dark")` adds the `dark` class to `documentElement` and persists
    to localStorage
  - `setTheme("light")` removes the `dark` class
  - `setTheme("system")` resolves from a mocked
    `matchMedia("(prefers-color-scheme: dark)")` (both true and false cases)
  - reading an existing persisted value from localStorage on mount
- This requires Vitest + RTL wiring in `apps/web` (currently `test` is a stub
  echo). Adding `vitest`, `@testing-library/react`, `@testing-library/dom`,
  `jsdom`, a `vitest.config.ts` (or merged into `vite.config.ts`), and a real
  `test` script is **in scope** because it is the first frontend test and the
  provider is non-trivial. If the orchestrator prefers to defer the web test
  harness to its own ticket, see clarifying question 1.
- shadcn `ui/*` primitives, `mode-toggle`, and the `/` demo are static/thin
  wrappers — no tests, per standards.

## Risks and open questions

- **Tailwind v4 vs. standards doc (v3 assumptions).** The standards name
  `tailwind.config.ts`. Tailwind v4 + current shadcn is CSS-first with no
  config file. Plan: use v4 (current, what `shadcn init` produces, smaller
  surface) and update the standards doc in the same PR per code-standards.md
  § Documentation and spec sync. This is the conforming path, not a contradiction.
- **`dropdown-menu` not in the ticket's component list.** It is required by the
  shadcn ModeToggle pattern, and "theme toggle works" is an explicit acceptance
  criterion. Added deliberately; not scope creep.
- **`@fontsource-variable/inter` vs. standards' `@fontsource/inter`.** Variable
  font is the current fontsource convention, one weight axis file, satisfies the
  weight set (400/500/600/700) the standards require. Documented as an
  assumption; doc updated to match.
- **First frontend test harness.** Wiring Vitest+RTL into `apps/web` is net-new
  infra. Reasonable to include since the provider genuinely needs a test, but
  the orchestrator may want it isolated. See clarifying question 1.
- **shadcn CLI generates non-conforming code** (default exports, missing return
  types, value imports for types under `verbatimModuleSyntax`). Mitigated by the
  explicit post-generation review step (6/7). The implementer must run
  `pnpm typecheck` and `pnpm lint` and fix generated files in place; do not add
  ESLint disables or loosen tsconfig.

## Implementation notes

**Actually shipped:** Tailwind v4 (CSS-first, `@tailwindcss/vite` plugin, no config file) + 12 shadcn/ui primitives (button, card, input, label, form, dialog, sonner, skeleton, sheet, table, badge, dropdown-menu) installed in `apps/web` with new-york style and stone base color. Global stylesheet in `apps/web/src/styles/globals.css` defines shadcn base vars + four domain-semantic pairs (success/emerald, error/red, push/stone, warning/amber) in OKLCH, mapped to Tailwind theme via `@theme inline`. Custom ThemeProvider context (system default, localStorage persistence, matchMedia reactivity) + ModeToggle component. Inter variable font via `@fontsource-variable/inter`. First frontend test harness stood up (Vitest + RTL + jsdom) with ThemeProvider unit test covering default theme, state changes, localStorage persistence, and matchMedia reactivity (6 test cases). Demo on `/` shows Button + Card + ModeToggle. All generated shadcn files conformed to code standards (type-only imports, explicit return types, no unused vars).

**Decisions made during implementation:**
- Removed `next-themes` (Vite SPA; custom ThemeProvider used instead for full control over system preference reactivity and localStorage behavior)
- Stripped `"use client"` directives from generated shadcn files (not needed in Vite SPA)
- Added `tslib` as direct dependency to `apps/web` (Vite 8 and pnpm isolated layout cannot resolve it transitively via `react-remove-scroll`)
- Tailwind v4 CSS-first config (`@theme inline` in `globals.css`, no `tailwind.config.ts`) per current shadcn convention; differs from pre-v4 standards doc assumption
- First frontend test harness included in scope (provider has genuine business logic; wiring Vitest+RTL is load-bearing for future web tickets)
