# Plan: FND-020 — Production migration workflow

**Status:** IMPLEMENTED
**Mode:** collaborative
**Author:** planner agent

## Goal
Make database migrations run automatically — and gate the deploy — every time the
app ships, so the live Neon schema is always in sync with the API code before that
code starts serving traffic. On a production deploy (and on every preview deploy),
the committed Drizzle migrations in `services/api/src/db/migrations/` are applied to
the deploy's target database *before* the API function goes live; if a migration
fails, the build fails and nothing ships (no partial deploys). This is the template
every future schema change rides on: edit schema → `db:generate` (FND-019 gate keeps
the migration committed) → migrate-on-deploy (this ticket) applies it.

## Acceptance criteria (from ticket)
- Vercel build step runs migrations before API boot
- Migration succeeds or deployment fails (no partial deploys)
- Post-deploy, API can read and write the latest schema

## Critical reconciliation: `push:pg` vs. generate + migrate

The ticket text says "Vercel build step includes `pnpm drizzle-kit push:pg` before
API boot." **This wording is stale and must not be implemented as written.** Three
facts from the actual repo override it:

1. **The repo already uses generated SQL migrations, not push.** There is a committed
   migration (`services/api/src/db/migrations/0000_happy_ravenous.sql`) with a Drizzle
   journal (`.../migrations/meta/_journal.json`), a `db:generate` script
   (`drizzle-kit generate`), and a `db:migrate` runner (`tsx src/db/migrate.ts` →
   `services/api/src/db/migrate.ts`). `drizzle.config.ts` is configured for
   `generate`/`migrate` (`out: "./src/db/migrations"`), not `push`.

2. **FND-019 is built entirely around committed migrations.** `schema-drift-check.yml`
   runs `pnpm db:generate:check` on every PR and fails if a schema edit has no
   committed migration. `push:pg` bypasses migration files entirely, which would make
   FND-019 meaningless and break the stated "always in sync" guarantee.

3. **`push:pg` is the wrong tool for production.** `drizzle-kit push` diffs the schema
   against the live DB and applies changes directly with no migration history, no
   review artifact, and a real risk of destructive auto-decisions (column drops,
   data-loss prompts) running unattended in CI. It also isn't valid Drizzle-Kit syntax
   in this version (`drizzle-kit 0.31.x` uses `drizzle-kit push`, not `push:pg`).

**Recommendation (and what this plan implements): reuse the existing
`pnpm db:migrate` path** (`services/api/src/db/migrate.ts`, which already applies
committed migrations in order via the Neon WebSocket migrator for non-local hosts).
This is safe, reviewable, idempotent, history-tracked, and complements FND-019 instead
of contradicting it. The plan documents this divergence and proposes updating the
ticket's acceptance-criteria wording. **This is a flagged risk, not a blocker** — see
"Risks and open questions."

## How this repo actually deploys (grounding for "where the step hooks in")

Investigated `.github/workflows/`, `vercel.json`, `scripts/build-vercel-output.sh`,
and `docs/deploy.md`:

- **There is no GHA deploy workflow.** All three workflows (`ci.yml`,
  `api-client-check.yml`, `schema-drift-check.yml`) trigger on `pull_request` only.
  None push prebuilt output to Vercel. The MEMORY note "hand-rolled GHA→Vercel" refers
  to the *build* being hand-rolled (Build Output API in `scripts/build-vercel-output.sh`
  instead of Vercel auto-detection) — **not** a GHA-orchestrated deploy. Deploys are
  driven by **Vercel's own Git integration**: per `docs/deploy.md` §"Deploy workflow",
  a PR triggers a Vercel preview build and merging to `main` triggers a Vercel
  production build.

- **Vercel runs `pnpm vercel:build`** (`vercel.json` → `buildCommand: "pnpm vercel:build"`
  → `scripts/build-vercel-output.sh`). This is the *only* step under our control that
  Vercel executes on every deploy, and **a non-zero exit from it aborts the deploy
  before the new function is published.** That is exactly the gate FND-020 needs.

**Conclusion: the migration step hooks into `scripts/build-vercel-output.sh`.** Running
`db:migrate` there, before (or as) the build proceeds, means: migrations run against the
deploy's `DATABASE_URL` before the function image is published; a migration failure
fails the build with a non-zero exit, so the old function keeps serving and nothing
partial ships. There is no separate "API boot" step to hook before — the serverless
function "boots" lazily on first request *after* the deploy is published, so gating the
build is the correct and only interpretation of "before API boot" in this architecture.

## Approach

1. **Add a build-time migration step to `scripts/build-vercel-output.sh`.** Near the top,
   after `set -euo pipefail` and before/around the web+function build, invoke the existing
   migrate runner:

   ```sh
   echo "Applying database migrations..."
   pnpm --filter @picksleagues/api db:migrate
   ```

   Because the script already runs under `set -euo pipefail`, a non-zero exit from
   `db:migrate` aborts the whole script → Vercel marks the deploy failed → no publish.
   This satisfies "migration succeeds or deployment fails."

   **Placement decision:** run migrations *first*, before the web build and esbuild
   bundle. Rationale: migrations are the riskiest, most-likely-to-fail step; failing fast
   avoids wasting a build, and there is no ordering dependency from the build on the DB.
   (Alternative — run after bundling — is also correct for the no-partial-deploy guarantee
   since publish happens only after the whole script exits 0; migrate-first is just
   cheaper on failure. Documented as an assumption.)

2. **Make `db:migrate` safe to run in the Vercel build environment.** Two concerns to
   verify and, if needed, handle in `services/api/src/db/migrate.ts`:
   - **`DATABASE_URL` presence.** `migrate.ts` already `process.exit(1)`s with a clear
     message if `DATABASE_URL` is unset. In Vercel, env vars are available to the build,
     so production/preview builds will have it (`docs/deploy.md` §"Environment variables"
     lists `DATABASE_URL` as required for both). No code change needed, but the plan calls
     out that **`DATABASE_URL` must be set for every Vercel environment that builds** — if
     a preview build has no DB URL, the build will (correctly) fail. (See open question on
     preview behavior.)
   - **Driver selection.** `migrate.ts` already selects the Neon WebSocket migrator for
     non-local hosts via `isLocalDatabaseUrl` — correct for Neon prod/preview. No change.

   The implementer should confirm `migrate.ts` needs no change; if `dotenv/config` import
   causes issues when no `.env` file exists in the Vercel build (it shouldn't — dotenv
   silently no-ops on a missing file and real env vars still win), leave it.

3. **Guard against the build needing `services/api` build tooling at deploy time.**
   `db:migrate` runs via `tsx` (already a `services/api` devDependency) and `vercel.json`'s
   `installCommand` is `pnpm install --frozen-lockfile` (installs devDependencies). So
   `tsx`, `drizzle-orm`, `@neondatabase/serverless`, `pg`, and `ws` are all present in the
   Vercel build container. No dependency changes required. The implementer should verify
   Vercel's install step does not prune devDependencies (default: it does not, for the
   build phase).

4. **Document the deploy-time migration in `docs/deploy.md`.** Add a short subsection under
   §"Build pipeline" (or a new §"Database migrations on deploy") stating: migrations run as
   the first step of `pnpm vercel:build`; a failed migration fails the deploy; production
   and preview each migrate their own `DATABASE_URL` target; this is the template for all
   schema changes (generate + commit via FND-019, applied on deploy here). Cross-reference
   FND-019 and the `db:migrate` script.

5. **Document it in `services/api/README.md`.** Add one line under §"Apply migrations" /
   §"Migration hygiene" noting that production/preview apply migrations automatically at
   deploy time via `vercel:build`; humans only run `db:migrate` for local Docker / manual
   Neon-branch setup.

## Tests
This ticket is deploy-pipeline plumbing; per CLAUDE.md "No DB integration tests for MVP,"
there is no unit test that runs a real migration. Verification is operational and falls
under the spine/E2E and manual-verification posture:

- **No new Vitest tests** — there is no pure domain logic here to table-test. (Stating
  this explicitly so the reviewer doesn't expect them.)
- **Build-script smoke check (manual / reviewer):** with a reachable test `DATABASE_URL`
  (a throwaway Neon branch or local Docker), run `pnpm vercel:build` and confirm it applies
  migrations and exits 0; then point `DATABASE_URL` at an unreachable/invalid DB and confirm
  the script exits non-zero and stops before publishing artifacts (proves the gate).
- **Post-deploy acceptance (manual, per `docs/deploy.md` §"Verification"):** after a real
  deploy, hit `GET /api/health` and exercise a read+write path against the latest schema
  (the existing health check plus any DB-backed route available at the time) to satisfy
  "API can read and write the latest schema."

If POL-003 / spine E2E later add an automated post-deploy smoke that performs a DB
write, that becomes the durable regression guard for this criterion — out of scope here.

## Risks and open questions

1. **`push:pg` wording conflict (FLAGGED — recommend ticket edit).** Implementing the
   ticket verbatim would contradict the entire committed-migration approach and FND-019.
   This plan implements `pnpm db:migrate` instead and recommends updating FND-020's
   acceptance criteria from "includes `pnpm drizzle-kit push:pg`" to "runs
   `pnpm --filter @picksleagues/api db:migrate` (committed migrations) as part of
   `pnpm vercel:build`." **Confirm this substitution is acceptable** (high confidence it
   is, given FND-019 and existing infra — proceeding under this assumption in collaborative
   mode unless told otherwise).

2. **Preview deploys and `DATABASE_URL`.** Today (`docs/deploy.md`) preview `DATABASE_URL`
   is a single Vercel env var, and per-PR Neon branches are FND-022 (not yet built).
   Once migrations run in `vercel:build`, **every preview deploy will migrate whatever
   `DATABASE_URL` preview points at.** If that is one shared preview database, every PR's
   migrations land there (usually fine — they're committed/forward-only — but worth noting).
   FND-022 later gives each PR its own branch and runs migrations against it; this ticket's
   build-step hook is exactly what FND-022 reuses. **Open question:** is preview pointed at
   a disposable shared DB today, and is auto-migrating it acceptable until FND-022? (If
   preview has no `DATABASE_URL`, preview builds will start failing — that would need either
   a preview DB URL set or a guard. See assumption.)

3. **Migration runtime vs. build timeout.** Long migrations on large tables could approach
   Vercel's build time limits. Not a concern at MVP (single tiny migration, no production
   data), but noted for when real data exists. No action now.

4. **No automatic rollback.** If a migration applies a forward change and the *function*
   bundle then fails to publish for an unrelated reason, the DB is migrated but the old
   code still serves. Mitigated by migrate-forward-only discipline and additive migrations
   (already the documented hygiene in `services/api/README.md`). Out of scope to add
   rollback tooling.

## Assumptions (collaborative — proceeding unless the parent objects)
- Substitute `pnpm db:migrate` for the ticket's `push:pg` wording (Risk 1). This is the
  single load-bearing decision; everything else follows from the existing repo.
- Run migrations as the **first** step of `scripts/build-vercel-output.sh` (fail fast).
- Preview `DATABASE_URL` is already set (per `docs/deploy.md`) and auto-migrating the
  preview target is acceptable in the interim before FND-022. If preview has no DB URL,
  that's surfaced as Open Question 2 rather than silently worked around.
- No code change to `migrate.ts` is needed; only the build script + docs change. Implementer
  verifies during execution.

## Out of scope
- **Per-PR Neon branches and preview env-var provisioning** — that's FND-022 (which depends
  on FND-020 and reuses this build-step hook). Do not build branch creation/cleanup here.
- **A separate GHA deploy workflow** — the repo intentionally uses Vercel's Git integration
  for deploys; do not introduce a parallel GHA→Vercel deploy job.
- **Switching the project to `drizzle-kit push`** — explicitly rejected (see reconciliation).
- **Automated post-deploy smoke tests** — POL-003 territory.
- **Migration rollback / down-migration tooling** — not part of MVP hygiene.
- **Changing FND-019's drift check** — it stays as-is; this ticket complements it.
