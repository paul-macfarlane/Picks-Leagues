# Plan: FND-006 ‚Äî Zod request validation middleware

**Status:** IMPLEMENTED
**Mode:** collaborative
**Author:** planner agent

## Goal

Establish the request-validation convention for the entire API: every route validates its inputs at the HTTP boundary with Zod via `@hono/zod-validator`, one schema colocated with each route handler, and invalid input is rejected with a consistent `400` + structured error body. This is the boundary-validation layer that the MVP spec assigns to the API ("Validating request bodies against Zod schemas") and that `code-standards.md` mandates ("Validate at system boundaries only ‚Ä¶ Zod"). It also unblocks FND-007 (OpenAPI export), which reuses these Zod schemas as the spec source.

## Acceptance criteria (from ticket)

- Route with Zod body schema rejects invalid payloads with `400` + structured error
- Sample route demonstrates header, query, and body validation
- Convention documented in `services/api/README.md`
- Establish the convention of one Zod schema per route, colocated with the handler

## Approach

1. **Add dependencies** to `services/api/package.json`: `zod` and `@hono/zod-validator` (runtime `dependencies`, matching how `hono` is listed). Versions resolved at install by the implementer; `@hono/zod-validator` must be compatible with `hono ^4.12` and the installed `zod` major.

2. **Create a shared validation-error formatter** at `services/api/src/lib/validation.ts`:
   - Export a `formatZodError(result, c)` style hook compatible with `@hono/zod-validator`'s `(result, c)` callback signature.
   - On `result.success === false`, return `c.json(<structured error>, 400)`.
   - Structured error shape (single canonical shape, used by every validated route):
     ```jsonc
     {
       "error": "ValidationError",
       "issues": [
         { "path": "body.email", "code": "invalid_string", "message": "Invalid email" }
       ]
     }
     ```
     `path` is the Zod issue path joined with `.`, prefixed by the validation target (`body` | `query` | `header` | `param`) so a consumer can tell which part failed. Derive `issues` from `result.error.issues` (Zod's flattened issue list) ‚Äî no `as` casts; the `@hono/zod-validator` result type is discriminated, so narrowing on `result.success` is sufficient.
   - Export a small helper, e.g. `zBody(schema)`, `zQuery(schema)`, `zHeader(schema)`, `zParam(schema)`, that each wrap `zValidator(target, schema, formatZodError)` so route files don't repeat the formatter wiring. This keeps "one place owns the error shape" (DRY for behavior, per code-standards ¬ß Dead code and DRY).
   - Explicit return types on all exported functions (code-standards ¬ß TypeScript).

3. **Create the sample echo route** at `services/api/src/routes/echo.ts`, following the exact shape of `health.ts` (factory `createEchoRoute(): Hono`, named export, no default):
   - One Zod schema object colocated at the top of the file (the convention this ticket establishes). Define a body schema, a query schema, and a header schema. Suggested minimal shape:
     - Header: require `x-echo-token` (non-empty string).
     - Query: optional `repeat` coerced to an integer with a sane bound (e.g. `1`‚Äì`5`, default `1`).
     - Body: `{ message: string (1..280), shout?: boolean }`.
   - A single `POST /` handler that applies header, query, and body validators (via the `zBody`/`zQuery`/`zHeader` helpers), reads validated values through `c.req.valid("body" | "query" | "header")`, and echoes back `{ message, repeated: string[], shout }` so behavior is observable in a test.
   - Explicit `Response` return type on the handler, matching `health.ts`.
   - "Protected" per the ticket description is satisfied at this stage by the required `x-echo-token` header (Better Auth middleware does not exist until a later foundation ticket ‚Äî see Risks). The header validation is the demonstrable protection mechanism here.

4. **Mount the route** in `services/api/src/app.ts`: `app.route("/api/echo", createEchoRoute());` placed alongside the existing health mount. No change to the `AppDeps`/clock wiring (echo needs no clock).

5. **Tests** ‚Äî `services/api/src/routes/echo.test.ts`, mirroring `health.test.ts` (Vitest, `app.request`, no DB, no mocks). See Tests section for the case list.

6. **Document the convention** in `services/api/README.md` under the "Adding new routes" section: add a "Request validation" subsection stating (a) every route validates inputs with Zod via `@hono/zod-validator`; (b) exactly one Zod schema per route, colocated at the top of the route file (no shared `schemas/` directory ‚Äî colocation is the rule); (c) use the `zBody`/`zQuery`/`zHeader`/`zParam` helpers from `src/lib/validation.ts` so the `400` error shape stays uniform; (d) show the canonical structured error JSON; (e) cross-link `docs/code-standards.md` ¬ß Error handling. Reference the new `/api/echo` route as the worked example.

## Tests

`services/api/src/routes/echo.test.ts` (Vitest, via `createApp()` + `app.request`, no DB/mocks):

- `valid request ‚Üí 200 and echoes message` ‚Äî valid header + body, default `repeat`, asserts `repeated.length === 1` and message echoed.
- `query repeat is honored` ‚Äî valid request with `?repeat=3`, asserts `repeated.length === 3`.
- `invalid body ‚Üí 400 with structured error` ‚Äî body missing `message` (or `message` too long); asserts `status === 400`, `body.error === "ValidationError"`, and an issue with `path` starting `body.`.
- `missing required header ‚Üí 400 with structured error` ‚Äî omit `x-echo-token`; asserts `400` and an issue with `path` starting `header.`.
- `out-of-range query ‚Üí 400 with structured error` ‚Äî `?repeat=99`; asserts `400` and an issue with `path` starting `query.`.
- `error body shape is the documented contract` ‚Äî assert the exact top-level keys (`error`, `issues`) and that each issue has `path`, `code`, `message`. This pins the convention so FND-007 and the frontend can rely on it.

No domain/scoring tests apply ‚Äî this ticket has no business logic; it is pure boundary plumbing.

## Risks and open questions

- **`zod` major version.** Zod v4 changed some issue/error internals (`error.issues` still exists; `flatten`/`format` APIs shifted). The formatter must read `result.error.issues` rather than rely on `.flatten()` specifics, and the implementer should pin whatever `@hono/zod-validator` peer-depends on. Low risk, flagged so the implementer verifies against the installed versions rather than assuming v3.
- **"Protected" wording in the ticket.** Better Auth middleware is a later foundation ticket; it is not available now. This plan satisfies "protected" via a required header (`x-echo-token`) as the demonstrable gate and validation target. If the human intends the echo route to be wired to real auth, that work does not exist yet and would belong to the auth ticket, not FND-006.
- **Helper surface (`zBody`/`zQuery`/...).** This is a thin convenience wrapper, not an abstraction layer. It exists so the `400` shape lives in exactly one place. If the reviewer prefers routes call `zValidator(..., formatZodError)` directly, that is an easy reduction ‚Äî flagged so it is a conscious choice, not a default.

## Out of scope

- OpenAPI generation / `@hono/zod-openapi` ‚Äî that is FND-007. Use plain `@hono/zod-validator` here; do not pre-adopt the OpenAPI-flavored Hono just to "save work later."
- Better Auth / real authentication middleware ‚Äî separate foundation ticket.
- Global error-handling overhaul ‚Äî keep the existing `app.onError` 500 handler in `app.ts` as is; this ticket only adds the `400` validation path.
- A shared `schemas/` directory or schema registry ‚Äî explicitly rejected; the established convention is colocation, one schema per route file.
- Validating any real domain route (leagues, picks, etc.) ‚Äî only the sample echo route is in scope.

## Clarifying questions

None. The ticket, code-standards, MVP spec, and existing FND-005 structure fully determine the approach. Proceeding on the documented assumptions above (header-based "protected", single canonical error shape, colocated schemas). Plan ready for implementer handoff.

## Implementation notes

**Actually shipped:** Added `zod@^4.0.0` (resolved 4.4.3) and `@hono/zod-validator@^0.8.0` as runtime dependencies. Created `services/api/src/lib/validation.ts` exporting `zBody`, `zQuery`, `zHeader`, `zParam` helpers, each wrapping `zValidator` to return the properly discriminated result type and format validation failures with the canonical 400 error shape `{ error: "ValidationError", issues: [{ path, code, message }] }` where `path` is prefixed by the validation target. Sample echo route at `services/api/src/routes/echo.ts` validates header (`x-echo-token`, required), query (`repeat`, optional int 1‚Äì5), and body (`message` + `shout`), mounted at `/api/echo` in `app.ts`. Tests in `services/api/src/routes/echo.test.ts` cover valid requests, header/query/body failures, error shape contract, and repeat honored (6 test cases; all pass).

**Decisions made during implementation:**
- Dropped the planned `makeFormatter` factory and hand-rolled `result` type in favor of inlining the hook body in each helper. This allowed `@hono/zod-validator`'s overload resolution to provide the properly discriminated `result` type, eliminating all `as`/`any` casts (a review requirement). Trade-off: ~6 lines of hook-body duplication across the four helpers, consciously accepted.
- Removed the planned `code ?? "custom"` fallback. Zod v4 guarantees `$ZodIssue.code` is always present (discriminated union); the fallback was unnecessary.
- "Protected" satisfied via required `x-echo-token` header (Better Auth is a later foundation ticket and does not exist yet). Recorded as an accepted assumption; real auth middleware will be wired in FND-014.
