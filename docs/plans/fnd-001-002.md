## Goal

Stand up the empty-but-correct monorepo foundation that every later Foundation ticket builds on: a pnpm workspace with `apps/web` and `services/api` packages, root tooling files (`.editorconfig`, `.gitignore`, `.npmrc`), a documented root README, and a shared `tsconfig.base.json` enforcing the strict TypeScript posture mandated by `docs/code-standards.md` (`strict`, `noUncheckedIndexedAccess`, `noImplicitOverride`). Nothing user-visible ships here; the deliverable is a repo where `pnpm install`, `pnpm -r build`, and `pnpm -r typecheck` all succeed on minimal real stubs, so FND-003 (lint), FND-004 (Drizzle), FND-005 (Hono), and FND-009 (Vite) can drop their real code into a structure that already conforms.

## Acceptance criteria (mapped to steps)

### FND-001
- `pnpm install` succeeds at the repo root ‚Üí **steps 1, 2, 3, 4, 8** (workspace + package manifests + `.npmrc`)
- `pnpm -r build` runs (even with empty stubs) ‚Üí **steps 2, 3, 4** (root `build` fans out; every package has an explicit echo no-op `build`)
- Root README documents how to install, build, dev, test ‚Üí **step 9**

### FND-002
- Both packages compile under strict mode ‚Üí **steps 10, 11, 12, 13** (`tsconfig.base.json` strict trio + per-package extends + non-vacuous `src/index.ts` so `pnpm -r typecheck` is a real assertion)
- A test file using `any` triggers an ESLint warning ‚Üí **DEFERRED to FND-003 by dependency design** ‚Äî partially satisfied now via strict tsconfig posture (steps 10‚Äì13) + README code-standards reference (step 9); the lint assertion itself is seeded as a rename-and-assert task via the committed `.txt` fixture in **step 15**. See Decisions D5 and the FND-003 hand-off note.
- Code standards referenced from root README ‚Üí **step 9**

## Decisions (resolved with human; previously open questions)

- **D1 ‚Äî Version pins.** Node 22 LTS, pnpm 10, TypeScript `^5` (latest 5.x). Root `package.json` sets `"engines": { "node": ">=22 <23" }` and `"packageManager": "pnpm@10.x.x"` (pin a concrete current 10.x patch at implementation time). `typescript` is a root `devDependency` at `^5`.
- **D2 ‚Äî Stub script mechanism.** Explicit `echo` no-op scripts that exit 0, easy to find/replace later. Form: `"build": "echo '@picksleagues/web: no build step yet'"` (substitute the real package name + script name per package/script). No silent `exit 0`; no omitted scripts.
- **D3 ‚Äî Bootstrap source.** Each package gets a minimal self-consumed `src/index.ts` exporting one real const so FND-002's strict-mode compilation check is non-vacuous. One-line WHY comment naming the owning replacement ticket (FND-005 for api, FND-009 for web) so the dead-code/no-stub standard is satisfied by intent and `knip` (added FND-003) does not later flag it.
- **D4 ‚Äî tsconfig split.** Base stays environment-agnostic (no `lib`/`jsx`/`paths`/`types`); each package overrides `lib`/`jsx`/`paths`/`types`. Confirmed.
- **D5 ‚Äî FND-003 seam.** Defer the ESLint-`any`-warning assertion to FND-003. Do NOT scaffold a throwaway ESLint config now (it would be rewritten in the very next ticket and churn the lockfile twice, risking divergence from the precise FND-003 spec). Instead: satisfy FND-002 now via tsconfig strictness + README reference, and commit a `.txt` lint fixture so FND-003 is a rename-and-assert task. This deferral is documented explicitly here and in the FND-003 hand-off note below.

## Approach

Steps are tagged with the ticket they satisfy. Package naming follows the pnpm workspace-state artifact already present in `node_modules/.pnpm-workspace-state-v1.json` (root pkg `picksleagues`, workspace patterns `apps/*` + `services/*`, package names `@picksleagues/web` and `@picksleagues/api`).

### FND-001 ‚Äî workspace skeleton

1. **[FND-001]** Create `pnpm-workspace.yaml` at repo root:
   ```yaml
   packages:
     - "apps/*"
     - "services/*"
   ```
2. **[FND-001]** Create root `package.json`:
   - `"name": "picksleagues"`, `"private": true`, `"type": "module"`, `"version": "0.0.0"`
   - `"packageManager": "pnpm@10.x.x"` (pin a concrete current 10.x patch) ‚Äî **D1**
   - `"engines": { "node": ">=22 <23" }` ‚Äî **D1**
   - `"devDependencies": { "typescript": "^5" }` ‚Äî shared compiler for both packages ‚Äî **D1**
   - Scripts that fan out so root-level acceptance commands work today and stay stable:
     - `"build": "pnpm -r build"`
     - `"dev": "pnpm -r --parallel dev"`
     - `"test": "pnpm -r test"`
     - `"typecheck": "pnpm -r typecheck"`
     - `"lint": "pnpm -r lint"` (fans out to per-package echo no-op until FND-003)
3. **[FND-001]** Create `apps/web/package.json`:
   - `"name": "@picksleagues/web"`, `"private": true`, `"version": "0.0.0"`, `"type": "module"`
   - Scripts (all present so `pnpm -r <script>` never errors on a missing script) ‚Äî **D2**:
     - `"typecheck": "tsc --noEmit -p tsconfig.json"` (real; exercised by step 5)
     - `"build": "echo '@picksleagues/web: no build step yet'"` (FND-009 replaces with Vite build)
     - `"dev": "echo '@picksleagues/web: no dev server yet'"` (FND-009)
     - `"test": "echo '@picksleagues/web: no tests yet'"` (Vitest arrives with later tickets)
     - `"lint": "echo '@picksleagues/web: no lint configured yet'"` (FND-003)
   - No runtime dependencies. `typescript` inherited from root via workspace hoisting.
4. **[FND-001]** Create `services/api/package.json` ‚Äî same shape as step 3 with `"name": "@picksleagues/api"` and the echo strings substituted (`@picksleagues/api: ...`). `build` no-op replaced by FND-005; `dev` by FND-005; `test` by later tickets; `lint` by FND-003.
5. **[FND-001 / FND-002]** Create one minimal, self-consumed, real bootstrap module per package ‚Äî **D3**:
   - `apps/web/src/index.ts` ‚Äî a single exported `const` (e.g. an app-name string) plus a one-line `// WHY:` comment stating this is a temporary bootstrap entry owned by FND-009. Self-consumed enough that strict `tsc` exercises it.
   - `services/api/src/index.ts` ‚Äî same approach; WHY comment names FND-005 as the owning replacement.
   - These are deliberately real (not empty) so FND-002's "both packages compile under strict mode" is a genuine assertion and so the no-empty-stub code standard is met by documented intent.
6. **[FND-001]** Create `.editorconfig` at root: UTF-8, LF, final newline, trim trailing whitespace, 2-space indent for `*.{ts,tsx,js,json,yaml,md}`.
7. **[FND-001]** Create `.gitignore` at root covering: `node_modules/`, build outputs (`dist/`, `.vercel/`, `.output/`), env files (`.env`, `.env.*`, but **not** `.env.example`), TanStack Router generated route tree (`apps/web/src/routeTree.gen.ts` ‚Äî anticipating FND-009), OS/editor cruft (`.DS_Store`, `.idea/`, `*.log`), and coverage dirs. (Drizzle-specific ignores left for FND-004 to extend.)
8. **[FND-001]** Create root `.npmrc` with the minimum for a clean `pnpm install` consistent with the existing workspace-state artifact (`auto-install-peers=true`, `node-linker=isolated`).
9. **[FND-001 / FND-002]** Rewrite root `README.md`: project one-liner + link to `docs/product-vision.md`; a "Docs" section linking `docs/picks-leagues-mvp-spec.md`, `docs/game-types.md`, `docs/code-standards.md`, `docs/ui-design-standards.md`, `docs/backlog/`; and a "Getting started" section documenting **install** (`pnpm install`), **build** (`pnpm -r build`), **dev** (`pnpm dev`), **test** (`pnpm -r test`). The `docs/code-standards.md` link satisfies FND-001's README criterion **and** FND-002's "code standards referenced from root README" criterion.

### FND-002 ‚Äî shared strict TypeScript config

10. **[FND-002]** Create `tsconfig.base.json` at repo root with the compiler options mandated by `docs/code-standards.md`:
    - `"strict": true`
    - `"noUncheckedIndexedAccess": true`
    - `"noImplicitOverride": true`
    - Forward-compatible environment-agnostic baseline: `"target": "ES2022"`, `"module": "ESNext"`, `"moduleResolution": "Bundler"`, `"esModuleInterop": true`, `"skipLibCheck": true`, `"forceConsistentCasingInFileNames": true`, `"isolatedModules": true`, `"resolveJsonModule": true`, `"verbatimModuleSyntax": true`, `"noUnusedLocals": true`, `"noUnusedParameters": true`.
    - No `lib`/`types`/`jsx`/`paths` here ‚Äî those are package-specific ‚Äî **D4**.
11. **[FND-002]** Create `apps/web/tsconfig.json`:
    - `"extends": "../../tsconfig.base.json"`
    - `"compilerOptions"`: `"lib": ["ES2022", "DOM", "DOM.Iterable"]`, `"jsx": "react-jsx"`, `"types": []` (minimal until FND-009 adds Vite/React types), `"baseUrl": "."`, `"paths": { "@/*": ["./src/*"] }` (the `@/` internal alias the code standards reference; FND-009 wires the matching Vite resolver), `"noEmit": true`
    - `"include": ["src"]`
12. **[FND-002]** Create `services/api/tsconfig.json`:
    - `"extends": "../../tsconfig.base.json"`
    - `"compilerOptions"`: `"lib": ["ES2022"]`, `"types": []` (`@types/node` added by FND-004/FND-005 when Node APIs are used), `"baseUrl": "."`, `"noEmit": true`
    - `"include": ["src"]`
13. **[FND-002]** Confirm each package's `typecheck` script (`tsc --noEmit -p tsconfig.json`) is exercised by the step 5 `index.ts` so `pnpm -r typecheck` green is a real, non-vacuous assertion of "both packages compile under strict mode."
14. **[FND-002]** README code-standards reference is satisfied by step 9 (called out here for criterion traceability).
15. **[FND-002 ‚Üí FND-003 hand-off]** Commit `services/api/src/__lint-fixtures__/any-usage.ts.txt` ‚Äî a `.txt` (so it is neither compiled nor linted yet) containing a minimal snippet that uses bare `any` with no explanatory comment. FND-003 renames it to `.ts` and asserts ESLint warns. This makes FND-002's deferred third criterion a concrete rename-and-assert task rather than a fresh design in FND-003 ‚Äî **D5**.

## FND-003 hand-off note (deferred FND-002 criterion)

FND-002's acceptance criterion *"A test file using `any` triggers an ESLint warning"* is intentionally **not** fully satisfiable in this PR because ESLint configuration is owned by FND-003 (`FND-003 Dependencies: FND-002`). FND-002 ships the substantive half (strict TypeScript posture + README reference). The lint-assertion half is carried forward as an explicit FND-003 deliverable, pre-seeded by the `.txt` fixture in step 15. **FND-003 implementer action:** rename `services/api/src/__lint-fixtures__/any-usage.ts.txt` ‚Üí `.ts`, run ESLint, assert a `@typescript-eslint/no-explicit-any` warning is produced, then re-evaluate whether to keep the fixture or fold it into FND-003's own lint test. **Important:** scope the fixture file OUT of `tsc` (e.g. via a dedicated lint-fixtures `tsconfig` exclude in `services/api/tsconfig.json`, or move it outside `src/`) so that once renamed from `.txt` ‚Üí `.ts` it does not land under `services/api/src/` (which `tsc` includes with `"include": ["src"]`); otherwise strict `noUnusedParameters` and other compiler settings would surface errors before ESLint runs, blocking the assertion. The reviewer of this PR should accept FND-002 as "config-complete, lint-assertion deferred to FND-003 by dependency design."
