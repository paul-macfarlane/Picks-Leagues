# Plan: FND-019 — Drizzle migration drift check (CI gate)

**Status:** IMPLEMENTED
**Mode:** collaborative

## Goal

Catch the common foot-gun where someone edits `services/api/src/db/schema/` but
forgets to run `drizzle-kit generate` and commit the resulting migration. Today
nothing flags this until a deploy applies a schema that doesn't match the committed
migrations. This ticket adds a CI gate that runs `drizzle-kit generate` on every PR
and fails if it produces a new, uncommitted migration — keeping the committed
migrations in `src/db/migrations/` always in sync with the schema source.

This is the descoped version of FND-019. The original ticket also specified a
production cron that introspected the live Neon DB and alerted via Discord/Slack/Sentry;
that was dropped (2026-05-29) as low-value/high-maintenance for a solo, pre-users MVP
where every schema change goes through a committed migration and FND-020 runs migrations
on every deploy. See the ticket Notes.

## Acceptance criteria (descoped)

- `db:generate:check` script (api package + root passthrough) runs `drizzle-kit generate`
  and fails on an uncommitted migration diff, mirroring `gen:api:check`.
- A dedicated `schema-drift-check.yml` workflow runs it on every PR (mirrors
  `api-client-check.yml`).
- Editing `schema.ts` without committing the generated migration fails the PR; a clean
  tree passes.
- No live database required (the check is schema-source-only).

## Key decisions

- **Mirror the existing `gen:api:check` pattern exactly.** That gate already exists for the
  OpenAPI client: a `gen:api:check` script (`pnpm gen:api && git diff --exit-code -- <path>`)
  plus a standalone `api-client-check.yml` workflow. We add the schema equivalent so the two
  staleness gates are symmetric and obvious to maintain.
- **No DB needed.** `drizzle.config.ts` already documents that `drizzle-kit generate` reads
  schema files only and never opens a connection, so the CI job needs no `DATABASE_URL` /
  Neon access — just `pnpm install` + the generate.
- **Standalone workflow, not a new `ci.yml` job.** `api-client-check.yml` is standalone; we
  follow that precedent for symmetry (the FND-016 comment explicitly left these gates as
  either-or, and chose standalone).

## Files touched

- Modified: `services/api/package.json` — add `db:generate:check` script.
- Modified: `package.json` (root) — add `db:generate:check` passthrough (mirrors root
  `gen:api:check`).
- Created: `.github/workflows/schema-drift-check.yml` — runs `pnpm db:generate:check` on PRs
  (mirrors `api-client-check.yml`).

## Script

In `services/api/package.json`, mirroring `gen:api:check`:

```jsonc
"db:generate:check": "drizzle-kit generate && git -C $(git rev-parse --show-toplevel) diff --exit-code -- services/api/src/db/migrations"
```

In root `package.json`, mirroring root `gen:api:check`:

```jsonc
"db:generate:check": "pnpm --filter @picksleagues/api db:generate:check"
```

`drizzle-kit generate` writes nothing when there are no schema changes ("No schema changes,
nothing to migrate"), so the `git diff --exit-code` passes on a clean tree and fails when an
uncommitted migration (and/or updated `meta/` snapshot) is produced.

## Workflow

`.github/workflows/schema-drift-check.yml`, mirroring `api-client-check.yml` (checkout →
pnpm setup → node 22 + pnpm cache → `pnpm install --frozen-lockfile` → run
`pnpm db:generate:check` with a clear failure message pointing the author to run
`pnpm db:generate` and commit).

## Tests / verification

This is a CI gate, not a Vitest target. Verification:

1. **Clean tree passes:** run `pnpm db:generate:check` on the current `schema/` — exits 0,
   no diff (the committed `0000_*` migration already matches the schema).
2. **Dirty tree fails:** temporarily add a column to a schema file, run
   `pnpm db:generate:check`, confirm it exits non-zero with the stale-migration message,
   then revert. (Done locally during implementation; not committed.)
3. Lint/typecheck/format unaffected (no TS source added); run `pnpm lint && pnpm typecheck &&
   pnpm format:check` to confirm the JSON/YAML edits are clean.
