## Goal

Make the web app sign-in-able. FND-014 shipped the API surface (`/api/auth/*`
Better Auth handler, `/api/me` with `MeResponse` + `Unauthorized` schemas, the
session-loading middleware that backs `c.get("user")`). This ticket consumes
that surface from the SPA: a Better-Auth-React-client wrapper, a `<AuthGuard>`
component that gates routes on session, a `/sign-in` page with Google + Discord
OAuth buttons, a sign-out control, and the four-state UX (loading / empty /
error / happy) that UI standards require.

This is the surface a real human first touches the product through — so it has
to feel like "Sunday afternoon with friends," not a corporate IDP screen. Two
buttons. Friendly copy. Mobile-first at 375px. Both themes verified.

Per spec § Auth — Better Auth, the SPA uses **Better Auth's own client SDK**
for `/api/auth/*` calls (not the generated typed client — that handler is
deliberately not in the OpenAPI spec per FND-014's documented exception). The
generated typed client is still used for `/api/me`, which is how the session
query gets typed via `MeResponse`.

## Acceptance criteria (from ticket)

- Unauthenticated users hitting a protected route are redirected to `/sign-in`.
- Sign-in via Google works end-to-end.
- Sign-in via Discord works end-to-end.
- Sign-out clears session and returns to `/sign-in`.
- All four states (loading, empty/initial, error, happy) implemented.

Implicit (from CLAUDE.md non-negotiables and UI standards): mobile-first at
375px, shadcn primitives only, both themes verified, no raw `fetch` in feature
code.

## Approach

Step-by-step. Each step says WHAT and WHERE.

### Phase 1 — Dependencies

1. **Install Better Auth in `apps/web`** as a runtime dependency:
   - Package: `better-auth` (same package the API uses; the React client is
     a sub-path export at `better-auth/react`).
   - Pin the **same minor version** that's installed in `services/api` to
     keep both halves of the auth contract in lockstep. Check
     `services/api/package.json` for the exact version and match it
     (currently `^1.6.x` per the FND-014 plan).
   - No separate `@better-auth/react` package exists — `better-auth/react`
     is a sub-path of `better-auth`. Verified against the Better Auth docs
     (*"The official package is `better-auth/react`... `import { createAuthClient } from 'better-auth/react'`"*).
2. **No `jwt()` client plugin** is wired in MVP. The web app authenticates
   via the cookie session that Better Auth issues by default; JWT mint is
   for the post-MVP mobile client (per FND-014 § Risks "Cookie + JWT
   simultaneously"). Adding the client plugin would be YAGNI today.
   Recorded under § Assumptions.
3. **No new shadcn primitives** added. Error UI composes `Card` + Lucide
   `AlertCircle` per the existing pattern in `routes/index.tsx`.

### Phase 2 — Auth client module

4. **Create `apps/web/src/lib/auth-client.ts`** as the single source for
   the Better Auth client instance:
   ```ts
   import { createAuthClient } from "better-auth/react";

   export const authClient = createAuthClient({
     // baseURL intentionally omitted: same-origin in both dev (Vite proxy)
     // and prod (single Vercel project). Better Auth defaults to /api/auth.
   });

   export const { signIn, signOut, useSession, getSession } = authClient;
   ```
   - No conditional `baseURL` based on env (per FND-014's same-origin
     reversal). If a future cross-origin consumer appears, this is the
     one-line edit.
   - Export the destructured helpers for ergonomic call sites
     (`signIn.social({...})`, `useSession()`, `signOut()`), per the Better
     Auth React docs pattern.
5. **`apps/web/src/lib/auth-client.test.ts`** — a small smoke test that
   the module imports without throwing and that `authClient.signIn`,
   `authClient.signOut`, `authClient.useSession`, `authClient.getSession`
   are all defined. (Insurance against an SDK version bump silently
   changing the export shape.)

### Phase 3 — Session query

The plan uses the **typed `/api/me` endpoint via TanStack Query** as the
SPA's source of truth for session, NOT Better Auth's `useSession()` hook
directly. See § Assumptions for why.

6. **Create `apps/web/src/lib/session.ts`** exporting `sessionQueryOptions`:
   ```ts
   import { queryOptions } from "@tanstack/react-query";
   import { apiClient } from "./api";
   import type { components } from "./api-client/types.gen";

   export type SessionUser = components["schemas"]["MeResponse"];

   export const SESSION_QUERY_KEY = ["session"] as const;

   export const sessionQueryOptions = queryOptions({
     queryKey: SESSION_QUERY_KEY,
     queryFn: async (): Promise<SessionUser | null> => {
       const { data, error, response } = await apiClient.GET("/api/me");
       if (response.status === 401) return null;
       if (error) throw new Error("Failed to load session");
       return data ?? null;
     },
     // Session changes are rare and user-initiated; refetch on focus is the
     // main signal. Stale time longer than the global 30s default to avoid
     // hammering /api/me as the user navigates.
     staleTime: 5 * 60 * 1000,
     // Always refetch on focus — if the user signed in/out in another tab,
     // we want this tab to notice quickly.
     refetchOnWindowFocus: true,
     // Don't retry 401s; that's a normal "not signed in" response, not a
     // network failure. Other failures retry per the default.
     retry: (failureCount, error) => {
       if (error instanceof Error && error.message === "Failed to load session") {
         return failureCount < 2;
       }
       return false;
     },
   });
   ```
   - Returning `null` for 401 (rather than throwing) makes "not signed in"
     a first-class success state for the query. `AuthGuard` and headers can
     read `data === null` as "show the sign-in CTA" without an `isError`
     dance.
   - The query function uses the typed client's
     `{ data, error, response }` triple (the `response` field carries the
     HTTP status, which is how we distinguish 401 from other errors).
7. **`apps/web/src/lib/session.test.ts`** — unit tests for the query
   function, mocking `apiClient.GET`:
   - `200 with user → returns SessionUser`
   - `401 → returns null`
   - `network/500 error → throws "Failed to load session"`
8. **Helpers for sign-out**: add a `useSignOut()` hook in `lib/session.ts`
   that wraps `authClient.signOut`, invalidates the session query on
   success, and shows a Sonner toast on failure. Returns
   `{ signOut: () => Promise<void>, isPending: boolean }`. This is the
   single sign-out entry point feature code uses (no raw
   `authClient.signOut()` calls in components).

### Phase 4 — AuthGuard component

9. **Create `apps/web/src/components/auth-guard.tsx`**:
   - Props: `{ children: React.ReactNode }`. No `redirectTo` prop — the
     guard always redirects to `/sign-in` (the only sign-in surface).
   - Reads `useQuery(sessionQueryOptions)`.
   - Reads the current TanStack Router location via
     `useRouter().state.location.pathname + search` to compute the
     `redirect` search param.
   - Renders one of four branches:
     - **Loading** (`isLoading || isFetching && data === undefined`):
       a full-screen spinner card with `aria-busy="true"` and a `Skeleton`
       for the user avatar slot — sized to match the eventual happy-path
       layout so no layout shift (per UI standards § "Avoid layout shift").
       Copy: "Checking your session…" (one line, friendly).
     - **Error** (`isError`): Card with `AlertCircle` icon + "Couldn't
       check your session" + `Retry` (calls `refetch()`) and `Sign in`
       buttons. Matches the existing health-card error pattern.
     - **Empty / unauthenticated** (`data === null`): a `<Navigate>`
       redirect to `/sign-in?redirect=<encoded-current-path>`. Using
       TanStack Router's `<Navigate>` component (not an effect) so the
       redirect happens declaratively and the guard's "I am not the
       content" state never paints. (See § Risks for the SSR vs. SPA
       consideration — moot here because we're SPA.)
     - **Happy** (`data !== null`): renders `{children}`.
   - Type-shape: `function AuthGuard({ children }: { children: React.ReactNode }): React.JSX.Element` — explicit return type per code-standards § TypeScript.
10. **`apps/web/src/components/auth-guard.test.tsx`** — RTL tests, mocking
    `apiClient.GET`:
    - `loading state — shows "Checking your session…" and Skeleton, hides children`
    - `error state — shows "Couldn't check your session", Retry button, hides children`
    - `unauthenticated — does not render children, navigates to /sign-in?redirect=...`
      (Use `MemoryHistory` + a tiny test router so `<Navigate>` is asserted
      via the resulting route state, not a window mock.)
    - `authenticated — renders children`
    - `retry — clicking Retry triggers refetch`
    Mirrors the test shape of `apps/web/src/routes/index.test.tsx`.

### Phase 5 — Routes

11. **Create `apps/web/src/routes/sign-in.tsx`** — file-based route for
    `/sign-in`:
    - `createFileRoute("/sign-in")` with a `validateSearch` Zod-ish schema
      that accepts an optional `redirect` string. TanStack Router's
      `validateSearch` is documented for this; we use a hand-rolled
      validator (no zod-as-router-validator dep) of shape
      `{ redirect?: string }`.
    - `beforeLoad` (or a component-level guard) that checks the session
      query and, if already authenticated, redirects to the `redirect`
      param or `/`. Avoids the "signed in, but staring at the sign-in
      page" footgun if the user navigates back.
    - **Component layout (mobile-first at 375px):**
      ```
      [ centered card, max-w-sm, p-6 mobile / p-8 desktop, w-full ]
      ┌─────────────────────────────────────────┐
      │   "Picks Leagues"   (CardTitle)         │
      │   "Sign in to make your picks."         │
      │   (CardDescription, friendly tone)      │
      ├─────────────────────────────────────────┤
      │ [ Continue with Google     ]  full-w    │
      │ [ Continue with Discord    ]  full-w    │
      │                                          │
      │ (optional error region)                  │
      └─────────────────────────────────────────┘
      ```
      - Buttons are full-width, vertically stacked, `space-y-3`. Each
        button uses shadcn `<Button variant="outline" size="lg">` with a
        Lucide icon on the left (`Mail` won't do — we use brand-neutral
        icons; per UI standards § Visual character "Lucide icons only",
        and shadcn defaults). Specifically: **placeholder Lucide icons**
        for now (e.g., `Chrome` for Google, `MessageSquare` for Discord);
        if the implementer prefers brand SVGs they may inline them, but
        Lucide is the default and is allowed per the standards. See
        § Open questions.
      - Each button shows a spinner-inside-button when its OAuth flow is
        in flight (`<Loader2 className="animate-spin" />` from Lucide is
        the standard shadcn loading-button pattern; existing components
        don't yet show this but the pattern is canonical).
      - Touch targets ≥ 44×44px — shadcn `size="lg"` is 44px high.
      - Bottom-anchored on small viewports: `min-h-screen flex
        items-center justify-center` keeps the card vertically centered
        on phone; on desktop the layout naturally falls out.
      - **Theme toggle in the top-right corner** (using the existing
        `<ModeToggle />`) so the user can verify both themes before
        committing — see § Open questions for the alternative.
    - **Four states on this page:**
      - **Loading**: while a sign-in click is in flight (button shows
        spinner; the other button is disabled to prevent double-OAuth);
        also a brief full-page session-check loading state if the user
        lands on `/sign-in` from a deep link (handled by reading the
        session query in `beforeLoad`).
      - **Empty / initial**: the page as drawn above — this is the
        "empty" state of "not signed in yet."
      - **Error**: the OAuth call returned an error (provider not
        configured, network failure, user-denied). Display an inline
        error region inside the card (Card-wrapped `AlertCircle` +
        message + Retry implicit via re-clicking the button). Specific
        copy by error type: "Google sign-in isn't set up yet. Try
        Discord, or come back later." for the "unconfigured provider"
        case the API will return when `GOOGLE_CLIENT_ID` isn't set.
      - **Happy**: OAuth redirect leaves the page — Better Auth's
        `signIn.social({ callbackURL })` initiates a top-level
        navigation to the provider; on return, Better Auth's server-side
        callback at `/api/auth/callback/<provider>` redirects the
        browser back to `callbackURL` (which we set to the
        `redirect` search param or `/`). The session query then
        refetches automatically on window focus, and `AuthGuard` lets
        the user through.
    - **OAuth click handler**:
      ```ts
      async function handleSignIn(provider: "google" | "discord"): Promise<void> {
        setPending(provider);
        try {
          await signIn.social({
            provider,
            callbackURL: search.redirect ?? "/",
            errorCallbackURL: "/sign-in?error=oauth", // brings them back here
          });
          // On success Better Auth navigates the browser away; this
          // branch typically never runs.
        } catch (e) {
          setPending(null);
          setErrorMessage(/* derived from e */);
        }
      }
      ```
      State is local React state (`useState`) — per code-standards § Frontend
      *"No `useState` for fetched data"*: this isn't fetched data, it's
      ephemeral UI state (which button is pending, what error to show).

## Risks and open questions

- **`<Navigate>` mid-render vs. `useEffect` for the unauthenticated
  redirect.** TanStack Router's `<Navigate>` component is documented for
  declarative redirects mid-render and avoids the "guard renders nothing
  for one frame then redirects" double-paint. If a future SSR ever
  appears, this would need revisiting — moot in MVP (pure SPA).
- **`beforeLoad` vs. component-level guard for the home page.** A
  route-level `beforeLoad` that checks the session query and throws
  `redirect({ to: "/sign-in" })` is the "right" TanStack Router idiom,
  but it requires hoisting the QueryClient into router context and
  calling `queryClient.ensureQueryData(sessionQueryOptions)` from the
  loader — non-trivial wiring for a single route. Plan picks the
  component-level `<AuthGuard>` wrap for this ticket because (a) it's
  the simpler primitive to test, (b) it's the pattern most teams reach
  for first, and (c) we can refactor to `beforeLoad`-based guards later
  without changing the component API. Override is fine if the
  implementer prefers `beforeLoad` from day one — both meet the
  acceptance criteria.
- **Sign-in buttons without configured OAuth providers.** If
  `GOOGLE_CLIENT_ID` isn't set locally, clicking "Continue with Google"
  hits Better Auth's `/api/auth/sign-in/social` which (per FND-014's
  plan) returns the documented unconfigured-provider error. The
  sign-in page must catch that and render a friendly inline message
  ("Google sign-in isn't set up yet."). Verified the page handles it
  via the catch branch in `handleSignIn`. The user's testing flow
  involves provisioning real OAuth apps before merge, so this is the
  edge case for contributors, not the happy path.
- **Spine E2E (Playwright) isn't set up.** Two of the five acceptance
  criteria ("Sign-in via Google works end-to-end", "Sign-in via Discord
  works end-to-end") are properly testable only with Playwright. The
  unit tests cover the SPA-side wiring; the actual OAuth round-trip is
  covered by **the human's manual verification** at PR-test time. Flag
  Playwright setup as future work — no backlog ticket owns it today.
  Suggest folding into the eventual web-side spine E2E ticket.
- **Cookie sameSite + secure flags in dev.** Better Auth's defaults
  generally produce cookies that work for `localhost` dev. If the
  human's manual OAuth test fails with "session cookie not set," check
  Better Auth's cookie config — this is the most likely first failure
  mode. Not a planning concern (FND-014 handled the cookie config),
  just noting it as the most likely surprise during testing.
- **Cross-tab sign-out.** `refetchOnWindowFocus: true` on the session
  query means tab B will notice a sign-out in tab A within seconds of
  returning to tab B. Not instant but adequate for MVP. A future
  storage-event listener can make this instant; out of scope.
- **Brand icons (Google, Discord).** Plan uses Lucide placeholders
  (`Chrome`, `MessageSquare`) per UI standards § Visual character
  *"Lucide icons (shadcn's default) only"*. Brand SVGs would arguably
  improve recognition (familiar Google G, Discord logo) and are
  near-universal on OAuth screens. **Plan default: Lucide.** See § Open
  questions — the human may want brand SVGs and explicitly allow them
  as a one-off exception.

## Assumptions (planner choices the human can override at review)

1. **`/api/me` over `authClient.useSession()` as the session source of
   truth.** Better Auth's `useSession()` works and is well-documented.
   Why not use it directly? (a) `/api/me` already exists and FND-014
   shipped the typed-client schema for it — there's no duplication cost.
   (b) Using TanStack Query for session keeps cache invalidation
   uniform with the rest of the app (one client to manage, one
   `queryClient.invalidateQueries({ queryKey: SESSION_QUERY_KEY })`
   pattern after mutations). (c) The narrow `MeResponse` projection
   (defined server-side) is what the SPA actually wants — Better Auth's
   raw session shape includes internal fields we don't need to surface
   in the SPA. (d) When the post-MVP mobile app appears, it'll use the
   same `/api/me` pattern, so the SPA pattern transfers. **Override:**
   use `authClient.useSession()` and skip `lib/session.ts` — fewer
   lines but loses the unified query-cache story.

2. **No `jwt()` client plugin in MVP.** Web auth runs on the cookie
   session. JWT mint via `/api/auth/token` is for mobile (post-MVP).
   Adding the plugin client-side is YAGNI today. **Override:** wire
   the JWT client plugin now if you want to dogfood the JWT path on
   web from day one (would require deciding whether to use JWT for
   `/api/me` calls too, which complicates the cookie story).

3. **Component-level `<AuthGuard>` wrap, not route-config `beforeLoad`.**
   See § Risks for the tradeoff. Override is mechanical (refactor
   later) if the implementer wants the more idiomatic loader.

4. **One sign-in surface (`/sign-in`), not a modal.** The product is
   simple enough that a dedicated page is clearer than an overlay.
   Also, OAuth redirects play badly with modals (the user returns to
   the page they signed in *from*, and a modal context doesn't survive
   a full-page redirect). Override: not recommended for MVP.

5. **Lucide icons for OAuth buttons** (`Chrome` for Google,
   `MessageSquare` for Discord). UI standards say Lucide only. The
   alternative (brand SVGs) would visually distinguish the buttons
   better but introduces icon files outside the standard. Override
   listed under § Open questions.

6. **Sign-out is in a dropdown on the home page, not a global header.**
   This ticket adds the minimum signed-in UI. A real app-wide header
   appears in epic 03 when there are multiple authenticated routes
   to navigate between. Override: add a minimal header in this ticket
   — would expand scope and isn't required by the acceptance criteria.

7. **Theme toggle stays on the sign-in page.** The user might want to
   verify dark mode before committing. Keeps the existing UX pattern
   (theme toggle visible everywhere). Override: hide it on sign-in for
   minimalism — small UX difference, no functional impact.

8. **No `Avatar` shadcn primitive added.** The user menu uses the
   user's name as the trigger label (no avatar image). Adding `Avatar`
   for one screen is overkill; future tickets can add it when there's
   a profile page that benefits from it.

9. **No `Alert` shadcn primitive added.** Error UI composes
   `Card` + Lucide `AlertCircle`, mirroring the existing health-card
   pattern in `routes/index.tsx`. Keeps the dependency surface small
   for one screen.

10. **`sessionQueryOptions` uses 5-minute `staleTime` and
    `refetchOnWindowFocus: true`.** Sessions change rarely but cross-tab
    sign-out should be noticed. Five minutes balances "don't hammer
    `/api/me`" against "notice changes." Tunable later if needed.

11. **The session query returns `null` (not throws) on 401.** Makes
    "not signed in" a first-class success state. The alternative
    (throw → error state) collapses "not signed in" and "network down"
    into the same branch, which is wrong UX.

12. **Sign-in errors render inline on the page, not as a Sonner toast.**
    Toasts are ephemeral and easy to miss; sign-in errors are the
    whole point of the page at that moment. Sign-out errors are
    different — they're transient and a toast is the right shape there.

**Decisions made during implementation:**

1. **Human override #1 — Brand SVGs:** Google G and Discord logos inlined in `apps/web/src/components/brand-icons/google.tsx` and `discord.tsx`. Documented in code comments as a Lucide-only exception per UI standards. Improves OAuth button recognition.

2. **Human override #2 — TanStack Router `beforeLoad`:** Applied on protected routes (`/` and `/profile`). `QueryClient` plumbed into router context via `createRootRouteWithContext<{ queryClient: QueryClient }>()` in `__root.tsx` and `createRouter({ context: { queryClient } })` in `main.tsx`. Each protected route's `beforeLoad` calls `ensureQueryData(sessionQueryOptions)` and throws `redirect({ to: "/sign-in", search: { redirect: location.href } })` on null user. `<AuthGuard>` reduced to presentational component for loading/error UI (now used as `pendingComponent` + `errorComponent` in routes). Cleaner than component-level wrap for multiple routes.

3. **Reviewer round 1 must-fixes:**
   - Open-redirect vulnerability: added `sanitizeRedirect()` helper that rejects protocol-relative URLs, full URLs, `javascript:` prefix, and non-path characters. Only allows same-origin paths starting with `/`.
   - `signOut()` silent errors: changed to destructure `{ error }` from the response and bail with a Sonner toast on failure, don't treat failure as success.
   - Dead exports: removed `useSession` and `getSession` which the planner mistakenly listed in `auth-client.ts`. Only `authClient`, `signIn`, `signOut` exported.
   - `sign-in.test.tsx` anti-pattern: rewritten to render the real `SignInComponent` via `createFileRoute` mock (injecting `useSearch`), not reimplementing the component in the test.

4. **Manual test round critical bug fixes:**
   - **`location.href` vs. `location.search`:** TanStack Router's `location.search` is the parsed search object (Record<string, any>), not a string. Attempted concatenation caused TypeError "Cannot convert object to primitive value". Fixed both `routes/index.tsx` and `routes/profile.tsx` `beforeLoad` to use `location.href` (which is "pathname + searchStr + hash, no origin"). `sanitizeRedirect()` still validates the input against open-redirect.
   - **Stale session cache after sign-out:** `invalidateQueries` alone triggers async refetch; `ensureQueryData` at `beforeLoad` returned the cached user synchronously. After sign-out on `/sign-in` redirect, `/`'s `beforeLoad` saw stale user → stuck loop. Fix: `queryClient.setQueryData(SESSION_QUERY_KEY, null)` synchronously clears cache after sign-out, plus `void queryClient.invalidateQueries(...)` marks it stale so cross-tab `refetchOnWindowFocus` works. Session stays fresh for 5 min per `staleTime` setting; cross-tab updates are noticed on next focus + stale, adequate for MVP per planner's cross-tab risks section.

5. **Local OAuth setup note:** `BETTER_AUTH_URL=http://localhost:5173` (the frontend origin, not `:3000`). OAuth round-trips through Vite proxy so session cookie is scoped to the SPA origin. Registering `:3000` would set the cookie on the API port and the SPA wouldn't receive it. Added to `services/api/README.md` "Local OAuth setup" subsection and `docs/deploy.md` `BETTER_AUTH_URL` row for next contributor.
